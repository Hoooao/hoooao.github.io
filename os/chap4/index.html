导言：

    存在三种情况会导致OS停下手边的东西转而去执行一些特定的代码，它们分别是：

系统调用：进程调用ecall来让内核去执行一些操作

Exception：当一条指令做了非法的事情，进程会被杀死

设备中断（Device interrputs）：一个设备会向CPU发送注意请求，例如磁盘读写完成

这三种让“CPU去执行另一项任务”的行为统一叫做Traps。一个Trap处理的流程通常是：

Trap处理机制让CPU切换控制（Control）

保存当前进程的一切寄存器，为了让进程继续执行。

执行对应Trap处理的代码

恢复之前的进程

对于这个被中断的进程来说，Trap是“透明的”，也就是说，当前进程并不会知道自己被一个Trap打断了，因为它会被在未来某一个时刻继续运行（或者被杀死）。







4.1 RISC-V Trap机制 RISC-V trap machinery

    上述的行为都需要被 控制与状态寄存器（CSRs：Control and Status Registers） 支持（satp也是其中之一），下面是一些常用的CSRs：

stvec：保存着Trap handler函数（uservec/kernelvec）的程序地址，进程通过它来跳转到处理Trap的代码

scause：记录着Trap发生的原因

sstatus：其中的SIE-bit指示是否能处理interrupts，其中的SPP-bit指示Trap是从什么模式出现的，并控制在调用指令sret后，跳转至目标模式（sret会把sepc寄存器的内容写到pc寄存器，实现跳转模式），例如0是UM。

sscratch：保存一些重要的进程与内核相关的数据并且用来保存进程的寄存器。

sepc：用来保存Trap发生时的pc（它之后会被重写），用来之后继续该进程。

    简化的流程：

Trap发生后，如果是被一个设备中断引起，同时SIE未设置，则忽略这个Trap；

把SIE清除（之后的过程中不会处理新的设备中断）； 把pc放入sepc；

把当前的模式放入SPP； 记录scause； 切换至SM（kernel）；

把stevec的值放入pc； 开始处理。

    注： 在处理Trap的时候，CPU不负责切换satp，kstack,和保存除pc之外的寄存器。这样给了软件更多的灵活度，比如对于一些操作系统，它们不需要切换satp去处理Trap。但是同时也要求处理Trap的程序来做这些工作。这里的“CPU不负责”指的是CPU的指令不会去执行一些内容。







4.2 用户空间的Traps Traps from user space

    处理Trap的函数调用顺序是：uservec->usertrap->usertrapret->userret。

    已知stvec会指向uservec，但问题是，在切换satp之后，uservec的地址必须要被内核的内存空间映射，才能继续运行。解决方法：xv6为每个进程，包括kernel，分配了一个page叫做trampoline（蹦床），其内容定义（包括uservec，userret两个函数）在trampoline.S（内存地址顶部），位于虚拟内存中的TRAMPOLINE。其在各个进程与内核中的VA是一样的（TRAMPPOLINE）。

    当Trap出现，使用stvec找到位于trampoline中的uservec。在uservec运行的开始，32个用户寄存器(非CSRs，它们可以被user code使用)保存着这个进程的数据，它会使用指令来交换sscratch与a0的值，这样原a0就被保存了：此时uservec就可以使用a0而不用担心数据损坏了。之前原来的sscratch中保存的实际上是一个page的地址，这个page被叫做trapframe，其地址邻接trampoline的正下方。它是一个每个进程都独有的内存空间，足够存放32个寄存器和其他数据，而现在a0指向了这个空间。uservec通过指令把每个寄存器依次放入了trapframe（从0x40开始是因为之前的偏移量中存放着其他数据）。最后在把sscratch中保存的原a0放入trapframe：所有的寄存器的对应的值都被保存了。

uservec:    
        //交换两个寄存器的值
        csrrw a0, sscratch, a0

        # save the user registers in TRAPFRAME
        sd ra, 40(a0)
        sd sp, 48(a0)
        sd gp, 56(a0)
        //···
        //共计31个寄存器
        //···
        sd t3, 256(a0)
        sd t4, 264(a0)
        sd t5, 272(a0)
        sd t6, 280(a0)

        //保存原a0
        csrr t0, sscratch
        sd t0, 112(a0)

        //······

    之后仍然适用trapframe，uservec会把这个进程与内核相关的数据取出，比如kstack，usertrap()，和kernel的PT的地址等。此时便可把satp指向kernel的PT了。最终在uservec中，由于trampoline也以一样的地址映射在内核的内存空间中，调用usertrap。

        //···接上···

        # restore kernel stack pointer from p->trapframe->kernel_sp
        ld sp, 8(a0)

        //···略···

        # restore kernel page table from p->trapframe->kernel_satp
        ld t1, 0(a0)
        csrw satp, t1
        sfence.vma zero, zero

        # a0 is no longer valid, since the kernel page
        # table does not specially map p->tf.

        # jump to usertrap(), which does not return
        jr t0

    usertrap首先会把stvec设置为kernelvec，为可能的内核中出现的Trap服务。然后把sepc的值放入p->trapframe->epc中（p指的是原进程），防止使用其他进程覆盖掉sepc从而丢失原pc。之后它会取出scause中的值，用作Trap类型的判断：

系统调用：执行syscall，并且把epc+=4，用来跳过原pc对ecall的调用指令。

设备中断：调用devintr。

Exception：杀死原进程。

void usertrap(void){

  int which_dev = 0;
    //只有来自用户空间的才能调用usertrap
  if((r_sstatus() & SSTATUS_SPP) != 0)
    panic("usertrap: not from user mode");

  //修改stvec
  w_stvec((uint64)kernelvec);

  struct proc *p = myproc();
  
  // 保存原pc
  p->trapframe->epc = r_sepc();
  
  //检查Trap类型
  if(r_scause() == 8){
    // system call

    if(p->killed)
      exit(-1);

    //修正偏移量
    p->trapframe->epc += 4;

    //RISV-C的Trap处理硬件自动关闭中断使用
    // 因为使用完了寄存器，可以允许Interrupts，手动开启
    intr_on();
    //系统调用
    syscall();

    //处理Interrupts
  } else if((which_dev = devintr()) != 0){
    // ok
  } else {
    //处理Exception
    printf("usertrap(): unexpected scause %p pid=%d\n", r_scause(), p->pid);
    printf("            sepc=%p stval=%p\n", r_sepc(), r_stval());
    p->killed = 1;
  }

  if(p->killed)
    exit(-1);

  // give up the CPU if this is a timer interrupt.
  if(which_dev == 2)
    yield();

    //调用
  usertrapret();
}

    最后在usertrap中调用usertrapret。后者把stvec又改回trampoline中的uservec为未来可能在进程中出现的Trap服务。同时进程中trapframe的内核与进程相关的数据也被更新（如kstack，kernel的PT等）。在重新设置sepc为pc后，调用userret。

void
usertrapret(void)
{
  struct proc *p = myproc();

  //在切换回用户空间前，禁止interrupts。
  //因为这段时间中，stvec指向usertrap：不能处理interrupts
  intr_off();

  //sevec切换回uservec
  w_stvec(TRAMPOLINE + (uservec - trampoline));

  //更新trapframe
  p->trapframe->kernel_satp = r_satp();         // kernel page table
  p->trapframe->kernel_sp = p->kstack + PGSIZE; // process's kernel stack
  p->trapframe->kernel_trap = (uint64)usertrap;
  p->trapframe->kernel_hartid = r_tp();         // hartid for cpuid()

  //···略（sstatus相关设置）···

  //切换sepc回原pc，供sret使用
  w_sepc(p->trapframe->epc);

  //这里的satp只是个变量，而非寄存器
  uint64 satp = MAKE_SATP(p->pagetable);

  //调用userret，传入参数a0=trapframe, a1=satp
  //fn计算到目标的Trampoline中的某行，即userret
  uint64 fn = TRAMPOLINE + (userret - trampoline);
  //使用fn作为一个函数指针，传参运行 
  ((void (*)(uint64,uint64))fn)(TRAPFRAME, satp);
}

注： 希望获得satp的指针是因为想把它作为参数传入Trampoline.S 这个汇编程序中。由于它是汇编程序，给予指针比在其中获取指针更方便

     userret的行为与uservec相反：

        //把satp换成原进程的pt
        csrw satp, a1
        sfence.vma zero, zero

        //临时保存原a0到t0
        ld t0, 112(a0)
        //交换sscratch与原a0位置
        csrw sscratch, t0

        //恢复所有寄存器
        ld ra, 40(a0)
        //···略···
        ld t6, 280(a0)

        //最后恢复a0
        csrrw a0, sscratch, a0

        //返回sepc指向的位置
        sret

注：

保存在trapframe中的a0的值已经被更改了： 在调用系统调用后，这个系统调用的返回值写入了a0。 除了修改pc的值为sepc的值，sret同时修改MODE，并且允许中断。







4.3 Code: Calling system calls

    在第二章，initcode.S调用exec最终开启了第一个进程console。细节地说，initcode.S中的start函数给寄存器a1，a0传递参数，并且把一个系统调用的特定号码放入a7。这个号码定义在syscall.c 的syscalls数组中，对应着特定的系统调用地址。start最后一行使用了ecall，它切换运行模式，保存pc与sepc， 使得trap产生并执行uservec，usertrap，syscall。在syscall中，a7的值会与上述的数组对比，得到目标系统调用：initcode.S调用了SYS_exec，从而调用exec系统调用。在运行完毕后，系统调用会把返回值放入a0，负数代表运行出错，0或正数代表成功：如果系统调用号码不存在，返回-1。







4.4 Code: System call arguments

    在第一章中讨论的所有以函数形式出现的系统调用实际上都是对于该系统调用的。这些封装函数统一定义在usys.S（通过usys.pl 生成）中。

.global fork
fork:
 li a7, SYS_fork
 ecall
 ret

这是fork()的定义，只是把特定的常数放到寄存器a7中并调用ecall进入trap。从trap处理中回来之后调用ret返回用户代码。当这些封装函数被传参时，如给其他函数传参一样：参数会按顺序放入寄存器（a0,a1...）中（并且在调用uservec时放入p->trapframe）。xv6使用argint，argstr，argaddr来分别从寄存器中拿到整型，字符串，和指针。比如：

// Fetch the nth 32-bit system call argument.
int argint(int n, int *ip) {
  *ip = argraw(n);
  return 0;
}

但它实际上也只是对于argraw进行的封装：

static uint64
argraw(int n)
{
  struct proc *p = myproc();
  switch (n) {
  case 0:
    return p->trapframe->a0;

  //...省略case 1～5，分别对应寄存器a1～a5...

  case 5:
    return p->trapframe->a5;
  }
  panic("argraw");
  return -1;
}

后者通过p->trapframe取出原进程保存的寄存器的值。

    虽然得到寄存器中的值对于内核来说相对简单，但一直调用某个系统调用的进程的内存空间与内核不共享PT，当参数是指针式，会出现两个问题：

1.内核无法直接使用进程提供的指针去得到指针指向的数据。

2.指针指向的数据可能是恶意的。 所以需要一种能把数据取出，又能保证安全的方法。

    xv6中使用一些函数达成此目的，比如fetchstr:

int 
fetchstr(uint64 addr, char *buf, int max)
{
  struct proc *p = myproc();
  int err = copyinstr(p->pagetable, buf, addr, max);
  if(err < 0)
    return err;
  return strlen(buf);
}

它调用了copyinstr，后者需要4个参数：原进程的PT，一个用来接收数据的buffer，目标VA，与读取Bytes的上限。copyinstr通过使用walkaddr来得到目标page的PA，而walkaddr实际上调用了walk来获取目标所在的PTE，然后翻译成PA（通过PPN）。最后copyinstr通过传入的目标VA中的偏移量来获取目标数据，并把数据传入buffer。







4.5 Traps from kernel space

    内核中的Trap不同于用户的Trap处理。当位于内核中时，stvec一定指向kernelvec，且在保存寄存器后可直接进入kerneltrap（保存寄存器是因为内核可能切换线程，这也导致寄存器必须要保存到这个进程自己的栈中）。在kerneltrap中，由于内核可以直接运行系统调用，它只处理报错和设备中断。如果是一个设备中断，内核会调用devintr；若是一个报错，内核会停止运行。另外，如果这个中断是一个 时钟中断（Timer interrupt），内核调用yield，使得另一个进程考试运行（而未来某一时刻原线程也会继续运行–Chap7）。当kerneltrap执行完毕，一切寄存器（包括sepc）都会复原，原线程继续运行。注： 在usertrap中，stvec会把uservec切换至kernelvec，换句话说，在切换之前，在内核运行时的stvec会有一段时间是uservec，这就是为什么一定要把中断关掉，防止在内核中使用uservec。







4.6 Page-fault exceptions

    xv6对于Exception的处理很简单：杀死出现报错的进程，或停止内核运行。但在现实中，Exception本身可以实现一些优化和功能。比如Page fault exception在其他的操作系统中有多个应用：写入时复制（COW fork：Copy-on-Write fork），懒分配（Lazy Allocation），硬盘分页（Paging from disk），Zero-fill on demand，Demand-paging，Memory map files。这些功能的实现全部依靠虚拟内存实现。除了分割各个进程的内存空间，在这个语境下更重要的是虚拟内存实现了 Level of Indirection：也就是虚拟与物理内存之间的映射关系。比如Trampoline,gurar都得益于这种indirection。这些功能都借助Page fault exception发生后OS的行为：

  1. 出现错误的虚拟内存地址会被放到```stval``寄存器中，
  2. 错误出现的种类会被放入scause中。13：读取导致；15：写入导致；12：执行导致。
  3. 导致错误发生的指令的虚拟地址保存在sepc（之后被放入epc）。这个不是上面“发生错误的地址”，而是导致错误（比如某个指令写入了没有权限的位置）的指令的地址。

    写入时复制（COW fork：Copy-on-Write fork）。当使用fork时，父进程的内存被完整复制了一份给子进程。但通常子进程的目的只是调用exec来开启另一个进程。所以为子进程分配完整的内存有些浪费。而COW实现了父子共用内存，并为子进程按需分配内存。当使用了fork后，OS并未分配任何新的内存给子进程，只是把子进程的内存空间映射到与父进程一样的物理内存地址，并且把子进程的PTE设置为只读（Read only）：所以当子进程想要对某一个内存地址进行操作的时候（除了读），会触发page fault exception。出发的原因会被记录在scause中，之后内核会把出发报错的page真正地复制给子进程的PTE，然后先前导致报错的指令就可以继续运行了。

    懒分配（Lazy Allocation）。通过sbrk积极分配（Eager allocation），一个进程可以要更多的物理内存，但有时进程会滥用sbrk。但是通过懒分配，当sbrk被使用的时候，OS虽然会增长内存空间（p->size增加），但会在PT中把这些PTE标记为无效（PTE_V）。所以当进程使用这个地址的时候，由于PTE无效（根本就没有map）会出现page fault exception。此时出错的虚拟地址小于进程内存空间大小，但是大于栈的位置（p->stack）。之后OS才会真正地分配物理内存，把这个新page映射到PTE中，重新开始运行导致错误的指令的地址。

代码层面：在调用sbrk之后，单纯只增加p->sz的大小，而不真正的分配空。修改Trap机制，让它可以监测到错误的发生。然后得到错误发生的虚拟地址，分配内存，初始化page，映射PTE，继续运行。 注：需要关注uvmunmap的机制。由于进程取消映射的PTE实际上可能只是懒分配的

uvmunmap: //这部分需要被修改，它本质是在禁止空间内可以存在PTE_V无效的page
 if((*pte & PTE_V) == 0)
      panic("uvmunmap: not mapped");

    硬盘分页（Paging from disk）。当内存使用过多内存时，OS会把一些page写入磁盘，并把这些page对应的PTE标记成无效。所以当进程使用这些page时会出现page fault exception。OS会把制定的page重新从磁盘中写入内存，并更新PTE（当一个page被召回时，如果内存不够用，会把另一个page写入硬盘）。选择哪个page被驱逐通常基于 LRU： Least Recently Used算法（通过看PTE中的Access bit得知是否是LRU）。同时会优先选择驱逐 非 Dirty Page（通过看PTE的dirty bit），也就是没有修改过的Page。

     Zero-fill on demand。虽然Xv6中只有一个segment，但是对于其他操作系统可能会有一个.bss段在ELF中，代表未赋值的静态变量。它们的值在赋值前实际上都是0。因为这些变量的值都一样，那么可以让这些静态变量全部指向1个值为0的page：也就是把它们对应的PTE指向一个内容为0的page。这样不需要为每个变量都分配自己的物理内存了。由于这个page的权限将会是只读，所以当程序试图去修改它的值的时候，会触发Page fault。之后分配新的page给这个变量，映射到它的PTE，重新运行指令。优点：省空间；可以让exec少做一些初始化工作。

     Demand Paging。当一个进程被加载到程序之后，实际上它的很多内容（比如在data和text段）不会用到，可以使用类似懒分配的逻辑来优化。当exec执行时，可以只分配虚拟内存，并且设置PTE_V无效。一旦这个程序开始运行，它必然从自己虚拟地址的0x0开始，由于PTE无效，产生Pagde Fault：从文件中真正读取这个pagde到内存中，并且映射到PTE，重新开始运行指令。

    Memory map files。为了读写一个文件，使用ld，sd指令（分别为load，store），同时需要使用mmap(va,len,protect,flags,fd,offset)：它可以把fd的文件从offset开始以protect定义的权限（r/w）去音社长len byte的内容到va中。结束之后使用unmap(va,len)来取消映射PTE，恢复Dirty bit 这一过程也是使用懒分配的：不会把整个文件读进内存，只有在使用的时候才出发Page Fault；之后kernel去磁盘中读取文件并且映射到PTE。在一些内存管理中存在VMA：virtual memory area，记录着虚拟内存的映射关系。这一文件的off，fd都会被记录在vma中供未来使用，虽然文件本身不在内存中。注： flag标注一些文件是否共享于进程之间的信息。







4.7 补充

     1. 与UM相比，SM实际上只多了两个权限：

  1. 可以读写控制寄存器，比如satp,stvec；
  2. 可以使用PTE中PTE_U未设置的Page。这个bit位于PTE的后10位之一，指示UM是否可以访问该PTE。但是假如PTE_U设置了，SM就不能访问了：SM并不一定向下管控UM。 注：SM中，对地址的访问仍然严格受限于satp提供的PT。换句话说，SM完全无法直接访问PA（即使在kernel中也只是直接映射使用VA）。

0000003ffffff000 0000000080007000 0000000000001000 r-x--a-

这是shell的PT，只有6个PTE。attr是权限bits：r/w/x：读/写/执行；u：PTE_U；ab：曾被读过/写过。

     3. 虽然Trapoline与Trapframe都在用户空间中，但UM无法写入他们的地址。

     4. ecall有三个功能（提供于CPU而非OS）：

  1. 改变MODE位SM。并没有直接的方式来看MODE，只能通过当前所运行的代码来猜当前的权限
  2. 保存pc到sepc，这就是为什么说CPU不负责保存寄存器，除了pc。
  3. 让pc跳转到stvec。 注：此时不能使用C语言，因为sp未指向任何栈，直到开始使用kstack。

     5. hartid：实际上就是当前代码所运行于的CPU核心的ID，保存在寄存器tp中。myproc()使用这个值去一个特定的数组中找到对应的进程，即“当前进程”。XV6只使用1个核心，所以hartid是个定值。
