导言：

    每个进程都有自己的分页表（PT：Page table）。通过分页表（PT：Page table），kernel可以为一个进程创建私有的内存空间：每个虚拟内存地址（VA：Virtual adress） 都有对应着的物理内存地址（PA：Physical address）。对于进程来说，他在使用这一段连续的VA，但是这些VA在DRAM中的映射的物理内存地址则不一定连续。PT告诉分页硬件一个VA代表着什么，指向哪里。







3.1 分页硬件 Paging hardware

    在计算机中虚拟内存（VM：Virtual memory） 的实现是由硬件支持的，这个硬件通常被叫做内存管理单元（MMU：Memory management unit）。CPU的指令通常使用VM：当它使用某一个VA的时候，MMU会把这个VA翻译成PA，然后执行这个PA里的指令或者读取/写入这个PA里的内容。

    但是在xv6中，我们使用的是qemu的虚拟机：MMU被抽象成了一个定义在kernel/vm.c中的walk()函数。这个函数的行为与MMU类似。

    xv6跑在Sv39 RISC-V处理器上（虚拟机），这个处理器指使用64bits VA中的后39位。其中前27位用来作为分页表项（PTE：Page table entries）的索引（Index），所以一个PT最多可以有2^27个PTE。而后12位被用来当做偏移量（剩余的bits用来为未来可能的硬件升级做准备）。每个有效的PTE包含一个44bits的PPN：Physical page number和10bits的各种用来限定这个页的权限。

当一个进程使用VA的时候，MMU/walk会通过该VA的27位的索引来找到这个进程的PT中唯一的PTE。然后这个PTE中的PPN会被取出，再把VA中的后12位偏移量接在PPN的后面组成一个56位的PA（64位PA中剩下的为未来准备），然后去DRAM中找该这个PA。由于偏移量有12位，他们可以与一个PPN组合2^12个连续的PA（一个地址1Byte），这也就组成了一个Page：即一块4096Bytes的内存空间。注： 一个Page中的PA连续，但是两个相邻的PTE所指向的Page不一定连续。

    在上述的过程中，walk似乎直接用27bits的索引找到了为一个PTE，但实际的视线中，这个过程被分成了三步：27bits的索引分成了三个9bits的索引，可以被看作三个偏移量。当一个进程在运行的时候，kernel会把CPU的satp寄存器中储存的数据需改成为这个进程分配的私有的PT的物理内存地址（PT被存储在内存中）。这个“PT”不同于上述的简化模型中的PT：它本质上是一个Page。但是由于这个Page储存着PTE，也可以叫它分页表页（PTP：Page Table Page）。与正常的Page一样，它也占据4096Bytes内存。由于一个PTE占64bits，即8Bytes，一个PTP可以存放512个PTE。Satp寄存器指向的PTP也被叫做根（Root）。我们使用27bits中前9bits作为偏移量，在根页中找到为一的PTE。不同于简化的模型，这个PTE的PPN将指向另一层PTP（PPN后面接12个0）。同样地，第二个9bits被用作于这个PTP的偏移量并找到对应的PTE。重复相同步骤，我们能找到第三层PTP中最后9bit对应的PTE。此时的PTE中的PPN将与最初的VA中的12bits结合，得到真正的56bits PA。

三层PTP是对一个有2^27个PTE的PT的优化。假如真的使用后者，那么一张PT需要2^27*2^3Bytes,超出了xv6所支持的内存上限（128MB），显然不现实。而前者的大小的变化是动态。比如一个进程只使用了1个Page，或者说它只用了0x0~0xFFF的虚拟地址,在这个情况下，27位索引全部为0，对于三层PTP，每次都导向最下面偏移量为0的PTE。换句话说，每个PTP只有最下层的PTE是有效的：第一层第二层PTP的其他PTE无效，因此不会为他们创建下层PTP。所以这个例子里面只有3个PTP存在，即3*4KB=12KB。由于分层的行为类似从文件夹中找文件，所以可以把PTP看作目录。

    这样的话优化也是有代价的：每使用一个VA，就需要多次访问VM，效率变低（对比假如指使用一张PT，那么只需要检索一层目录）。缓存TLB（Translation lookaside buffer）（一个硬件）一定程度减轻了这个代价。每当使用了一个VA后，改VA与对应的PA就会被缓存入TLB。当再次使用这个VA时，会直接从TLB中调取其PA，不需要让walk重新走三层目录。注：每当进程切换的时候，TLB中的数据就会被清理（即Flush，通过指令sfence_vma调用）：这样防止之后的进程访问到之前的进程的PA。

    如Fig.3下半部分所示，除了PPN，一个PTE中还包括10bits的空间供一些Flag使用。V：如果为1则有效，反之说明改PTE无效；R/W/X：读写权限，以及是否被当作指令运行。

思考：为什么每层PTP都使用PA来找下层PTP？为什么不能使用VA？







3.2 内核内存空间 Kernel address space

    每个进程都有属于自己的分页表，除此之外，kernel也有自己的分页表（任何指令都是使用VM的，包括在kernel中）。虽然VM存在的意义是隔离，阻止进程使用物理内存，但是VA=PA的情况也是存在，kernel（大部分区域）就是这样。这种方式被叫做直接映射（Direct mapping）。

    RAM（物理内存）起始于0x80000000，而非0x0。由于kernel是直接映射且在RAM的最下方，kernel的内存空间（由VM构成的“连续的”内存）也起始于0x80000000：这个值被称为KERNBASE。选择这个数字作为物理内存的起点只是设计者的决定。0x80000000以下的地址留给了其他硬件。对这些地址的访问不是对内存（RAM）的访问，而是对对应硬件的I/O接口的访问。这种直接映射是的kernel对PA的读写更有效率。比如当fork()使用分配内存的函数给子进程复制父进程的内存内容是，这个函数会返回它分配的内存的PA。由于直接映射，fork()可以直接操作于与PA一样的VA。kernel的内存空间只有两个部分不是直接映射：

  1. Trampoline page (Chap 4)。

  2. kstack&guard page。kstack（即内核栈）与它的guard会被放在虚拟内存的顶层。guard作为kstack与内核内存空间其他内容的缓冲带存在：若一个进程的kstack溢出，没有guard的话可能会写入到其他进程的kstack。而由guard存在时，一旦guard的VM被访问的时候，就报错，从而保护其他内容。不把kstack和guard同样放在低处来直接映射是因为guard的区域显然不能存放任何有价值的数据：若为直接映射，必须给每个guard一些物理内存地址，占用不必要的空间，且可能会使得guard无效。**注：**guard的VA对应的PTE是无效的。

    RAM终止于最少0x86400000的地址（共计最少100MB）。RAM的终点被叫做PHYSTOP。由于RISC-V使用了56位的地址，PHYSTOP的上限可以很高。







3.3 新建内存空间 Creating an address space

    xv6中大多与VM有关的内容全部放在了kernel/vm.c中。其中最重要的struct是pagetable_t。它指向一个根PTP。kernel的该结构叫做kernel_pagetable，进程的该结构存放在各自的proc结构体中。最重要的函数有两个：walk(),mappage。前者为VA找到对应的PTE，后者为一个VA安装PTE/映射。注： 一vm开头的函数通常是kernel相关，以yvm开头的通常是用户进程相关，其他的共享。

    在开机早起，分页机制为启动，即不使用VM，直接使用PM。main会调用kvminit()，它创建一个根PTP并赋值给kernel的pagetable_t。实际上，kvminit()，kvmmake()是对一系列函数的封装。

void kvminit(void){
  kernel_pagetable = kvmmake();
}

kvmmake最开始使用kalloc()真正地在物理内存中创建了一部分供根PTP使用。之后多次调用kvmmap()为Fig.4中的各个部分分配PTE，指向PA的对应位置（比如硬件部分，KERNBASE，PHYSTOP）。最后调用一个函数为每个进程设置kstack与guard。

pagetable_t kvmmake(void){
  pagetable_t kpgtbl;

  kpgtbl = (pagetable_t) kalloc();
  memset(kpgtbl, 0, PGSIZE);

  // uart registers
  kvmmap(kpgtbl, UART0, UART0, PGSIZE, PTE_R | PTE_W);

  //省略多次调用kvmmap...

  // map kernel text executable and read-only.
  kvmmap(kpgtbl, KERNBASE, KERNBASE, (uint64)etext-KERNBASE, PTE_R | PTE_X);


  // map kernel stacks
  proc_mapstacks(kpgtbl);
  
  return kpgtbl;
}

但是kvmmap()也只是对mappage()的封装:

void kvmmap(pagetable_t kpgtbl, uint64 va, uint64 pa, uint64 sz, int perm)
{
  if(mappages(kpgtbl, va, sz, pa, perm) != 0)
    panic("kvmmap");
}

mappage()接收参数如物理内存地址，虚拟内存地址，大小等等。walk()在mappage()中被使用，它通过VA来找对应的PTE，并且把这个PTE的PA返回到mappage()。后者会通过内存地址就该这个PTE的内容（PPN，权限等）。walk()寻找PTE的过程就是上述的三段查找。但对应的PTP可能不存在（即上级PTE为非valid），比如由于最开始只有根PTP。此时如果walk()接收到了特定的参数，它就会分配新的PTP给这个PTE。

    在布局完成后，main()会调用kvminithart()。

void kvminithart()
{
  w_satp(MAKE_SATP(kernel_pagetable));
  sfence_vma();
}

它的工作很简单但是巨大的转折。它将kernel的根PTP放在寄存器satp中，这标志着CPU开始完全使用VM（对kernel没影响，因为它使用直接映射）。之后main()会调用procinit()来创建kstack和guard（同样通过kvmmap()）。procinit()最后会使用kvminithart()，把更新后的PT加载到satp，然后使用sfence.vma()晴空TLB缓存，准备进入用户空间。







3.4 物理内存分配器 Physical memory allocator

    分配与释放物理内存是kernel的工作。kernel分配kernel结束位置到PHYSTOP之间的区域。每次分配与释放的大小为4096Bytes，即一个page。通过特定的数据结构，kernel便可知哪个page可用。这个数据结构是一种自由表（Free List）。

自由表：

自由表被用来作为管理内存的数据结构存在，本质上只是一个链表。每一个空闲的page都是这个链表中的一个节点。对于kernel来说，它不需要储存这个链表，只需要储存其头节点，即某一个空闲的page，而这个头节点会指向“下一个”空闲page，下一个空闲的page又会指向另一个空闲的page…由于形如page.nextPage的属性存放在每个page的内部，它们不会占用kernel的内存。

    **分配器（Allocator）**实际上只是对kalloc.c的抽象（分配器同时负责分配与释放）。其中最重要的结构体属性是kmem，它有两个属性：

struct {
  struct spinlock lock;
  struct run *freelist;
} kmem;

后者即自由表的头节点。每个page都有自己的run结构体，储存着下一个可用的page。

    在main.c中，kinit会被调用：这个调用发生在对虚拟内存的初始化之前。kinit，与其他和内存分配/释放相关的重要内容大多定义在kalloc.c文件中，组成所谓的分配器。kinit实际上只是封装了两个函数：

void kinit() {
  initlock(&kmem.lock, "kmem");
  freerange(end, (void*)PHYSTOP);
}

前者将在第六章介绍，后者用来释放多个连续的page，其中第一个参数是释放目标的起始地址，后者是终止地址。由于kinit的目的是初始化从kernel结束到物理内存终点的所有内存到数据结构中，所以第一个参数是end[]，一个被定义在kernel.ld中的列表，储存着kernel结束后的第一个地址，而第二个参数PHYSTOP就是物理内存的终点。由于单次释放/分配一个page，freerange必然不是将参数所定义的内存区间一次性释放，而是以循环的方式多次调用kfree()来讲连续的pages一个接一个释放。为了保证每次释放的都是完整4096Bytes且规整（即不从某个page的中间释放）的page，在调用kfree之前，freerange调用PDROUNDUP函数（一个定义在defs.c里的宏定义）来检查参数所指向的地址是否是一个可行的page。在kinit之前，分配器没有课管理的page，但之后便有了。

    kfree的过程：kfree接受一个指针作为参数，该参数会被检查，通过之后调用memset把这个page中的内容覆盖为1，让page在再次被使用时无法访问到先前保存的内容。之后会把这个void指针强转成一个run类型的指针/结构体。这个指针的next属性将指向kmem中的自由表原头节点，且把头节点指向这个run指针：最新释放的page在整个自由表的最前面。

    分配器有两种对地址的理解。1.整数：当使用freerange时，对地址做算术运算来的相邻的oage的地址。2.指针：在自由表中，一个page的地址有被保存在先于它的page中，以一个指针的方式指向它，实现了page之间的连接。

void kfree(void *pa) {
  struct run *r;
  //检查地址是否合法
  if(((uint64)pa % PGSIZE) != 0 || (char*)pa < end || (uint64)pa >= PHYSTOP)
    panic("kfree");

  //清空page内容
  memset(pa, 1, PGSIZE);

  r = (struct run*)pa;

  //不需要关心lock相关
  acquire(&kmem.lock);
  //调换自由表头
  r->next = kmem.freelist;
  kmem.freelist = r;
  release(&kmem.lock);
}






3.5 进程内存空间 Process address space

    每个进程都有自己的内存空间，虽然RISC-V使用了39位虚拟内存地址，但xv6只使用其后38为，所以一个进程最多可以有2^39Bytes（即256GB）的虚拟内存。进程的内存空间起始于0x0，上限为MAXVA当一个进程问xv6更多虚拟内存时，主要的函数是kalloc，它给进程分配指定数量的page，这些page会被加入到进程的PTE中。

虚拟内存的三个优势/应用

  1. 进程可以有自己的私有的内存空间，实现隔离。
  2. 进程会认为自己的内存空间是连续的。
  3. 进程的内存空间定不会有trampoline部分，这部分对应的一个page是所有进程共享的。

    栈是一个被exec创建的page，它的下面是一个guard page，与kstack的guard用法一致，但由于进程使用虚拟内存，进程的栈的guard可以直接放在正常的位置。

    图中是main(argc,argv)的内存布局。其栈从上到下分别是调用main时给的参数，以及一个指向这些参数的数组和参数数量。最后面的是返回main使用的地址，供程序计数器（即指令指针）使用。

    sbrk是一个系统调用，用来修改进程的内存大小。在xv6中通过函数growproc(int n)实现。这个函数接受一个参数，代表希望修改（增加或减小）的内存大小（Bytes）。若其为正数，growproc调用uvmalloc。后者对一些列条件进行检查，一次使用kalloc与mappage分别为进程取出一个可用的page并且把这个page放到PTE中，最终返回新的内存空间大小。若其为负数，growproc调用uvmdealloc。在对参数进行检查后调用uvmmap。后者接受三个重要的参数：分页表地址，目标新大小（小于原大小），需要见效的page页数。uvmmap通过使用walk来得到目标PTE，用kfree把PTE中的page释放（若页数大于一，则使用循环多次释放）。

注： 虽然growproc的参数可以是一个不等于4096的数字，但是修改内存大小的最小单位仍然是4096Bytes。比如n=10Bytes，那么这个进程会增加4096Bytes，若n=4097Bytes，那么这个进程会增加(4096*2)Bytes。若n=-10Bytes，那么这个进程内存不会减少，若n=4097Bytes，那么这个进程只会减少4096Bytes。（分别是uvmalloc与uvmdealloc中反应的机制）。 上述的函数大多在proc.c与vm.c中。







3.7 exec 系统调用 Exec

    当一个进程开始运行前，如第一章所说，启动该进程的进程会调用exec来让目标进程“代替”自己从而达到开启的目的。更详细的说，exec中会调用naimei函数来打开指定路径中的ELF文件。这个ELF有两个重要的结构体：ELF Header (struct elfhdr)与Program Header(struct proghdr)。前者包含着这个ELF的元数据，比如文件类型，proghdr的数量等。其中的uint magic将会和ELF_MAGIC（后者值为0x7F,'E','L','F'）对比：若两者一致，则会认为这个ELF是正确的（Well-formed）。后者则是对这个进程的**段（Segment）**的描述。每个段都会有自己的proghdr，其包含这个段的类型，大小等。**注：**xv6的ELF只有1个段（也就是一个proghdr），但其他系统通常有多个。exec随后会调用proc_pagetable()来创建一个只有trampoline的PT，并且把所有类型为LOAD的proghdr内容加载到内存。

$ objdump -p _init              

_init:	file format elf64-littleriscv

Program Header:
    LOAD off    0x00000000000000b0 vaddr 0x0000000000000000 paddr 0x0000000000000000 align 2**3
         filesz 0x0000000000000980 memsz 0x0000000000000998 flags rwx

这是一个第一个进程_init的Program Header，其中的LOAD是这个段的类型；off是一个偏移量，指的是从文件的开始到这个段的距离；vaddr, paddr分别指该段在虚拟内存和物理内存的起始点，但进程运行在虚拟内存，后者被忽略；filesz指的是这个段的内存大小，而memsz指的是需要分配的内存大小：通常后者大于前者，两者之间的空间供全局变量使用，初始化为0。

    之后exec开始分配内存。通过调用kalloc，这个进程获得一个ustack。传入的参数会被一个接一个从上到下放入这个栈中。最终最下层的三个元素分别为一个假的返回地址，argc，与argv，参考Fig.5 进程的内存布局。

    在配置PT的过程中，若exec发现任何错误，就会返回-1，并且会回到调用exec的进程中。由于布置内存的过程不一定百分之百成功，而exec会最终把调用它的进程释放掉，保险的方法是先不让之前的进程被释放，等新的进程的内存相关的不知结束后便释放。由于exec是把一个外部的ELF文件假如内存中，若ELF是恶意设计的，可能会让内核崩溃，所以存在保护措施，如：

if(ph.vaddr + ph.memsz < ph.vaddr)
      goto bad;

通过计算ELF某段的大小是否在加载到内存中后会超过64bits上限导致溢出为一个更小的数字，防止通过溢出达成的恶意行为。（不过由于虚拟内存的隔离，这种溢出也难以对内核产生效果）。
