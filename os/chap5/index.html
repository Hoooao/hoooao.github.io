导言：

    驱动（Device Driver）本质上是管理某个设备的代码。他告诉该设备去执行什么操作，处理中断，与进程进行I/O交互等。由于并发性（Concurrency），与设备的借口通常很繁琐，导致驱动的设计复杂。

    一个设备假如需要被CPU“注意”，它一定有能力发出中断。在发出中断后，中断会被当作Trap处理。不同的设备有自己的中断处理代码。在xv6中，devintr()通过判断是哪个设备发出中断，然会会调用该设备对应的代码。

    通常，一个驱动可以分为 上下两部分（Top/bottom half）。上部代码运行于进程的内核线程，表现为这个进程使用系统调用：比如对控制台的输入输出。下部代码指的是中断期间的代码，比如上述的中断处理代码。上部代码在发送了执行某一个操作的请求后进入等待，此时设备开始运行。在设备运行完毕后，它会发出一个设备中断。此时对应的处理代码（下部代码）会知道什么操作完成了，并且在需要的情况下叫醒等待中的进程，然后让硬件去执行下一个操作。本章通过控制台介绍这一过程。







5.1 控制台输入 Code:Console input

    控制台的输入输出是通过硬件UART实现的。一定程度上可以把UART看作控制台本身（控制台驱动管理UART）。在qemu中，UART是通过代码模拟的。在现实中，UART将会链接RS232接口：

在xv6中，UART连接着键盘和显示器。回忆第三章中的内存空间，UART是作为一段物理内存地址暴露给软件的。软件可以通过内存地址访问UART的 内存映射的控制寄存器（Memory-mapped control registers）：UART开始于UART0，即0x10000000；每个寄存器占1byte，通过偏移量访问。这些寄存器功能不一，比如：

  1. LSR：指示是否传入的char在等待软件接收
  2. RHR：保存传入的char，软件从这里读取数据；每读取一个，UART就会删除被读取的；当没有待读取的数据时，LSR被清空
  3. THR：UART的传输与接收是相当独立的：当THR接受到1Byte，UART通过别的寄存器传输。[2]

    在启动xv6的过程中，main会调用consoleinit去初始化UART（consoleinit只封装uartinit，后者给其寄存器赋值），让UART在每接受1byte数据时就出发一个接收中断（Receive interrupt）。

void consoleinit(void) {
  initlock(&cons.lock, "cons");
    //调用uartinit()
  uartinit();
  //右侧的两个是console.c定义的函数
  devsw[CONSOLE].read = consoleread;
  devsw[CONSOLE].write = consolewrite;
}

void uartinit(void){
  // disable interrupts.
  WriteReg(IER, 0x00);

  //...设置寄存器...

  // enable transmit and receive interrupts.
  WriteReg(IER, IER_TX_ENABLE | IER_RX_ENABLE);

  initlock(&uart_tx_lock, "uart");
}

在shell启动前，其“壳子进程”会开启两个fd，指向控制台（init.c）。

//init.c
if(open("console", O_RDWR) < 0){
    mknod("console", CONSOLE, 0);
    open("console", O_RDWR);
  }
  dup(0);  // stdout
  dup(0);  // stderr

在通过read读取控制台数据时，最终该系统调用会使用consoleread（console.c）。它只有在完整地被输入一行数据后才会把字符串放到特定位置。在此之前，它会进入sleep（第七章）。当用户输入一个char时，UART会触发一个中断，激活xv6中的Trap处理机制。通过scause，devintr可以知道中断触发的原因，同时使用另一个硬件PLIC来看到是哪个硬件发出了中断后。会调用指定的处理代码：这个例子中是uartintr。

if((scause & 0x8000000000000000L) &&
     (scause & 0xff) == 9){
    // this is a supervisor external interrupt, via PLIC.

    // irq indicates which device interrupted.
    int irq = plic_claim();

    if(irq == UART0_IRQ){
      uartintr();
    } else if(irq == VIRTIO0_IRQ){
      virtio_disk_intr();
    } else if(irq){
      printf("unexpected interrupt irq=%d\n", irq);
    }

    // the PLIC allows each device to raise at most one
    // interrupt at a time; tell the PLIC the device is
    // now allowed to interrupt again.
    if(irq)
      plic_complete(irq);

    //..timer interrup...

    return 1;
  }

uartintr会不断地读取用户输入到UART（控制台）的数据，把它们移交到consoleintr。

void
uartintr(void)
{
  //通过uartgetc读取输入
  while(1){
    int c = uartgetc();
    if(c == -1)
      break;
    consoleintr(c);
  }

  // send buffered characters.
  acquire(&uart_tx_lock);
  uartstart();
  release(&uart_tx_lock);
}

后者会不断地把前者送来的数据添加到一个buffer（con.buf）中，直到收到指示一行结束的符号。然后consoleintr会叫醒consoleread，后者开始把传来的整行数据复制到目标用户空间并返回用户空间。







5.2 控制台输出 Code: Console output

    当一个对控制台的写入（write）被使用时，这个系统调用最终会走到uartputc。由于控制台驱动维护着一个buffer uart_tx_buf，进程不需要等待设备传输完1Byte后再次传输：uartputc的功能就是不断接受来自进程的输出，并把它们放入这个buffer中：uartputc唯一进入等待的情况是，buffer满了。在第一个Byte进入uartputc时，它就会调用uartstart。

//uartputc的一部分

while(1){
    if(uart_tx_w == uart_tx_r + UART_TX_BUF_SIZE){
      // buffer满了
      sleep(&uart_tx_r, &uart_tx_lock);
    } else {
      uart_tx_buf[uart_tx_w % UART_TX_BUF_SIZE] = c;
      uart_tx_w += 1;
      //首次调用uartstart
      uartstart();
      release(&uart_tx_lock);
      return;
    }
  }

后者会从buffer中得到数据，向控制台传输1Byte。当uartstart传送完1Byte，它会触发一个中断。由于中断的发生，又会进入uartintr，它会检查数据真的传输完毕了（通过调用uartgetc，如果LSR为空，返回-1，参考uartintr代码）。并且再次调用uartstart去buffer中传输数据（所以它在top/bottom-half中都被调用）。所以只有第一个Byte的uartstart是从uartputc中调用的，之后的uartstart都是在使用（通过前面1Byte的输出造成的中断而产生的）uartintr中的uartstart。 注： uartstart会叫醒uartputc如果它因buffer满而睡眠。且它的传输依靠于传递数据给THR。

    从上面可以看出，软件和设备的 解耦（Decoupling） 是通过buffer和中断实现的。在这种解耦中，软件不需要等设备执行完毕才去执行下一个操作，设备也可以处理输入即使没有软件在读取它：下一个read总会读到。这种策略被叫做I/O并发性。







5.3 驱动并发 Concurrency in drivers

    并发性有三个危险：

  1. 两个进程在两个CPU中同时调用了同一个驱动（访问同一个设备）

  2. 某个硬件触发了一个设备中断，但该CPU已经在运行驱动了（比如consoleread）。

  3. 在consoleread时，一个硬件可能会对另一个CPU发起一个中断。

他们通过 锁（Lock） 解决（第六章）。

    另外，当一个进程等待设备运行操作结束，在设备发出中断时这个进程可能会不在运行，另一个进程在运行。此时的中断处理什么也做不了。比如当一个进程使用read时，处理设备中断的代码并不能使用当前的进程的PT做任何事情：这就是区分上下部代码的意义。这种情况下下部代码能做的很少，通过唤醒上部代码，让上部去解决。







5.4 时钟中断 Timer interrupts

分时的存在由 时钟（Timer/clock） 硬件发出的**时钟中断（Timer interrupt）**实现。xv6使用时钟中断去维护它的时钟和切换进程。在usertrap与kerneltrap中的yield就是导致线程切换的函数。xv6使用程序去模拟时钟。由于RISC-V要求时钟中断的处理代码必须要运行在机器模式中，所以它不能像其他trap中在SM中直接解决。start.c中初始化了时钟并设置了出发中断的周期（1/10s）与一个scratch area，用来当作trapframe存放寄存器和重要的地址。最后把mvec设置成timervec并允许机器模式下的中断。

     时钟中断以时间为单位被触发，无论CPU当前运行在内核或者用户空间，它都可以出现。所以对时钟中断的处理要很小心，防止内核的损坏。基本策略是：让RISC-V发出一个软件中断（Software interrupts），走过正常的Trap处理流程，对内核进行保护。这个软件中断也会进入devintr处理。通过scause，devintr可以知道这实际上是一个时钟中断，然后返回2，在Trap处理函数中触发yield（无论usertrap还是kerneltrap都会检查devintr的返回值）。这个过程中，timevec会保存寄存器于scratch area中。并告诉硬件CLINT（通过内存映射的设备地址找到，这个地址保存在scratch area）什么时候去执行下一次中断，然后让RISC-V发出一个软件中断。这个软件中断会快速返回，之后timevec会恢复寄存器，并且返回，
