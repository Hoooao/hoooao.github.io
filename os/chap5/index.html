导言：

    驱动（Device Driver） 本质上是管理某个设备的代码。它告诉该设备去执行什么操作，处理中断，与进程进行I/O交互等。由于并发性（Concurrency），与设备的接口通常很繁琐，导致驱动的设计复杂。

    一个设备假如需要被CPU“注意”，它一定有能力发出中断。在发出中断后，中断会被当作Trap处理。与系统调用和报错不同：

  1. 外部设备中断是 异步（Asynchronous） 的，也就是说硬件所产生的中断可能和房钱正在运行的进程完全没有关系。
  2. 设备中断拥有更多的并发性：多设备同时运行，比如UART+网卡+CPU同时运行
  3. 可编程的设备：每种设备都有自己的寄存器，使用说明，功能等相关的编程手册

不同的设备有自己的中断处理代码。在xv6中，devintr()通过判断是哪个设备发出中断，并调用该设备对应的代码，这里指的主要是外部设备中断而非时钟中断和软件中断（软件中断指的是一个CPU给另一个CPU发送中断）。

    通常，一个驱动可以分为 上下两部分（Top/bottom half）。上部代码运行于进程的内核线程，表现为这个进程使用系统调用：比如对控制台的输入输出。下部代码指的是中断期间的代码，即上述的中断处理代码。上部代码在发送了执行某一个操作的请求后进入等待，此时设备开始运行。在设备运行完毕后，它会发出一个设备中断。此时对应的处理代码（下部代码）会知道什么操作完成了，并且在需要的情况下叫醒等待中的进程（上部代码），然后让硬件去执行下一个操作。本章通过控制台介绍这一过程。







5.1 控制台输出 Code: Console output

    控制台的输入输出是通过硬件UART实现的，控制台驱动管理UART。在qemu中，UART是通过代码模拟的。在现实中，UART将会连接RS232接口：

在xv6中，UART连接着键盘和显示器。回忆第三章中的内存空间，UART是作为一段物理内存地址暴露给软件的。软件可以通过内存地址访问UART的 内存映射的控制寄存器（Memory-mapped control registers）：UART开始于UART0，即0x10000000；每个寄存器占1byte，通过偏移量访问。这些寄存器功能不一，比如：

  1. LSR：指示是否传入的char在等待软件接收
  2. RHR：保存传入的char，软件从这里读取数据；每读取一个，UART就会删除被读取的；当没有待读取的数据时，LSR被清空
  3. THR：当THR接受到1Byte，UART通过Data Trandmission Line输出数据。[2]

可以从图中看到，同一个地址在进行读写操作的时候，可以被认为是不同的寄存器。

    RISC-V有一些为中断服务的寄存器（不是上述UART的寄存器）：

SIE: Supervisor Interrupt Enable: 其中使用3个bits，分别指示当前CPU是否允许外部设备中断，软件中断，和时钟中断。 SSTATUS: Supervisor Status: 一个bit，指示当前CPU是否允许中断。 SIP: Supervisor Interrupt Pending: CPU通过该寄存器去看什么中断进来了。

    本节主要介绍中断的初始化和第一个shell中的$的产生

main()
{
  if(cpuid() == 0){
    consoleinit();
    printfinit();
    printf("\n");
    printf("xv6 kernel is booting\n");
    printf("\n");
    kinit();         // physical page allocator
    kvminit();       // create kernel page table
    kvminithart();   // turn on paging
    procinit();      // process table
    trapinit();      // trap vectors
    trapinithart();  // install kernel trap vector
    plicinit();      // set up interrupt controller
    plicinithart();  // ask PLIC for device interrupts
    binit();         // buffer cache
    iinit();         // inode table
    fileinit();      // file table
    virtio_disk_init(); // emulated hard disk
    userinit();      // first user process
    __sync_synchronize();
    started = 1;
  } else {
    while(started == 0)
      ;
    __sync_synchronize();
    printf("hart %d starting\n", cpuid());
    kvminithart();    // turn on paging
    trapinithart();   // install kernel trap vector
    plicinithart();   // ask PLIC for device interrupts
  }

  scheduler();        
}

    在启动xv6的过程中，首先会进入start.c来初始化一些寄存器，比如上述的SIE。之后会开始运行main。通过最开始的if/else语句，使得只有第0号CPU去进行这些硬件的初始化工作（不需要让其他CPU做重复的工作）。main会调用consoleinit去初始化UART（consoleinit只封装uartinit，后者给其寄存器赋值），让UART在每接受1byte数据时就出发一个接收中断（Receive interrupt）。UART是第一个初始化的外部设备。

void consoleinit(void) {
  initlock(&cons.lock, "cons");
    //调用uartinit()
  uartinit();
  //右侧的两个是console.c定义的函数
  //这两个函数未来会在对控制台的W/R中调用
  devsw[CONSOLE].read = consoleread;
  devsw[CONSOLE].write = consolewrite;
}

void uartinit(void){
  // disable interrupts.
  WriteReg(IER, 0x00);

  //...设置其他Fig.2中的一部分寄存器...

  // enable transmit and receive interrupts.
  WriteReg(IER, IER_TX_ENABLE | IER_RX_ENABLE);

  initlock(&uart_tx_lock, "uart");
}

     理论上此时UART就可以发出中断了，但是sstatus还没有被设置，所以相当于当前CPU禁止中断。之后会有一段对控制台的文本显示输出，但是由于现在中断并不允许，实际上这里的输出是跳过了中断，直接输出到UART转而输出到控制台的。这里的printf是在kernel/printf.c中定义的特殊的printf，通过uartputc_sync函数直接进行输出。而通常使用的printf实际上是user/printf.c中定义的，后者需要进入中断处理。这一特殊的printf的使用场景就是这些输出文本和对用户未来输入的echo（即把用户的输入“输出”到显示器上，这个“输出”同样不需要经过中断）。

     仍然在main中，会调用plicinit去允许PLIC接受来自UART的 中断请求（IRQ：Interrupt request）。PLIC是一个硬件，作为处理中断请求的中转站。如上图所示，PLIC可以把IRQ任意分配给人以一个CPU核心，分配的原则是通过对PLIC的设计决定的（PLIC中没有代码，只有电路）。如果当前所有核心都无法处理中断（比如禁止了中断），那么PLIC会暂时保存这一中断，直到有核心可以处理它。

     之后main中的plicinithart会被调用。这个函数设置“PLIC可以给当前CPU发送哪些硬件产生的IRQ”。

     main中每个CPU最后都会调用scheduler()，它会调用intr_on，后者实际上设置了sstatus。此时如果PLIC中等待的中断会被处理。 而0号CPU会调用userinit，然后调用init启动shell。在shell启动前，其“壳子进程”会开启三个fd，指向控制台（init.c）：输入输出和报错全部指向控制台。

//init.c
if(open("console", O_RDWR) < 0){
    mknod("console", CONSOLE, 0);
    open("console", O_RDWR);
  }
  //此时fd 1，2闲置，所以被直接分配
  dup(0);  // stdout
  dup(0);  // stderr

shell运行getcmd，这个函数会调用fprintf，在内部调用putc，putc调用write系统调用，这个系统调用使用filewrite函数，这个函数会检查write传入的fd是什么，如果是控制台的话，会调用consolewrite，这个函数内部调用uartputc，参数是一个字符(consolewrite循环调用uartputc)，uartputc会把这个字符放入它维护的buffer uart_tx_buf（32Bytes） 中，并且每次都会调用uartstart。uartstart让设备真正开始干活（通过写入THR寄存器来让UART把这个字符传输到目标设备）并发起中断。

//uartputc的一部分

while(1){
    if(uart_tx_w == uart_tx_r + UART_TX_BUF_SIZE){
      // buffer满了
      sleep(&uart_tx_r, &uart_tx_lock);
    } else {
      //数据保存到buffer
      uart_tx_buf[uart_tx_w % UART_TX_BUF_SIZE] = c;
      uart_tx_w += 1;
      //首次调用uartstart
      uartstart();
      release(&uart_tx_lock);
      return;
    }
  }

注：uartputc唯一进入等待的情况是buffer满了。由于buffer的存在，输出访不需要等传输完毕再输出下一个Byte。

void
uartintr(void)
{
  //通过uartgetc读取输入
  while(1){
    int c = uartgetc();
    if(c == -1)
      break;
    consoleintr(c);
  }
  //如果输入不存在，说明是输出操作
  // send buffered characters.
  acquire(&uart_tx_lock);
  uartstart();
  release(&uart_tx_lock);
}

     当uartstart传送完1Byte，它会触发一个中断。由于中断的发生，又会进入uartintr，它会检查数据真的传输完毕了（通过调用uartgetc，如果LSR为空，返回-1：此时uartintr将不会执行consoleintr）。并且再次调用uartstart去buffer中传输数据（所以它在top/bottom-half中都被调用）。所以只有第一个Byte的uartstart是从uartputc中调用的，之后的uartstart都是在使用（通过前面1Byte的输出造成的中断而产生的）uartintr中的uartstart。

void uartstart() {
  while(1){
    if(uart_tx_w == uart_tx_r){
      // 两个变量相等buffer为空
      return;
    }
    
    if((ReadReg(LSR) & LSR_TX_IDLE) == 0){
      // the UART transmit holding register is full,
      // so we cannot give it another byte.
      // it will interrupt when it's ready for a new byte.
      return;
    }
    
    //读取数据并且增加变量
    int c = uart_tx_buf[uart_tx_r % UART_TX_BUF_SIZE];
    uart_tx_r += 1;
    
    // maybe uartputc() is waiting for space in the buffer.
    wakeup(&uart_tx_r);
    
    //写入THR，用来传输这个c
    WriteReg(THR, c);
  }
}

注： uartstart会叫醒uartputc如果它因buffer满而睡眠。在这个例子中只会输出一次，因为只有一个字符$。







5.2 控制台输入 Code: Console input

     控制台的输入行为也是类似的，不同的只是几个函数的调用。在通过read读取控制台数据时，最终该系统调用会使用fileread，在其中调用consoleread。它只有在完整地被输入一行数据后才会把字符串放到特定位置。在此之前，它会进入sleep（第七章）。当用户通过键盘输入一个char时，UART会触发一个中断，激活xv6中的Trap处理机制。通过scause，devintr可以知道中断触发的原因，同时使用PLIC来看到是哪个硬件发出了中断后，会调用指定的处理代码：这个例子中是uartintr。uartintr会不断地读取用户输入到UART（控制台）的数据，把它们移交到consoleintr（通过uartgetc得到输入的字符）。后者会一个一个地把前者送来的字符添加到一个buffer（con.buf，128Bytes）中，直到收到指示一行结束的符号。然后consoleintr会叫醒consoleread，后者开始把传来的整行数据复制到目标用户空间并返回用户空间。注： consoleintr同时也会调用consputc，这个函数echo输入的命令“输出”到显示器： 也是通过uartputc_sync，不需要中断。

    uartgetc() 是通过读取寄存器实现读取输入的：

int uartgetc(void) {
  if(ReadReg(LSR) & 0x01){
    // input data is ready.
    return ReadReg(RHR);
  } else {
    return -1;
  }
}






5.3 驱动并发 Concurrency in drivers

    并发性有三个危险：

  1. 两个进程在两个CPU中同时调用了同一个驱动（访问同一个设备）

  2. 某个硬件触发了一个设备中断，但该CPU已经在运行驱动了（比如consoleread）。

  3. 在consoleread时，一个硬件可能会对另一个CPU发起一个中断。

他们通过 锁（Lock） 解决（第六章）。

    另外，当一个进程等待设备运行操作结束，在设备发出中断时这个进程可能会不在运行，另一个进程在运行。此时的中断处理什么也做不了。比如当一个进程使用read时，处理设备中断的代码并不能使用当前的进程的PT做任何事情：这就是区分上下部代码的意义。这种情况下下部代码能做的很少，通过唤醒上部代码，让上部去解决。







5.4 时钟中断 Timer interrupts

分时的存在由 时钟（Timer/clock） 硬件发出的 时钟中断（Timer interrupt） 实现。xv6使用时钟中断去维护它的时钟和切换进程。在usertrap与kerneltrap中的yield就是导致线程切换的函数。xv6使用程序去模拟时钟。由于RISC-V要求时钟中断的处理代码必须要运行在机器模式中，所以它不能像其他trap中在SM中直接解决。start.c中初始化(timerinit())了时钟并设置了出发中断的周期（1/10s）与一个scratch area，用来当作存放寄存器和重要的地址的空间地址保存在mscratch）。最后把mtvec设置成timervec并允许机器模式下的中断。

timerinit()
{
  // 通过这个函数或渠道当前cpu的id，每个cpu都有自己的时钟
  int id = r_mhartid();

  // ask the CLINT for a timer interrupt.
  int interval = 1000000; // cycles; about 1/10th second in qemu.
  *(uint64*)CLINT_MTIMECMP(id) = *(uint64*)CLINT_MTIME + interval;

  //就是sscratch的机器模式版本的寄存器
  // prepare information in scratch[] for timervec.
  // scratch[0..2] : space for timervec to save registers.
  // scratch[3] : address of CLINT MTIMECMP register.
  // scratch[4] : desired interval (in cycles) between timer interrupts.
  uint64 *scratch = &timer_scratch[id][0];
  scratch[3] = CLINT_MTIMECMP(id);
  scratch[4] = interval;
  //这个函数把scratch写入内存
  w_mscratch((uint64)scratch);

  //设置时钟中断处理， timervec是之前定义的一个函数，这里当作地址存入。
  w_mtvec((uint64)timervec);

  // enable machine-mode interrupts.
  w_mstatus(r_mstatus() | MSTATUS_MIE);

  // enable machine-mode timer interrupts.
  w_mie(r_mie() | MIE_MTIE);
}

     时钟中断以时间为单位被触发，无论CPU当前运行在内核或者用户空间，它都可以出现。所以对时钟中断的处理要很小心，防止内核的损坏。基本策略是：让RISC-V发出一个软件中断（Software interrupts），走过正常的Trap处理流程，对内核进行保护。这个软件中断也会进入devintr处理。通过scause，devintr可以知道这实际上是一个时钟中断，然后返回2，在Trap处理函数中触发yield（无论usertrap还是kerneltrap都会检查devintr的返回值）。这个过程中，timervec会保存寄存器于scratch area中。并告诉硬件CLINT（通过内存映射的设备地址找到，这个地址保存在scratch area）什么时候去执行下一次中断，然后让RISC-V发出一个软件中断。这个软件中断会快速返回，之后timervec会恢复寄存器，并且返回，

//devintr中处理时钟中断的部分：
else if(scause == 0x8000000000000001L){
    // software interrupt from a machine-mode timer interrupt,
    // forwarded by timervec in kernelvec.S.

    //xv6支启动了1个CPU，即0
    if(cpuid() == 0){
      //增加ticks
      clockintr();
    }
    
    // acknowledge the software interrupt by clearing
    // the SSIP bit in sip.
    w_sip(r_sip() & ~2);

    return 2;
  }

5.6 补充

     处理中断的过程：

  1. 假如SIE被设置了，硬件会清空SIE，临时禁止任何其他中断进入当前核心，不然处理中断的代码可能被中断。

  2. 在usertrap中，调用devintr，通过调用特定函数得到当前和行政在处理的IRQ（UART的这个值是10）。

if((scause & 0x8000000000000000L) &&
     (scause & 0xff) == 9){
    // this is a supervisor external interrupt, via PLIC.

    // irq indicates which device interrupted.
    int irq = plic_claim();

    if(irq == UART0_IRQ){
      uartintr();
    } else if(irq == VIRTIO0_IRQ){
      virtio_disk_intr();
    } else if(irq){
      printf("unexpected interrupt irq=%d\n", irq);
    }

    // the PLIC allows each device to raise at most one
    // interrupt at a time; tell the PLIC the device is
    // now allowed to interrupt again.
    if(irq)
      plic_complete(irq);

    //..timer interrup...

    return 1;
  }
  1. 通过检查IRQ的值，调用uartintr，它又调用consoleintr或者uartstart。

    软件和设备的 解耦（Decoupling） 是通过buffer和中断实现的。在这种解耦中，软件不需要等设备执行完毕才去执行下一个操作，设备也可以处理输入即使没有软件在读取它：下一个read总会读到。这种策略被叫做I/O并发性。就如控制台的输出输入一般，有了两个buffer，通常输出和输入都不用等待。

     中断和并发：

  1. 设备和CPU平行进行：UART在读写的时候，CPU可能在处理shell的Trap。这被叫做producer/consumer parallelism。比如在UART的驱动中uart_tx_buf有两个pointer：uart_tx_r,uart_tx_w。每当有一个写入操作（即使用uartputc），uart_tx_w指针就会后移，如果uart_tx_w指向了buffer末尾，那么就开始等待。所以在输出一个$的例子中，shell就是producer，因为它写入UART，而uartintr则是consumer，通过调用uartstart来增加uart_tx_r，达到使用（consume）buffer的效果。

  2. 中断会暂停当前在运行的进程：对于用户进程来说问题不大，但是内核进程也可能被暂停，而有一些内核的代码可能要求原子化（要么完全发生，要么完全不发生），那么就需要暂时禁止中断。

  3. 驱动上下部也可以平行运行：上下部分别在不同的CPU运行，但由于这个驱动可能维护1个buffer，为了不导致并发问题，会使用lock。

     uartintr的行为类似一个中转站，它会检查是否有等待输入的内容，如果有，那么它会执行consoleintr，作为consumer；如果没有，那么会执行uartstart执行输出，作为producer。
