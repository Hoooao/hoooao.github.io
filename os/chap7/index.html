导言

    OS希望在同时运行多个进程，除了使用多核处理器，CPU需要实现分时。在理想状态下，分时对于进程来说应当是透明的，也就是说进程不应当知道分时的进行，或者说，进程会认为自己独享一个CPU。这被叫做将进程多路复用（Multiplexing） 在CPU上。

    xv6通过让CPU在不同的进程之间切换实现多路复用。这种进程切换主要体现在两种情形之下:

  1. 一个进程调用了sleep让另一个进程开始运行，或者一个进程调用了wakeup让另一个睡眠的进程继续运行（7.5）。

  2. 强制的分时，即时钟中断，让每个进程都有机会运行。

在真正实现这一特性时有六个困难：

  1. 如何切换进程。
  2. 如何让切换的过程对进程透明。
  3. 需要解决多个CPU同时切换进程可能引发竞争问题。
  4. 进程在退出之后的资源释放（一个进程不可能完全释放自己）。
  5. 需要让每个内核都知道自己运行的进程是谁，这样才能在正确的核心上使用系统调用。
  6. 实现sleep与wakeup。

注：多路复用，调度，和分时没必要在下文中区分。







7.1 上下文切换 Code: Context switch

    进程之间的切换本质上就是它们的**上下文（Context）**的切换。上下文包括进程的寄存器/代码/栈（实际上都由寄存器决定）。

    进程之间的切换主要包括下面4点：

  1. 从当前进程的用户空间跳转到它的内核线程（Trap）。

  2. 通过内核线程跳转到scheduler线程：该线程是每个CPU独有的线程，不共享。加入共享的话可能会出现多个CPU运行一个进程（栈）的非法情况。

  3. 通过scheduler线程跳转到目标进程的内核线程。

  4. 同样通过Trap机制跳转回目标进程的用户空间。

    这里的跳转，或者说上下文切换，实际上是针对寄存器数据的切换（寄存器的切换标志着运行的代码/栈的切换等等）。最重要的函数是swtch()，它是一个汇编程序，保存所有旧进程的callee-saved 寄存器(C编译器自动保存caller-saved寄存器)。它不会保存其pc（因为此时pc指向swtch中的指令，没意义），但是会保存ra寄存器，这个寄存器保存着swtch被调用的时候的地址。注：这里同样不能使用sepc，即使它没有被重写，它保存的也是用户线程的pc。

.globl swtch
swtch:
//a0,a1是传入的上下文结构体地址

        sd ra, 0(a0)
        sd sp, 8(a0)
        //...保存寄存器...
        sd s11, 104(a0)

        ld ra, 0(a1)
        ld sp, 8(a1)
        //...复原寄存器...
        ld s11, 104(a1)
        
        ret

和trapframe相仿，这些被保存的数据是通过偏移量来访问的。在保存完了旧进程的寄存器之后，swtch会复原新进程的寄存器，并读取其ra，作为运行结束后的返回地址。注：swtch使用的返回指令是ret，不是trap中常用的sret，所以不需要思考sepc/epc等等问题。     和其他重要的数据一样，这些进程的“上下文”（寄存器）被保存在内核中。context是每个进程的结构体中的一个属性，它也是一个结构体，定义在proc.h：

struct context {
  //返回地址ra和stack-pointer
  uint64 ra;
  uint64 sp;

  // callee-saved
  uint64 s0;
  //...省略...
  uint64 s11;
};

swtch它需要两个这样的结构体的地址作为参数，即旧进程的和新进程的上下文。（swtch不关心谁是新/旧，它只是在保存/恢复寄存器并返回）。

    思考一个可能的进程切换的路径:usertrap->yield->sched->swtch->scheduler->swtch->usertrap->userret. 这是一个时钟中断造成的分时，所以Trap处理中会调用yield。

// Give up the CPU for one scheduling round.
void
yield(void)
{
  struct proc *p = myproc();
  acquire(&p->lock);
  //切换状态
  p->state = RUNNABLE;
  //调用sched切换线程
  sched();
  release(&p->lock);
}

yield继续调用sched，后者会多这个进程做一些检查，然后调用swtch(&p->context, &mycpu()->context);:第一个参数是当前进程的上下文，后者是scheduler的上下文。

void
sched(void)
{
  int intena;
  struct proc *p = myproc();

  if(!holding(&p->lock))
    panic("sched p->lock");
  if(mycpu()->noff != 1)
    panic("sched locks");
  if(p->state == RUNNING)
    panic("sched running");
  if(intr_get())
    panic("sched interruptible");

  intena = mycpu()->intena;
  swtch(&p->context, &mycpu()->context);
  mycpu()->intena = intena;
}

    思考：为什么trapframe和context看起来很像，前者甚至更早保存了更多的寄存器，还要用context？

        因为两者保存的内容完全不一样：trapframe保存着进程用户空间在进入trap之前的状态，而后者是对进程内核空间状态的记录。两者必须协同才能实现进程切换。







7.2 调度 Scheduling

    本节主要介绍从旧进程的内核线程切换到scheduler(后文计作s)再切换到另一个进程的过程。 s线程实际上就是对scheduler函数的调用。那为什么说进程之间的跳转需要“跳转到s线程”，而不是“去调用scheduler函数”？因为实际上scheduler只调用了一次，后续的每一次运行都是这一个线程在工作，而非重新被调用。s作为一个中转站，决定着接下来运行哪一个进程。

    一个进程放弃CPU使用的过程（yield，sleep，exit都是类似的过程）：

  1. 获取自己的进程锁
  2. 释放其他锁
  3. 更新自身状态（p->state）
  4. 调用sched。

以yield为例。一个进程在放弃对CPU的使用之前，其状态会首先被修改。这个进程的状态会从running变成runnable。这里的进程锁十分关键，它会贯穿sched，swtch的执行。假如没有这个锁，可能在状态被修改之后，在swtch切换kstack之前的窗口期之间，另一个CPU会发现存它是runnable的进程并运行：在这段时间内，有两个CPU运行着同一个stack/进程，显然不对。这个锁保护着p->state,p->context中的不变量，让进程的切换安全。虽然下方释放了这个锁，但这个函数中的获取和释放并不是一对。通常来讲，一个线程会负责自己使用的锁的获取和释放，但是由于我们希望达成上述贯穿整个跳转的对进程的保护，这个规律必然会被破坏。

    旧进程会切换到s进程。

void
scheduler(void)
{
  struct proc *p;
  struct cpu *c = mycpu();
  
  c->proc = 0;
  for(;;){
    // Avoid deadlock by ensuring that devices can interrupt.
    intr_on();

    for(p = proc; p < &proc[NPROC]; p++) {
      acquire(&p->lock);
      if(p->state == RUNNABLE) {
        // Switch to chosen process.  It is the process's job
        // to release its lock and then reacquire it
        // before jumping back to us.
        p->state = RUNNING;
        c->proc = p;
        swtch(&c->context, &p->context);

        // Process is done running for now.
        // It should have changed its p->state before coming back.
        c->proc = 0;
      }
      release(&p->lock);
    }
  }
}

for(;;)保证了这个函数会一直运行（即使没有可以运行的其他进程），当旧进程切换到s之后，后者复原的ra会使得CPU从之前跳出s的位置继续运行，也就是swtch的位置。在归0进程后，会释放锁，此时释放的其实是旧进程在自己的yield中获取的锁: 变量p必定是旧进程，因为旧进程曾经以“新进程”的形态被s获取并保存在变量p中，恢复s的栈时必然会恢复这个变量（这里也体现出来每个CPU都有自己的s线程的好处，假如共享一个线程的话，这里的p就可能是别的CPU之前使用的了）。之后会执行内部循环体：for(p = proc; p < &proc[NPROC]; p++)，检索当前是否有runnable的进程，如果有，便获取这个进程，获取其锁，切换状态为running，设置“当前进程(c->proc)”为这个进程，并且调用swtch。这个swtch的第一个参数是s的上下文，第二个参数是新的进程的上下文。前者被保存使得下一次调度发生后仍然可以跳转到当前的位置。在swtch返回之后，会跳转到新进程的sched(然后yield)中，此时新进程的yield会释放一个锁，这个锁是在s中获取的那个，不是yield上方代码获取的锁。

     存在一个特殊情境：当一个进程第一次运行（第一次被调度）时，它保存的context.ra是forkret()。也就是说s会跳转到forkret而非sched（因为首次运行的进程显然还没能从Trap中进入过s）。 这个函数同样释放锁，并且调用usertrapret()返回用户空间。初始的context.ra是在allocproc中被设置的。但这个进程下一次进入s之前，就必将是从sched的入口进入了。

// will swtch to forkret.
void
forkret(void)
{
  static int first = 1;

  // Still holding p->lock from scheduler.
  release(&myproc()->lock);

  if (first) {
    // File system initialization must be run in the context of a
    // regular process (e.g., because it calls sleep), and thus cannot
    // be run from main().
    first = 0;
    fsinit(ROOTDEV);
  }

  usertrapret();
}

    一种去理解s的存在的视角是把它看作确保不变量在不为真的时候保证安全的代码。 两个不变量的例子：

  1. 当一个进程为running状态，那么一个时钟中断的yield必须能够把它安全的切换，这意味着CPU的寄存器必须有这个进程的寄存器数据，c->proc必须指向这个进程。

  2. 当proc为runnable时，它必须被安全地通过一个闲置的CPU的scheduler运行，这意味着p->context必须保存这进程的寄存器，且没有任何CPU运行于这个进程的内核线程，且没有任何一个CPU的c->proc指向它。

可以看到在整个跳转过程中，这些不变量常常不为真。







7.3 函数 myproc(), mycpu()

    xv6的很多内核操作常常会想使用当前进程的信息。当前进程以一个结构体的形式暴露给内核 (proc)。在单核处理器中，可以保存一个全局变量指向这个结构体。但是在多核处理器中，每个核心会运行不同的进程，此时可以使用每个核心的寄存器储存。

    xv6储存着每个cpu本身的结构体，这个结构体内储存着当前正在运行的进程。除了进程，CPU结构体还储存着s线程的context，检查是否开启中断的计时器noff。

// Per-CPU state.
struct cpu {
  struct proc *proc;          // The process running on this cpu, or null.
  struct context context;     // swtch() here to enter scheduler().
  int noff;                   // Depth of push_off() nesting.
  int intena;                 // Were interrupts enabled before push_off()?
};

RISC-V会知道各个CPU是什么，因为在开机的时候，mstart指令会给每个CPU一个hartid，这个id会被放入各自的tp寄存器中。通过hartid，CPU可以访问到自己的信息(通过mycpu())。对tp的处理需要小心。usertrapret会把tp，也就是hartid保存在trapframe中，这样用户空间就可以使用这个寄存器了，同样地，在uservec中会把tp复原，供trap使用hartid。看起来如果可以直接获取hartid会更方便，但是这个操作只允许在MM中使用。

    cpuid()通过读tp得到这个cpu的id，而mycpu()通过这个id可以找到对应的CPU的结构体。

// Must be called with interrupts disabled,
// to prevent race with process being moved
// to a different CPU.
int
cpuid()
{
  //读取tp
  int id = r_tp();
  return id;
}

// Return this CPU's cpu struct.
// Interrupts must be disabled.
struct cpu*
mycpu(void) {
  int id = cpuid();
  //cpus是内核维护的一个cpu结构体数组
  struct cpu *c = &cpus[id];
  return c;
}

    这两个函数的返回过程可能会由于分时而错误。比如一个内核线程调用了cpuid并得到了返回值，但是时钟中断发生：当它再次运行的时候，它运行在了不同的CPU上：得到的返回值和真实的正在运行的CPU将对不上。所以xv6要求只要使用mycpu，那么就一定要禁止中断，直到对cpu的结构体的使用结束后才再次允许中断。

    myproc就实现了上述的操作:

struct proc*
myproc(void) {
  //关闭中断
  push_off();
  struct cpu *c = mycpu();
  struct proc *p = c->proc;
  //视情况开启中断
  pop_off();
  return p;
}

它使用push_off/pop_off禁止中断并获取当前proc。使用myproc得到proc结构体后不需要关心它在哪







7.4 睡眠和唤醒 Sleep and wakeup

    上面所说的分时使得进程之间的切换对于进程来说是透明的，但是进程可能也会出现有意的切换: 一个进程可能会等待另一个进程调用某些函数去让某些条件为真，而后者可能会在完成某些指令之后让前者停止等待继续运行，即睡眠（Sleep） 与 唤醒（Wakeup）。这种行为叫做 Sequence coordination 或者 conditional synchronization。参考下面的例子：

100  struct semaphore {
101       struct spinlock lock;
102       int count;
103   };
104
105   void
106   V(struct semaphore *s)
107   {
          //通过锁来序列话对count的操作
108       acquire(&s->lock);
109       s->count += 1;
110       release(&s->lock);
111   }
75
112
113   void
114   P(struct semaphore *s)
115   {
          //通过自旋实现等待
116       while(s->count == 0)
117       ;
118       acquire(&s->lock);
119       s->count -= 1;
120       release(&s->lock);
121   }

semaphore结构体中保存着一个锁和一个计数器count，下面有两个方法：V增加计数器，当计数器不是0的时候，P减少计数器。循环可以帮助P实现这一“等待”。但是缺点很明显：假设V很少运行，那么P会一直自旋，这一自旋行为是对性能的浪费，被叫做busy wait （不过有的设备驱动中存在这种自旋，因为设计者知道这一等待并不会等太久）。每当V运行之后P再开始运行效果会更好。参考下面的代码:

1  void
2  V(struct semaphore *s)
3  {
4     acquire(&s->lock);
5     s->count += 1;
6     wakeup(s);
7     release(&s->lock);
8  }
9  void
10 P(struct semaphore *s)
11 {
12   while(s->count == 0)
13    sleep(s);
14   acquire(&s->lock);
15   s->count -= 1;
16   release(&s->lock);
17 }

这里使用了sleep和wakeup函数: 当P发现计时器为0时就进入睡眠，直到V增加计数器并且通过某种方式唤醒P，P再开始运行。逻辑上似乎没什么问题，但是仍然需要考虑并发带来的问题：假如P刚运行完了12行，检查count为0，可是在进入睡眠之前，V在另一个CPU上增加了计数且调用了wakeup。由于P已经检查完了count，纵使V在P踏进睡眠之前增加了计数器且调用了唤醒，P仍会进入睡眠，这被叫做唤醒丢失（Lost wake-up）。如果V之后又运行了一次就问题不大：P会被叫醒并且归零计数器。但假如V再也不运行了，P会永远睡眠且不会终止运行。注：sleep和wakeup的参数是一个等待频道（Wait channel），sleep(s)解释为该进程睡在s上，wakeup(s)将会唤醒所有睡在s上的进程，如果没有睡在s上的进程，就继续运行。

    本质的问题实际上是不变量被破坏：即P永远只在count==0的时候睡眠。根据之前节的说明，目前直觉的解决方案似乎是使用一个锁来保护这个不变量。参考代码：

300 void
301 V(struct semaphore *s)
302 {
303     acquire(&s->lock);
304     s->count += 1;
305     wakeup(s);
306     release(&s->lock);
307 }
308
309 void
310 P(struct semaphore *s)
311 {   //移动锁的位置，保护不变量
312     acquire(&s->lock);
313     while(s->count == 0)
314         sleep(s);
315     s->count -= 1;
316     release(&s->lock);
317 }

假设P已经获取了锁并处于运行314行之前的位置，此时假如V希望平行执行增加计数器，会获取锁而不能，便进入自旋。问题随之出现：虽然P在这个状态下可以保证不变量被保护时进入睡眠，但由于P在自旋，它也永远不会被唤醒。造成死锁。

    不过现在解决的路径更清晰了：我们需要用锁保障不变量可以复原，即当P休眠时count==0，也需要P在睡眠后V仍然可以运行，即锁会被释放，让V停止自旋。参考代码：

400 void
401 V(struct semaphore *s)
402 {
403     acquire(&s->lock);
404     s->count += 1;
405     wakeup(s);
406     release(&s->lock);
407 }
408
409 void
410 P(struct semaphore *s)
411 {
412     acquire(&s->lock);
413     while(s->count == 0)
414         sleep(s, &s->lock);
415     s->count -= 1;
416     release(&s->lock);
417 }

这里修改了sleep函数，第一个参数仍然是chan，即等待频道，第二个参数是上述的锁lk。在P进入睡眠后，sleep函数会自动释放传入的锁，从而让V解除自旋。这个锁被叫做条件锁（Condition lock），因为获取这个锁的原因是想要某个条件为真或修改这个条件。







7.5 代码：睡眠和唤醒 Code： Sleep and wakeup

    sleep重要的行为是把当前进程标记状态为SLEEPING并调用sched开启调度。wakeup会把在同一频道上的所有SLEEPING的进程标记为RUNNABLE并开始调度。通常频道chan使用的数字是一个两个进程都能访问到的数据，xv6中常用的是与等待本身相关的数据结构。比如uartputc里使用&uart_tx_r作为等待输出时的sleep的频道。调用sleep的进程会在调用它之前获取一个条件锁（为了避免丢失唤醒）lk，并且sleep内部还会调用这个进程的进程锁 p->lock 用来保护进程的不变量（为调度服务），这个进程锁会在scheduler中释放。所以sleep实际上有一段时间会持有两个锁。lk保障了在它被释放之前，没有其他进程可以使用wakeup唤醒这个进程导致唤醒丢失，因为在调用wakeup的进程中也必须要持有这个条件锁，比如V。

void
sleep(void *chan, struct spinlock *lk)
{
  struct proc *p = myproc();
  acquire(&p->lock); 
  //释放条件锁，允许wakeup
  release(lk);
  //修改状态
  p->chan = chan;
  p->state = SLEEPING;

  //调度
  sched();

  //清理
  p->chan = 0;
  //释放
  release(&p->lock);
  //再次获取条件锁
  acquire(lk);
}

一旦lk被释放，其他进程就可以去唤醒这个进程了，但是由于p->lock没有被释放，使用wakeup的进程会在某一时刻进入自旋，直到前者释放进程锁。sleep的过程：记录频道，改变进程的状态为SLEEPING，调用sched开始调度。

void
wakeup(void *chan)
{
  struct proc *p;
  //使用一个循环遍历所有进程
  for(p = proc; p < &proc[NPROC]; p++) {
    if(p != myproc()){
      acquire(&p->lock);
      //找到状态为睡眠并且频道一样的进程
      if(p->state == SLEEPING && p->chan == chan) {
        //唤醒（实质上就是把状态换成RUNNABLE参与调度）
        p->state = RUNNABLE;
      }
      release(&p->lock);
    }
  }
}

在V中会获取条件锁，然后把P所等待的“条件”，即counter==1，实现，然后调用wakeup。虽然wakeup在内部不会获取条件锁，但是要求 在外部获取。wakeup会进入一个循环，遍历所有进程，直到找到状态为睡眠且频道（频道是进程结构体的一个属性）的值相等的进程，并把这个进程状态修改为RUNNABLE。这个循环使得所有在同一个频道上睡眠的进程都可以被唤醒参与调度。有的时候多个进程会睡眠在同一个频道上，比如多个进程都在等待一个管道的输入。通常不会造成什么问题。







7.6 代码：管道 Code: Pipe

    管道是一个使用sleep和wakeup的复杂的例子。通过这个特性管道可以去同步生产者和消费者。两者在pipewrite和piperead中都有使用。

    每一个管道实际上都是一个结构体:

struct pipe {
  struct spinlock lock;
  char data[PIPESIZE];
  uint nread;     // number of bytes read
  uint nwrite;    // number of bytes written
  int readopen;   // read fd is still open
  int writeopen;  // write fd is still open
};

这个结构体中包含管道所维护的一个缓冲器data。指针nread/nwrite指向读取端和写入端，还有一个管道的锁。和uart_tx_buf类似，管道的data也采用了循环数组的方式：也就是说每当两个指针超出了data的长度，就会从头开始。用nwrite==nread+PIPESIZE表示缓冲器满了（PIPESIZE为512），用nwrite==nread代表数组空了。

    假设pipewrite和piperead被同时调用了且pipewrite先获取了pipe->lock。

int
pipewrite(struct pipe *pi, uint64 addr, int n)
{
  int i = 0;
  struct proc *pr = myproc();

  acquire(&pi->lock);
  while(i < n){

    //省略：检查管道和进程信息
    
    if(pi->nwrite == pi->nread + PIPESIZE){ //DOC: pipewrite-full
      //这里分别用nread/nwrite作为chan
      wakeup(&pi->nread);
      sleep(&pi->nwrite, &pi->lock);
    } else {
      char ch;
      if(copyin(pr->pagetable, &ch, addr + i, 1) == -1)
        break;
      pi->data[pi->nwrite++ % PIPESIZE] = ch;
      i++;
    }
  }
  wakeup(&pi->nread);
  release(&pi->lock);

  return i;
}

在检查完进程和管道的一些信息后，pipewrite开始写入data。它通过传入的物理地址以偏移量的方式逐个获取其中的数据，并拷贝到data中。假如data满了，它会唤醒之前进入睡眠的piperead并且调用sleep，释放锁后进入睡眠。如果传输完毕，则同样释放锁结束运行。

int
piperead(struct pipe *pi, uint64 addr, int n)
{
  int i;
  struct proc *pr = myproc();
  char ch;

  acquire(&pi->lock);
  //循环用来处理虚假唤醒
  while(pi->nread == pi->nwrite && pi->writeopen){  //DOC: pipe-empty
    if(pr->killed){
      release(&pi->lock);
      return -1;
    }
    sleep(&pi->nread, &pi->lock); //DOC: piperead-sleep
  }
  for(i = 0; i < n; i++){  //DOC: piperead-copy
    if(pi->nread == pi->nwrite)
      break;
    ch = pi->data[pi->nread++ % PIPESIZE];
    if(copyout(pr->pagetable, addr + i, &ch, 1) == -1)
      break;
  }
  wakeup(&pi->nwrite);  //DOC: piperead-wakeup
  release(&pi->lock);
  return i;
}

由于piperead进入自旋，pipewrite的释放使得它可以继续运行。同样地，piperead会获取管道的锁且检查一些属性，查看data是否为空。假如为空，那么会进入睡眠，等待下一次写入发生后被唤醒。如果不为空就开始读取数据，在读取完毕后释放锁并结束运行。

    多个进程读取同一个管道的情况可能发生。它们会sleep在同一个频道上。每当写入函数调用wakeup，这些进程都会被唤醒。但是只有其中一个进程可以首先获取到lock并且开始读取。在它读取完毕之后，其他后来的进程可能发现data已经是空的了。假如认为“只有管道不为空的情况下piperead才会被唤醒”是一个不变量的话，可以把这一唤醒叫做虚假唤醒（Spurious wakeup） 。这种假的唤醒不会导致问题，因为piperead会检查条件是否为真，如果不为真，则还可以继续睡眠（循环）。也是由于虚假唤醒的存在，sleep必须在返回之前重新获取条件锁，用来防止唤醒丢失。多数对于sleep和wakeup的使用都有这样的机制存在。







7.7 代码：等待，退出，杀死 Code: Wait, exit, and kill

    sleep和wakeup可以被用做很多等待相关的实现中。比如wait和exit。

    当父进程调用wait，它会等待子进程exit之后继续运行。子进程调用exit：exit会把子进程的状态修改为ZOMBIE。当父进程调用wait时，可以发现子进程的状态为ZOMBIE，然后会把子进程的状态再次改成UNUSED，并且复制子进程的exit状态（传给作为参数传入wait的地址中），然后返回子进程的PID。

    当父进程比子进程先完成，也就是当父进程不wait，直接exit：子进程会被作为子进程交给init进程，而init会一直运行wait，这确保了任何进程都有父进程并且可以被父进程清理。

int
wait(uint64 addr)
{
  struct proc *np;
  int havekids, pid;
  struct proc *p = myproc();

  acquire(&wait_lock);

  for(;;){
    havekids = 0;
    for(np = proc; np < &proc[NPROC]; np++){
        //检索整个进程表，找到自己的子进程
      if(np->parent == p){
        // make sure the child isn't still in exit() or swtch().
        acquire(&np->lock);

        havekids = 1;
        //检查子进程是否运行完毕
        if(np->state == ZOMBIE){
          // Found one.
          pid = np->pid;
          if(addr != 0 && copyout(p->pagetable, addr, (char *)&np->xstate,
                                  sizeof(np->xstate)) < 0) {
            release(&np->lock);
            release(&wait_lock);
            return -1;
          }
          freeproc(np);
          release(&np->lock);
          release(&wait_lock);
          return pid;
        }
        release(&np->lock);
      }
    }

    // 省略：不存在子进程的处理
    
    // Wait for a child to exit.
    sleep(p, &wait_lock);  //DOC: wait-sleep
  }
}

调用wait的进程会在最开始获取wait_lock，用来保证lost wakeup 不会出现。当它发现有进程的父进程是自己并且状态为ZOMBIE时，便会释放该进程的资源，释放子进程的lock和wait_lock。但如果没有找到这样的进程，便会进入睡眠。wait会获取两个锁：wait_lock+子进程的lock，顺序是wait_lock，子进程的lock，防止死锁。

exit(int status)
{
  struct proc *p = myproc();
  //initproc不能退出
  if(p == initproc)
    panic("init exiting");

  //省略：关闭打开的文件

  acquire(&wait_lock);

  // 把子进程交给init
  reparent(p);

  // Parent might be sleeping in wait().
  wakeup(p->parent);
  
  acquire(&p->lock);

  p->xstate = status;
  p->state = ZOMBIE;

  release(&wait_lock);

  // Jump into the scheduler, never to return.
  sched();
  panic("zombie exit");
}

    exit会冻结一些资源，并且调用reparent函数，把每一个子进程的父进程重新定向为init，唤醒可能存在的正在sleep的父进程，把当前进程的状态修改为ZOMBIE，调用sched。之后这个进程便不会参与调度了。exit一定会获取p->lock，这样可以保证在状态修改成ZOMBIE后不会在通过swtch切换进程之前直接被父进程清理（进程锁也是在scheduler被释放）。exit使用wait同样的锁顺序来保证无死锁。虽然exit在唤醒父进程之后才获取锁修改其状态，但是由于它有自己的进程锁并且父进程去要获取这个进程锁来释放其资源，父进程会进入自旋。即使在子进程获取锁之前父进程就被唤醒并且执行其循环体，由于子进程的状态不是ZOMBIE，父进程也不会对它进行操作。

    exit可以终止自己的运行，kill可以让一个进程终止另一个进程。假如kill可以直接终止一个进程，那么可能导致一些错误，比如：

目标进程可能运行于一个不同的CPU。那么两个CPU将操作于同一个进程

目标进程可能正在运行一些对内核敏感的代码。

int
kill(int pid)
{
  struct proc *p;

  for(p = proc; p < &proc[NPROC]; p++){
    acquire(&p->lock);
    if(p->pid == pid){
        //修改killed属性并通过修改状态让该进程参与调度
      p->killed = 1;
      if(p->state == SLEEPING){
        // Wake process from sleep().
        p->state = RUNNABLE;
      }
      release(&p->lock);
      return 0;
    }
    release(&p->lock);
  }
  return -1;
}

所以kill只会把目标进程的状态改成killed。可以预期目标进程很快会因为进入/离开trap/收到时钟中断等因素导致killed属性被检查。在这些情景的代码里，假如killed被设置，进程会调用exit去真正终止这个进程：说到底仍然是这个进程自己调用exit把自己终止了。kill会唤醒目标进程（通过修改状态RUNNABLE直接唤醒）。所以假如想要杀死的目标进程正在睡眠，可能造成上述的虚假唤醒。不过由于通常xv6中对于sleep的使用会同时使用while来检查条件，如果不为真就可以继续睡眠。同时，有些这样的循环也会检查killed属性，这样就能直接终止这个进程了，比如管道中就是这样的。不过也有一些sleep的使用不伴随着对于killed的检查，这类代码通常把sleep看作其过程中的一步且希望整个过程完成后再关闭目标进程，也是安全的。







7.8 进程锁 Process Lock

    对于这些进程结构体的属性的访问必须使用进程锁：

p->state;
p->chan;
p->killed;
p->xstate;
p->pid;

除了进程结构体本身，进程锁同样保护着xv6中的一些进程相关的的数据结构和算法，下面是进程锁的所有使用场景：

• Along with p->state, it prevents races in allocating proc[] slots for new processes.

• It conceals a process from view while it is being created or destroyed.

• It prevents a parent’s wait from collecting a process that has set its state to ZOMBIE but has not yet yielded the CPU.

• It prevents another core’s scheduler from deciding to run a yielding process after it sets its state to RUNNABLE but before it finishes swtch.

• It ensures that only one core’s scheduler decides to run a RUNNABLE processes.

• It prevents a timer interrupt from causing a process to yield while it is in swtch.

• Along with the condition lock, it helps prevent wakeup from overlooking a process that is calling sleep but has not finished yielding the CPU.

• It prevents the victim process of kill from exiting and perhaps being re-allocated between kill’s check of p->pid and setting p->killed.

• It makes kill’s check and write of p->state atomic.

进程中的parent属性不被进程锁保护，它的访问伴随着wait_lock，即被wait_lock保护。这样一来，对parent属性的访问也就被这个锁序列化：父进程和子进程同时exit时，后者不需要担心自己的parent为NULL。进程使用这样一个全局锁而不是自己的进程锁给parent属性是因为这样可以限制获取锁之前，该进程不能知道谁是其父进程，实现上述的安全机制。







7.9 补充 More

    不同的线程系统的区别是它们是否给线程共享内存。在xv6中，各个进程的内核线程是共享内存的，即互相可以“看”到对方地址内容的修改，但是用户空间线程的内存是不共享的：xv6的进程全部是单线程，所以线程即进程，必然不会共享内存。更高级的操作系统中一个进程可以有多个线程，比如Linux：Linux中多线程的实现实际上是让多个“进程”共享一个内存空间，从而把它们在形式上归为“一个进程的多个线程”。本章介绍的线程分时实际上效率不高，但是相对于其他分时方式更容易实现。

    时钟中断导致的调度叫做抢占式调度（Pre-emptive schedule），对立于 自愿调度（Voluntary schedule） 。在xv6的实现中，时钟中断导致进程的抢占式调度，而内核负责处理这一trap又是自愿调度：所以上xv6中的时钟中断是两种调度的结合。

    一个进程的内核线程栈保存的是进程当前的内核线程的寄存器，线程切换也是内核线程之间的切换，而不是直接切换用户进程本身；

    swtch期间是禁止进程使用除了进程锁以外的任何锁的。假设在一个单核处理器中一个进程使用了某个锁并进入了调度，另一个进程开始运行并且调用了同一个锁，造成死锁。swtch也会检查这个进程是否携带了除了进程锁之外的锁。

    任何main函数在调用return 之后，C语言保证了有exit会被调用，让这个进程真的释放资源。







7.10 练习 Exercise


> 1. Sleep has to check `lk != &p->lock` to avoid a deadlock Suppose the special case were eliminated by replacing 
    if(lk != &p->lock){
        acquire(&p->lock);
        release(lk);
}

with

        release(lk);
        acquire(&p->lock);

Doing this would break sleep. How?

  1. Implement semaphores in xv6 without using sleep and wakeup (but it is OK to use spin locks). Replace the uses of sleep and wakeup in xv6 with semaphores. Judge the result.

  2. Fix the race mentioned above between kill and sleep, so that a kill that occurs after the victim’s sleep loop checks p->killed but before it calls sleep results in the victim abandoning the current system call.

  3. Design a plan so that every sleep loop checks p->killed so that, for example, a process that is in the virtio driver can return quickly from the while loop if it is killed by another process.

  4. Modify xv6 to use only one context switch when switching from one process’s kernel thread to another, rather than switching through the scheduler thread. The yielding thread will need to select the next thread itself and call swtch. The challenges will be to prevent multiple cores from executing the same thread accidentally; to get the locking right; and to avoid deadlocks.

  5. Modify xv6’s scheduler to use the RISC-V WFI (wait for interrupt) instruction when no processes are runnable. Try to ensure that, any time there are runnable processes waiting to run, no cores are pausing in WFI.
