文件系统性能以及恢复 FS Performance & Recovery



导言：

    回忆在xv6中，文件系统维护在磁盘上维护着一个日志系统，他的结构十分简单：header block+logged blocks。header中记录着当前日志中记录着的block的数量和每个block的number。重要的是，在某个系统调用使用end_op之前，使用的都是内存中的cached block，而只有在所有被修改的block都同步到了磁盘上的日志中的时候，日志的header才会更新指示着对应的block number。而更新header的这一瞬间就是commit-point：在此之前发生的崩溃都会导致这些还没有写入磁盘原来位置的数据丢失（即使已经保存在了日志中），而在此之后这些数据都可以被恢复。

    任何使用日志系统的文件系统都需要遵守两个原则：

Write-ahead rule：在log中的内容写入磁盘之前，任何日志中对应的block都不能提前写入磁盘原位置。这保证了原子化，是日志系统最重要的准则；

Freeing rule：不能在log写入磁盘真正位置之前重写或者重新利用任何log中的block。

    xv6中的日志系统被称为“synchronous”：在一个系统调用执行完一对begin_op, end_op之前，不可能返回用户空间，并且最重要的是在committing的过程中，没有任何其他线程可以使用日志系统，效率十分低。






9.1 Linux Ext3 文件系统

    ext3是Linux所使用的第三代文件系统，本质上是ext2+Journaling：journaling与logging完全等同。与xv6一样，Linux也在内存中保存着一个缓冲缓存器，储存着一些从磁盘中复制来的block。但是不同的是它同样保存着transaction的信息。在ext3中，允许多个交易并发进行，所以它保存着每一个交易的信息：

Sequence：是当前transaction的次序；

当前transaction所修改的blocks的block numbers；

Handles：当前transaction中正在运行的系统调用。

在磁盘上，ext3有和xv6一样的文件目录，bitmap，以及log等。但是ext3的log的格式不太一样：

在其Log中，第一个block是一个superblock，保存着当前Log系统中的元数据，比如第一个有效Transaction的偏移量以及其Sequence。由于Log是一个循环数组，所以superblock后面第一个block可能不是一个transaction的开始。Descriptor block 标志着一个transaction的开始，它记录着这个transaction的元数据，如这个交易的Sequence以及其所涉及的block numbers，类似xv6中的header block。紧随其后的是这个交易的logged block。最后是一个Commit block，标志着这个交易的结束，也保存着Sequence。这个block的后面是另一个交易的descriptor block，以此往复。

为了把descriptor和commit这两个特殊的block和普通的数据block区分开，前两者的开头是一个magic number（某个32位数字），用来让文件系统区分。

    不像xv6支持一个transaction，如图中所示，每一个descriptor block的出现都标志着一个交易的出现。虽然锁ext3的日志系统支持并发，但是一个时刻也只可能有一个 active transaction。这个所谓的活跃的交易必然还没有被复制到磁盘上的日志中，而仍然在内存中修改着缓存blocks。所以某一时刻的所有交易应当为：躺在磁盘上的日志里等待同步回原位置的交易和一个正在内存中读写的交易。

    下面介绍这种日志系统的三个优势：

  1. 非同步的系统调用（Asynchronous Syscalls）: Ext3中的写入磁盘的系统调用可以在系统写入磁盘的时候返回用户空间执行其他任务：系统调用只需关心对于缓存在内存的的文件的读写，不像xv6这个线程需要等待写入；

  2. 批处理（Batching）：可以把多个系统调用批处理到一个交易中（xv6实现了一定程度的批处理）；

  3. 并发：同时可以有多个交易保存在日志中；







9.2 Ext3 日志系统优势

    非同步的系统调用除了可以让一个系统调用更快的返回用户空间，这一特性同时也增强了整个系统的并发能力：在返回用户空间后，磁盘会写入信息，与此同时这一线程也可以并发地处理其他计算：I/O 并发。如果没有这种非同步化，线程的计算无法与其写入并发。同时这一特性也帮助批处理

    不过这也可能会有个问题：即系统调用的返回不意味着磁盘写入的完成。比如设计一个程序在写入信息之后输出完成字符串，但是由于系统调用提前返回，会造成输出完成并不代表真正的写入完成，而此时发生崩溃仍会导致数据丢失。 Ext3通过提供系统调用fsysc(fd) 强制对于一个文件的写入结束之后再从某系统调用中返回。

    在批处理中，多个系统调用可以被打包在一个交易中。它提供了三个优势。

单一的正在进行的交易：ext3中，每隔5秒会产生一个新的交易，五秒之后，这个交易会被提交（放入磁盘的log中）并且开启下一个交易。在这5秒钟所发生的一切系统调用都会被打包在这一个交易之中。这之所以是一个优势是因为它把多个系统调用的交易的提交（放入磁盘上的log）变成了一个固定的成本（成本来自对于写入磁盘log所造成的latency等）。

写入吸收（Write absorption）：当多个系统调用需要写入同一个block的时候，只需要更改缓存在内存中的这一个block即可。比如多个系统调用创建很多文件，要分配很多inode，但可能只是对几个inode block和bitmap block做出了改变，而这些改变都是对于内存中的复制品做出的，所以很快。而在同步的时候，写入吸收也会同样地使得磁盘涉及相对更少的block的写入。

磁盘规划：磁盘规划的优势体现在从内存写入log。Log在磁盘上是一段连续的区域。在磁盘上顺序写入相同的数量的block要比非顺序写入相同数量的block效率更高，甚至也比给同一个block写入相同的次数更高。打包本身会导致相比不打包的情况下，一次顺序写入更多的block。与此同时，磁盘规划也会提高从log写入磁盘原位置的效率：磁盘驱动可能会把目标block number排序，依次写入。

    首先很显然，ext3的日志系统让多个系统调用可以并发运行，并且允许它们并发地修改缓存着的blocks，互相无需等待。

    其次，同一时刻可能会有处于不同阶段的交易在文件系统中并发进行：

  1. 最新的1个交易：正在内存中上接收写入，还没有放入磁盘中的log区；
  2. 正在向log提交的交易：它正在被从内存复制到磁盘中的log；
  3. 正在写入磁盘原位置的交易；
  4. 成功写入的交易：改交易的区域被释放，会被后来的新交易重新利用。

可以看到这里可能会有一个问题：当一个新的交易在处理写入的时候，可能会使用正在向log提交的交易中所正在提交的block。也就是说一个内存中的block正在被系统调用写入，同时也正在被复制到log中。显然一个交易中block中包含着来自另一个交易中的写入是不可接受的。Ext3解决这一情景的方法是，每当一个交易开始提交，会把整个它所修改了的block全部再次复制一份在内存中，所以不用担心后来的交易修改缓存中的相同block造成问题。







9.3 Ext3 日志过程

    类似xv6中的磁盘写入过程，ext3也需要系统调用标明写入的开始和结束位置：

h = start();
get(h, blocknum);

//operations...

stop(h);

star()开始把接下来的操作记录到一个transaction中（内存中操作），返回一个handler：这个handler是每个写入磁盘的系统调用所独有的。调用get()，传入handler和目标block number，再获取目标block的同时标记“这个block正在被系统调用h使用”。在执行一系列的写入操作后，调用stop()，告诉文件系统当前系统调用h结束。不像xv6，这里的stop只是通知系统一个系统调用结束。5秒内会很有很多这样的系统调用发生并且被分配到这一个交易中。很好理解在5秒结束的瞬间，很可能还有一些系统调用正在执行写入，此时文件系统就需要等待这些操作完成，然后再开始提交。

    提交的步骤：

  1. 暂停所有系统调用：不让新的系统调用开始执行。这可能导致一点性能损失；
  2. 等待正在执行的系统调用结束，并且复制修改了的block（保护信息）；
  3. 开启1个新的交易，新的系统调用开始运行；
  4. 写入descriptor block以及所有修改了的block 的 block numbers；
  5. 把block写入日志；
  6. 等待4，5结束；
  7. 写入 commit block；
  8. 等待7完成：7完成之后标志着达成commit-point；
  9. 写入block原始位置；
  10. 这个交易使用的Log中的blocks会被重新利用。

注：上述步骤都是在后台线程完成，没有其他进程会等待其中某一步的完成而继续；如果log中剩余的blocks不够这次交易提交，那么这个交易会等待，直到有足够的空间（等待之前的几个交易完成第9步）。

    在重启后数据恢复写入磁盘的时候有一些细节需要注意。Superblock中永远保存着当前最新的待写入磁盘原位置的交易的descriptor block。文件系统会从这个block开始遍历整个日志找到当前保存在日志中的所有交易最末尾的位置（最后一个交易的commit block），然后开始一个接一个写入磁盘原位置。如果存在未完成的提交（只有descriptor block，没有commit block），会被文件系统忽略。这样的不完整的交易只可能是最后一个交易。

    这里可能会出现一些问题。如之前所说，文件系统区分某block是否为descriptor/commit block的方法是通过检查这个block是否以magic number开头，那么考虑这样的情景：一个交易commit block结束之后没有别的交易了，下一个block是之前存在过的交易曾使用的block，而碰巧这个block也是以magic number开头（但不是descriptor block）。实际上这样的情况由于文件系统的设计不会发生：Ext3在提交的过程中（内存->Log）会检索每个block，如果发现任何一个block以magic number开头，会把这部分数据归0，然后在这个交易的descriptor中记录归零的位置。同时，已完成的交易的commit block也不会造成任何问题，因为Sequence属性本身也用来验证一个交易。

    补充： 如上所述，再开始提交的瞬间任何新的系统调用不得开始写入。除了由于时间已到需要开始提交，另外一个原因是为了原子化。如果仍允许系统调用，可能会导致后来的系统调用修改了和之前的系统调用一样的block。而在提交完成的时候后来的系统调用仍没结束却崩溃发生：后来的系统调用部分发生，原子化失败。这里发生的问题是在block被复制之前（即等待系统调用结束），所以block复制的机制也帮不上忙。







用户程序对于虚拟内存的应用 Virtual Memory for Applications



导言：

    如第三章所介绍的，操作系统内核使用虚拟内存（Page fault）实现了很多有创意的行为，比如懒分配，COW等。这部分内容主要介绍用户空间对于虚拟内存类似的应用以及原理。






9.4 原语 Primitives

    原语（Primitives） 是对一些代码的抽象，它们是为了实现某些功能的单位。在我们的语境下，为了实现用户空间对于虚拟内存的应用，需要有这几个原语：

  1. Trap：这个原语使得当Trap发生后，内核可以返回到用户空间执行特定的用户空间代码（比如当某种Page fault发生后，可以让用户代码来执行一些后续的操作）；

  2. Prot1（Protect1）：降低某一个Page的权限，或者说降低 “accessibility”；比如从读写全乡可以降低为只读/只写/无法访问；

  3. ProtN：类似Prot1，但是同时可以降低多个Page权限；相比前者，ProtN效率更高：每当一个Page（PTE）的权限被修改之后，TLB会被Flush，而flush的成本很高，一次修改多个Page分摊了这一成本；

  4. Unprot：上述的逆向；没有UnprotN，单纯因为实现用户空间虚拟内存应用这一目标不需要；

  5. Dirty：找到那个Page是dirty（被修改过）的；

  6. Map2：把一个范围内地址映射两次到一个程序的内存空间，有不同的权限（Accessibility）。

Xv6不支持其中任何一个，但现代的其他操作系统，比如Linux/Unix，支持。比如Unix中的函数 mprotect(addr,len,R/W/NONE)可以修改某一特定范围的地址为某一权限(三选一)；以及unmap：取消对某一地址范围（本质上仍然是对Page的操作，但特殊）的映射；还有最重要的sigaction： 即signal handler的设置：signal指的是某一错误发生后的“指示”，比如page fault会发起一个 Signal：segfault：若一个程序设置了特定的segfault 处理代码，那么当错误发生就会转向这个代码；mmap可以把一些Object映射到用户内存中，对它的多次调用可以实现上面的map2；

    这些接口的实现也获益于虚拟内存的涉及。除了分页表，另一个重要的组成是虚拟内存区域（VMAs：Virtual Memory Areas）。它们是保存在内核中的对于每一个进程的内存空间各个部分的描述，比如它会指明某一个进程所打开的一个文件在内存中的位置，同时记录文件的fd，偏移量等等。每一个VMA必定描述的是一段连续的内存，这些地址（Page）权限都一样，并且以所映射的Object为单位。对于mmap的每次调用都会产生一个VMA。







9.5 用户空间Trap

    用户空间Trap实际上遵循普通Trap的处理方式。假设一个用户空间的内存地址被访问，而该内存地址被标记为了禁止访问：处理器会进入内核模式，经过正常的Trap处理然后会问虚拟内存系统接下来干什么。比如VM系统给这一报错设置了处理代码（handler），那么此时会出现一个 Upcall， 它使得处理器跳转回用户空间，执行对应的处理代码

    这里有一个简单的例子解释用户空间Trap处理的结构。假设有一个程序的目的是计算大量的平方根，一个有效的优化手段是预处理平方根，当需要某特定数字的平方根时直接查表。如果可能出现的数字的数量大于物理内存空间，这种优化显然不实际。但是可以实现类似懒分配一般，只在虚拟内存只能给分配足够的空间，而不实际映射，只有当目标位置被访问的时候，即有需求某平方根，才真正在收到Page fault之后分配空间并且计算平方根：这样下次使用这一数据的时候就不用计算了。

    最重要的问题是如何在接收到报错之后返回用户空间并执行内存分配以及计算的用户接口。

static void
setup_sqrt_region(void) 
{
    struct sigaction act;

    // Only mapping to find a safe location for the table.
    // 这里的意义是找到一块可以分配的位置
    sqrts = mmap(NULL, MAX_SQRTS * sizeof(double) + AS_LIMIT,PROT_NONE,
                MAP_PRIVATE | MAP_ANONYMOUS,-1,0);
    if (sqrts == MAP_FAILED){
        fprintf(stderr, "Couldn't mmap() region for sqrt table; %s\n",
            strerror(errno));
        exit(EXIT_FAILURE);
        }

    // Now release the virtual memory
    //释放掉
    if (munmap(sqrts, MAX_SQRTS * sizeof(double)+AS_LIMIT)==-1){ 
        fprintf(stderr, "Couldn't munmap() region for sqrt table; %s\n",
            strerror(errno));
        exit(EXIT_FAILURE); }

    // Register a signal handler to capture SIGSEGV. 
    // 这里设置SIGSEGV的handler
    act.sa_sigaction = handle_sigsegv;
    act.sa_flags = SA_SIGINFO; 
    sigemptyset(&act.sa_mask);
    // 正式设置
    if (sigaction(SIGSEGV, &act,NULL)==-1){
        fprintf(stderr, "Couldn't set up SIGSEGV handler;, %s\n", strerror(errno)); 
        exit(EXIT_FAILURE);
    } 
}

在setup_sqrt_region函数中，进程会会真正地设置handler。当对应的Signal发生时，就会调用这里所设置的用户函数handle_sigsegv:

static void
handle_sigsegv(int sig, siginfo_t *si, void *ctx) 
{
    uintptr_t fault_addr=(uintptr_t)si->si_addr;
    double *page_base = (double *)align_down(fault_addr,
            page_size); 
    static double *last_page_base =NULL;
    //开始映射
    if (mmap(page_base, page_size, PROT_READ | PROT_WRITE,
            MAP_PRIVATE| MAP_ANONYMOUS | MAP_FIXED,-1,0) ==MAP_FAILED){ 
        fprintf(stderr, "Couldn't mmap(); %s\n", strerror(errno));
        exit(EXIT_FAILURE); 
    }
    //计算平方根
    calculate_sqrts(page_base, page_base - sqrts, page_size / sizeof(double)); 

    //释放之前分配的Page，压缩所使用的空间
    if (last_page_base && munmap(last_page_base,page_size)==-1){    
        fprintf(stderr, "Couldn't munmap(); %s\n", strerror(errno)); 
        exit(EXIT_FAILURE);
    }

    last_page_base = page_base; 
    }






9.6 垃圾回收器 Garbage Collector

    垃圾回收器通常是编程语言自身设计的回收机制，在此机制之下可以避免资源的浪费。主流语言中，除了C语言不存在这种机制，必须手动调用free，其他语言皆存在。对于这些语言，程序员只需要调用内存分配接口，而不用担心内存释放，由垃圾回收器决定某一内存空间是否在被利用，如果没有，则释放。本节介绍的垃圾回收器叫做 Copying Garbage Collector，是基于Cheney 算法的实现[1]。

     可以看到在这一系列行为中，其他一切行为需要停止，知道回收完毕。Baker‘s 算法修改了一些步骤：在最开始，只有root指向的obj会移动（转发（Forward））到to-space，其他不动；直到调用了new创建新的obj才会再转发一些obj。而对于现有的被root联通的obj的访问（对指针的解引用），也会导致对这个obj未知的检查：如果在from-space，就转发。最终也可以实现有效obj和垃圾的分隔，完成回收。转发的成本被分摊。但缺点仍然明显：每次解引用和检查都需要成本，并且并没有对平行运行有什么帮助，而且可能会出现竞争问题：上面提到的两种转发原因同时对于一个obj发生，造成两次转发。

     用户进程对于虚拟内存的使用解决了上述的问题。大部分步骤是与Baker的方式是一样的，不同的是to-space被分成了两部分：以扫描/未扫描部分（scanned/unscanned）。后者的page权限都是NONE，即无法访问。在最开始，to-space完全是未扫描的，但是回收器会转发root指向的obj到to-space。当用户进程访问这个obj的时候就会触发一个Page-fault。Handler会把出现错误的这部分地址标记为可使用的权限，使得这部分变成了“以扫描”区域，然后会把当前目标obj的指针所指向的其他obj转发到to-space未扫描部分（实际上转发了1Page的obj，因为“扫描”的单位也是1page）。这里看上去有个问题：已知目标地址权限是NONE，那回收器又是如何在修改权限之前进行转发的？这里的解决方法是使用Map2原语，用户进程映射to-space未扫描的部分为NONE，而回收器映射这部分空间为R+W。

注：转发本身并不是“扫描”，“以扫描”是用户进程出现page-fault之后handler行为导致的，是对目标obj所在page的“扫描”。

    这种实现即维持了Baker算法的分摊，也取消了检查指针指向位置的需要（如果没导致报错，说明在to-space，反之则仍在from-space）。更重要的是，它实现了并发：用户进程和回收器可以同时运行而不用担心之前的竞争问题，因为用户空间对to-space中未扫描部分的访问会导致报错，而回收器不会再使用以扫描的部分：两者永不相交。

9.7 内核对比 Kernel comparison

    很多系统都采用宏内核（Monolithic kernel）。宏内核为用户进程提供了巨大的抽象，把底层藏了起来，比如文件系统。这样做有很多优势：

  1. 移植性（Portability）：软件无需关心它们运行在什么硬件上；

  2. 隐藏复杂性：软件直接使用接口即可，无需关心如磁盘读写的实现等等；

  3. 资源分配：内核进行资源分配，也是对于硬件的管理。

它包括了很多子系统，如文件系统和日志系统等等，这使得不同的子系统之间可以很轻松的访问对方的数据结构：比如exec()会读写磁盘，同时也会调用鹅你存分配，这样做的成本很小。但是它也有缺点：

  1. 内核过大且复杂，这导致bugs和安全问题；
  2. 宏内核很多功能/子系统或许冗余，比如一个服务器可能并不需要某些特性，这导致速度相对慢；
  3. 设计受限：比如虽然可以wait子进程，但是不能等待其他进程。这也导致可拓展性相对较低。

    微内核（Microkernel） 是为了解决上述问题的一个方案。它的主要思想是把内核控制得大小和功能受限，主要通过内核的 IPC（Inter-process communication） 来实现不同进程之间的交流，下文的Tasks和进程的概念类似。

如上图中，应用程序如果想要和服务器，比如文件系统，交流，需要通过向内核发送一个IPC，然后内核会把这个IPC传递给目标的服务器。服务器之后会处理信息并且通过一样的路径传递给引用程序。实际上除了内核本身，每一个元素都是独立的用户进程 ，不像宏内核中，文件系统会被认为是内核进程在工作。

    使用微内核的原因：1～4得益于体积小；5～6的抑郁子系统事用户进程

  1. 安全性：内核越小，代码越少，bug越少，越安全；这种安全性是可证明的；

  2. 高优化：简单的代码更好优化，更快；

  3. 更灵活：对于内核设计者来说，需要做出更少的决策，更加灵活的决策；

  4. 让不同的子系统（一些用户进程）之间更加模组化；

  5. 客制化：子系统可以更容易被修改；

  6. 鲁棒性：一个子进程崩溃，不会导致内核崩溃；

  7. 可以让别的操作系统作为一个子系统在微内核上运行。

    微内核的设计有挑战：

  1. 控制系统调用数量，但系统调用要强大；显然必须支持exec, fork等；

  2. IPC速度问题；







9.8 seL4

    L4是一款微内核，它只有7个系统调用，一万多行代码。其中的Task的概念类似进程，一个Task有一个内存空间，支持多线程

    系统调用：

  1. Thread create：创建线程；

  2. send/receive IPC：IPC收发；

  3. Mapping：内存映射：通过这个系统调用一个线程可修改另一个Task的内存布局；

  4. Device Access：把设备映射到自己的内存空间；

  5. 中断：来自设备的中断会变成一个IPC被Task接收；同时当一个Task发生page-fault的时候，这个Task也可以发送一个IPC给预先设定的Pager-Task，运行后者的代码，实现一些功能；

  6. Yield：线程切换；

  7. call：实现远程过程调用。

     IPC设计：这里以一个简单但是慢速的设计为例。

当T1想给T2发送IPC的时候，它会调用send()系统调用，标注目标线程的ID以及内容Message。内核中可能保存着一个message缓冲器，保存着待执行的IPC。在发送IPC，把message放入缓冲器之后，T1会继续运行。某一时刻T2会调用recv()，然后得到在缓缓冲器中的message。在处理完这一行为message之后，T2会调用send，执行一样的过程给T 发送IPC。这种设计叫做Asynchronous，因为T1在发送完之后会继续执行其他任务。这种设计代价高，因为前后一共有4次系统调用，并且需要切换context，且T1在调用recv之后可能会睡眠。

    现在L4的设计中采用了快速IPC，它是Synchronous的。在这种设计下，T1发送IPC之后会在内核中等待T2来接受IPC，一旦后者调用了recv，前者会直接进入后者的用户空间，把msg复制到后者的用户空间。这个设计中不需要缓冲器。进入后者用户空间的行为不需要context切换。当msg体积很小的时候，可以直接把msg放入寄存器中传输给T2，这样甚至不需要涉及内存复制信息。而当msg体积过大的时候，可以通过mapping系统调用直接把目标内容所在的page映射到T2的内存空间，同样不需要内存复制。

    远程过程调用（RPC：Remote procedure call） 通过对call()和sendrecv()系统调用的使用实现。前者结合了send,recv两个系统调用：发送IPC并且等待回复，而服务器会调用sendrecv()发送回复，并且等待下一个输入。这一对系统调用减少了对kernel的使用（把两个系统调用变成了一个）。

    兼容性：微内核会希望它可以运行其他操作系统的文件，比如seL4运行Linux的程序。重构一个应用的成本很大，所以可以直接在L4上运行一个修改版的Linux完整内核：

Linux内核中会包括其原有的子系统，整个Linux作为一个巨大的Task存在。而其他Linux进程并不会在这个内核中运行，而是作为独立的Task在L4中运行。每当Linux希望修改进程的内容，比如PT的时候，就会想L4的内核发送IPC达到此目的。相似地，每当Linux用户进程希望使用Linux的系统调用的时候，L4会把这一请求作为IPC发送给Linux（而并不是L4直接运行这些系统调用）。由于是L4在真正控制着那个进程去运行，也就是说它控制着调度，这使得Linux的优秀的调度设计无法使用（之后版本的L4为Linux提供了发送调度的能力）。

    微内核使用不广泛，不止因为性能不够，更多的是找不到切换到微内核的必要。但是微内核的研究对操作系统有影响。
