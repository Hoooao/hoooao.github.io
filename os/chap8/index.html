Chap 8: 文件系统 File system

目录



导言：

    一个文件系统的目的是储存和组织数据，把这些数据供给与多个进程和用户，同时也要保证重启后数据的保留。     创建一个文件系统有几个难点：

  1. 文件系统需要有一个在磁盘上的树类结构去储存文件目录和文件本身。并且也要知道哪些磁盘块（block/sector）是被占用的，哪些是可以使用的。

  2. 文件系统必须有崩溃修复机制。也就是说在崩溃发生，比如断电，重启后，文件系统必须正常工作。不然可能造成类似并行错误的发生。

  3. 不同的进程可能会同时去访问一个文件，所以需要保证并发时文件系统也能正常运行。

  4. 从磁盘中获取数据很慢，所以文件系统必须在内存中有对部分常用的磁盘内容缓存。

本章解释xv6是如何解决这些困难的。







8.1 概述 Overview

    xv6的文件系统有7层:

从下至上分别是：

  1. 磁盘层（Disk）：对磁盘直接的读写。xv6中使用的磁盘是Qemu提供的虚拟硬盘（Virtio hard drive）。

  2. 缓存缓冲层（Buffer cache）：这一层维护着一个数组，数组中的元素是对磁盘中某些操作部分的内容复制。用来保证并发正常和支持缓存减少磁盘访问次数。

  3. 日志（Logging）：对一系列磁盘的操作的记录。把多个操作一次性传递给磁盘，减少了磁盘的访问的同时保证了行为的原子化。最重要的是帮助在系统崩溃后的数据恢复和文件系统正常。

  4. Inode（Index node）：实际上就是文件本身。它是一种数据结构，每个inode保存着一个文件相关的信息。文件本身的内容并不存放在inode中。

  5. 目录（Directory）：是一种特殊的inode，其内容是一系列的目录元素，每个元素包含着一个文件名和inode-number，即i-number，用来识别特定的inode。

  6. 路径（Pathname）：即文件路径。

  7. 文件描述符（File descriptor）：是对文件，设备，以及管道等资源的抽象。提供接口来操作这些资源。

    数据的最小单位时bit，但是对于磁盘的每次读写的最小单位是 512Bytes，也称之为 “一块（Block/sector）”。这里的block和操作系统中的block的大小可能并不总是相等，但是后者通常是前者的整数倍。它们在磁盘上连续排列：sector0, sector1…。xv6保存着一部分block的复制，这些“复制品”被保存在独立的buf结构体中。这些复制品有的时候可能和磁盘上的本体的数据是不一样，也就是“不同步”。比如可能因为磁盘工作于本体block，而OS还没有读取；更常见的情况是OS对复制品进行了修改，但是还没更新本体来同步。

    文件系统必须良好的组织不同的东西，比如如何储存inode和inode/文件的内容本身。xv6中，磁盘被分成了多个部分

section0是boot，储存着开机所需的内容比如一些代码，这部分文件系统不会使用。section1储存着Superblock，它是关于这个文件系统的元数据，比如文件系统大小，block的数量，文件的数量，和log中所储存的block的数量。section2和后面的一些储存log。之后是inode，也就是文件（的元数据）。之后是bitmap(xv6中只占1个block)，它追踪每个block正在被使用或者可以被使用。后面的便是真正的数据block了。接下来每一节介绍一层文件系统。







8.2 缓存缓冲层 Buffer cache layer

    这一层的代码放在bio.c中。它主要有两个作用：1.确保一个时刻只能有一个进程访问某个block，保证并发正确 2.缓存常用的block，减少磁盘访问。它维护着一个结构体bcache，这个结构体中有一个数组buf，储存着buf类型的结构体。

struct {
  struct spinlock lock;
  struct buf buf[NBUF];

  // Linked list of all buffers, through prev/next.
  // Sorted by how recently the buffer was used.
  // head.next is most recent, head.prev is least.
  struct buf head;
} bcache;

每个buf结构体都代表着一个从磁盘中读取的block。最重要的函数时bread和bwrite。前者会从磁盘中读取一个block放入结构体，然后返回这个结构体（上持有睡眠锁）。后者把buf结构体写回磁盘。同时还有函数brelse，每当一个进程使用完了一个buf之后，都需要调用它来释放这个buf的睡眠锁。在缓冲缓存层，每个buf都有一个属于自己的睡眠锁，这使得每次只能有一个进程使用这个buf。由于bcache的长度是有限的，每当使用完全部的槽位之后而又有新的block需要被复制进来，那么就需要重新利用某些槽位。原则是，舍弃最近最少使用（LRU：Least recently used） 的buf的草味，因为通常假设LRU位置上的是最不可能再次被使用的block。

8.3 代码：缓存缓冲 Buffer cache

    每个buf可以被看作bcache中维护的那个数组的节点：

struct buf {
  int valid;   // has data been read from disk?
  int disk;    // does disk "own" buf?
  uint dev;
  uint blockno;
  struct sleeplock lock;
  uint refcnt;
  struct buf *prev; // LRU cache list
  struct buf *next;
  uchar data[BSIZE];
};

其内部保存着前节点和后节点，所以bcache的数组（下文bcache指其内部的数组）是一个双向环形链表，长度为NBUF。通过bcache.head来访问该数组，而非直接访问bcache.buf。     每个buf（block在内存中的复制品）有两个属性：valid，disk。前者表达这个buf是否保存着一个block的复制品，后者表达这个buf上的复制品是否被同步到了磁盘上（如果还没同步，就不能装别的复制品）。当然还有重要的属性data，即数据本身。

    bread实际上调用了bget来申请一个bacache的槽位（注意，bget是申请一个可用的buf槽位，而不是从磁盘中读取任何东西），也就是buf。

static struct buf*
bget(uint dev, uint blockno)
{
  struct buf *b;

  acquire(&bcache.lock);

  // Is the block already cached?
  for(b = bcache.head.next; b != &bcache.head; b = b->next){
    if(b->dev == dev && b->blockno == blockno){
      b->refcnt++;
      release(&bcache.lock);
      acquiresleep(&b->lock);
      return b;
    }
  }

  // Not cached.
  // Recycle the least recently used (LRU) unused buffer.
  for(b = bcache.head.prev; b != &bcache.head; b = b->prev){
    if(b->refcnt == 0) {
      b->dev = dev;
      b->blockno = blockno;
      b->valid = 0;
      b->refcnt = 1;
      release(&bcache.lock);
      acquiresleep(&b->lock);
      return b;
    }
  }
  panic("bget: no buffers");
}

bget会先去bcache中找传入的blockno是否存在，如果存在则直接返回，注意这个循环是从头部开始正序遍历，因为假设bread可能是来访问一个已经被载入缓存的block的。但是假如目标复制品不存在，就需要放弃一个保存的复制品了。注意这里的循环是从尾部（由于是一个环形链表，尾部在头部之前）开始遍历到头部来找refcnt，也就是对于有多少个指针指向这个buf的计数，为0的buf，如果有就把这个buf替换成新的buf的信息（device，sector number等），如果没有，就崩溃。在返回这个buf之前，bget会为它开启sleeplock，这样就保证了只有使用bread的进程可以使用这个buf。 在bread得到一个buf之后，假如valid没有被设置，也就是说得到的是一个全新的buf，bread会通过调用函数 virtio_disk_rw从磁盘中把目标block的内容读取进来。

    每一个block只能在bcache中有一份复制，不然未来同步的时候可能起冲突。bget保证了这一不变量：在遍历bcache寻找目标buf前，bget会获取bcache的自旋锁。这样别的进程就无法同时操作这个数组了。假如两个进程想要获取同一个block，当前一个进程得到锁并且把目标block加载入buf中后，另一个进程间会在循环中看到其目标block已经在buf中了。这保证了对于目标buf是否存在以及对不存在的block的载入的原子化。在代码中，bget先释放了bcache才给目标buf上锁，这是安全的因为此时refcnt已经不是0，别的进程无法把这个buf分配给另一个block了。注意，这个过程中bcache.lock保证了bcache的不变量（只有一份复制），而b.sleeplock保证了buf的内容的安全（只有一个进程可以使用它）。

    使用了bread的进程将会得到一个buf的使用权，假如这个进程对其内容做出了修改，那么必须要调用bwrite

void
bwrite(struct buf *b)
{
  if(!holdingsleep(&b->lock))
    panic("bwrite");
  virtio_disk_rw(b, 1);
}

把做出的修改写回到磁盘中去。同样地，它也使用了virtio_disk_rw。注意，调用bwrite之前，b的sleeplock一定是在被获取的状态，不然的话将会无法成功写入：因为程序无法保证这一写入是否被并发破坏。虽然在bwrite中没有释放这个锁，有专门的bresle函数用来释放这个锁。

void
brelse(struct buf *b)
{
  if(!holdingsleep(&b->lock))
    panic("brelse");

  releasesleep(&b->lock);

  acquire(&bcache.lock);
  b->refcnt--;
  if (b->refcnt == 0) {
    // no one is waiting for it.
    b->next->prev = b->prev;
    b->prev->next = b->next;
    b->next = bcache.head.next;
    b->prev = &bcache.head;
    bcache.head.next->prev = b;
    bcache.head.next = b;
  }
  
  release(&bcache.lock);
}

除了会把这个锁释放掉，该函数也会把这个buf移动到bcache的最前面，这样规范了最久没用的buf一定在末尾，使得bget的性能得到优化。注：当refcnt不为0，证明有别的进程也在使用这个指针，不会切换buf的位置；所以先释放睡眠锁再b->refcnt--也是安全的：别的进程肯定不会运行切换buf位置的代码







8.4 日志层 Logging Layer

    文件系统的设计的一个重要的方面就是崩溃恢复机制（Crash recovery）。假如不存在这样的机制，可能造成磁盘上的文件系统的不连贯状态。比如一个崩溃发生在了一个文件删除操作的过程当中，考虑在崩溃前做了哪些操作，可能导致两种情况：

  1. 一个inode会指向一个被标记为free的block;
  2. 一个被分配的block没有任何inode指向它。

后一种情况不会造成很大问题，但是前者可能导致在重启后另一个进程也使用了一个已经被一个inode指向的block，导致错误。假如一台电脑有多个用户，那么可能会导致多个用户都能访问到同一个block，造成安全问题。

    xv6解决这种问题的方式是通过一个简易的日志系统。xv6中的系统调用并不会直接操作于磁盘上的数据，而是先把它所希望做出的操作记录在磁盘上的日志中(初春的实际上是修改后的block复制品)。当这个系统调用记录完了它的所有操作之后，会再发出入一个提交（Commit），表达这里结束了一套完整的操作。然后系统调用才会真正地把做出的修改同步到磁盘上的文件。最终系统调用会清空之前的日志。

    在这种模式下，假如类似的崩溃发生，重启后文件系统会首先运行一些恢复机制相关的代码来检查日志中是否存在还没有执行的完整操作日志（即被提交的日志），如果有就同步，如果没有或者操作日志并不完整（无提交），就会忽略。最终这些代码会清空日志。这样对一系列修改的打包可以让它们原子化。







8.5 日志设计 Log Design

    日志存放在磁盘上的固定位置，这个位置写在superblock中(磁盘上的一块区域，储存文件系统元数据)。一个日志系统由两部分组成：header block和跟在其后的logged blocks；前者存放着一个sector/block number的数组，指明后面的logged block所属的sector/block，同时也保存着logged block的个数。当count=0说明没有任何完整的操作存在（无commit）。如果不是0，说明有完整的操作存在。在调用commit后，磁盘上的日志的header才会被修改；在把logged block复制回真实的位置上后，count归零。所以假如在commit之前系统崩溃，重启后将会观察到commit=0，而忽视保存的操作，非0则相反。

    每个系统调用都必须把自己对block的操作的开头和结尾指明。同时为了实现文件系统的并发，xv6也允许不同的进程的操作集合在一次commit中。也就是说，每次commit所包含的一系列操作可能是来自不同进程的无关的操作的结果。注：对一个进程来说，它的所有操作必须打包在一个commit中，这是为了原子化；让进程标记自己的磁盘指令的开始和结尾帮助这一实现。

    把多套完整的交易（Transaction）一次commit叫做group commit。这一实现再次减少了磁盘的访问次数，同时也让磁盘在一个时刻进行更多的并发操作（比如磁盘转一圈能写入的东西可能会更多）。xv6的虚拟硬盘不支持这一行为，但是xv6的文件系统支持。

    xv6中日志可以使用的空间是固定的，也就是说一次提交的block数量是有上限的（logged blocks）。每当一个进程希望把自己的操作打包到一次提交中，文件系统必须要知道日志系统中剩余的block数量够不够用。由于一个进程的操作不能分在多次提交中，当数量不够的时候不应当进行。注：这里考虑的是被操作的block的数量，而不是操作的空间大小。大多数系统调用不存在这个问题（也就是对大量的block进行），但是write和unlink除外。write可能常会写入大量数据到一个文件中，这导致它会使用很多数据的block，bitmap block，和inode block（不是只有对于文件内容block的修改才会被考虑）。unlink可能导致使用很多bitmap block和 inode block。xv6中会通过把一个write分割成多次传输，并且xv6的文件系统中只有一个bitmap，所以unlink也不会有问题。







8.6 代码：日志 Code：logging

    一个通常的写入磁盘的步骤大致是这样的：

  begin_op();
  ...
  bp = bread(...);
  bp->data[...] = ...;
  log_write(bp);
  ...
  end_op();

注：这里涉及的"日志"实际上有两个：内存中log结构体，和磁盘上真正的log区域。由于后者本质上也是磁盘内容，前者可以理解为后者的“复制品”，用来同步修改。

struct log {
  struct spinlock lock;
  int start;
  int size;
  int outstanding; // how many FS sys calls are executing.
  int committing;  // in commit(), please wait.
  int dev;
  struct logheader lh;
};
//log的header block的复制品
struct logheader {
  int n;
  int block[LOGSIZE];
};

    begin_op()先去获取log的锁，假如有别的进程正在使用begin_op或者end_op就进入自旋。

void
begin_op(void)
{
  acquire(&log.lock);
  while(1){
    //committing只是一个bit
    if(log.committing){
      //等待提交完毕后的唤醒
      sleep(&log, &log.lock);

      //LOGSIZE=30，MAXOPBLOCKS=10
    } else if(log.lh.n + (log.outstanding+1)*MAXOPBLOCKS > LOGSIZE){
      // this op might exhaust log space; wait for commit.
      //如果剩余空间不够，就睡眠
      sleep(&log, &log.lock);
    } else {
      log.outstanding += 1;
      release(&log.lock);
      break;
    }
  }
}

顺利进入循环后会检查当前是否log系统正在提交操作，为真的话进入睡眠。若不为真，会给当前系统调用预留log中的logged block位置：通过把header中统计的logged block数量增加（这个是内存中的log结构体，而不是磁盘上真正地log），每次增加的数量是MAXOPBLOCKS，也就是10；换句话说，xv6的log系统保守的认为进程会最多使用10个log block。outstanding属性统计请求预留空间的系统调用的个数。增加这个属性既保证了空间预留，也保证了不让committing发生（只有outstanding==0才会commit）

void
log_write(struct buf *b)
{
  int i;

  acquire(&log.lock);
  if (log.lh.n >= LOGSIZE || log.lh.n >= log.size - 1)
    panic("too big a transaction");
  if (log.outstanding < 1)
    panic("log_write outside of trans");

  for (i = 0; i < log.lh.n; i++) {
    if (log.lh.block[i] == b->blockno)   // log absorption
      break;
  }
  log.lh.block[i] = b->blockno;
  if (i == log.lh.n) {  // Add new block to log?
    bpin(b);
    log.lh.n++;
  }
  release(&log.lock);
}

    log_write()会把一个block的编号写入header的block数组中。假如这个编号已经存在了，那么不会把这个编号再次记录在数组末尾，这一机制叫做Absorption：它节省了空间的同时也会提高效率因为只会有一个block的复制品会被读写。如果不存在的话就添加到末尾。最终对于新添加的block，log_write会把它通过bpin“钉”在bcache中，后者实际上就是给这个block/buf添加了一个refcnt。这样保证了该buf在被提交之前不可能会被重新利用掉。

void
bpin(struct buf *b) {
  acquire(&bcache.lock);
  b->refcnt++;
  release(&bcache.lock);
}

end_op()减少outstanding-=1。假如outstanding==0，就可以commit了。

void
end_op(void)
{
  int do_commit = 0;

  acquire(&log.lock);
  log.outstanding -= 1;
  if(log.committing)
    panic("log.committing");
  if(log.outstanding == 0){
    do_commit = 1;
    log.committing = 1;
  } else {
    // begin_op() may be waiting for log space,
    // and decrementing log.outstanding has decreased
    // the amount of reserved space.
    wakeup(&log);
  }
  release(&log.lock);

  if(do_commit){
    // call commit w/o holding locks, since not allowed
    // to sleep with locks.
    commit();
    acquire(&log.lock);
    log.committing = 0;
    wakeup(&log);
    release(&log.lock);
  }
}

调用commit()。

static void
commit()
{
  if (log.lh.n > 0) {
    write_log();     // Write modified blocks from cache to log
    write_head();    // Write header to disk -- the real commit
    install_trans(0); // Now install writes to home locations
    log.lh.n = 0;
    write_head();    // Erase the transaction from the log
  }
}

commit首先会把log结构体中记录的所有block的复制品从bcache拷贝到磁盘上的log的对应位置，然后把log结构体的head放入真log.head。最后调用install_trans(0)把真log中的数据同步到block们的原始位置。然后把两个log（磁盘上的和结构体）初始化。

    recover_from_log()在initlog中被调用，initlog在fsinit中被调用，后者在启动阶段被调用。它会读取logheader，然后模仿end_op的行为：作为崩溃恢复机制的重要存在

    注：filewrite中可以看到write是如何被分割成多次写入的：通过一个循环多次提交。







8.7 代码：Block分配器 Code：Block Allocator

    文件和目录内容都储存在磁盘的block中，这些block是从一个free pool中分配的。xv6的文件系统在磁盘上保存着一个free bitmap，每个bit对应着一个block，指示对应的block是否可用：1代表对应的block正在使用；0代表可以被分配。程序mkfs给boot sector,log blocks, inode blocks, bitmap blocks设置这一数字。

    分配器有两个重要的接口：balloc分配一个新的block，bfree释放一个block。

static uint
balloc(uint dev)
{
  int b, bi, m;
  struct buf *bp;

  bp = 0;
  //BPB：Bits per block:1024*8
  for(b = 0; b < sb.size; b += BPB){
    //sb: superblock
    //把一个bitmap block读入内存
    //BBLOCK是一个宏函数，计算当前的bitmap的block num
    bp = bread(dev, BBLOCK(b, sb));
    for(bi = 0; bi < BPB && b + bi < sb.size; bi++){
      //使用位运算，以bits为单位检索
      m = 1 << (bi % 8);
      if((bp->data[bi/8] & m) == 0){  // Is block free?
        bp->data[bi/8] |= m;  // Mark block in use.
        log_write(bp);
        brelse(bp);
        //归零这个block；bi被用作偏移量
        bzero(dev, b + bi);
        return b + bi;
      }
    }
    brelse(bp);
  }
  panic("balloc: out of blocks");
}

balloc通过双重循环遍历bitsmap的所有bits：外部循环遍历所有bitmap block，内部循环遍历当前bitmap block中对应的所有block的BPB（bits-per-block） bits。如果找到了可用的block，将会把这个block归零并返回。注：这里使用了压位：虽然bitsmap看起来像是一个布尔容器因为它指明一个block“是否”可以使用，但是bool类型占1Bytes；由于布尔类型只是1/0，这里把它们压缩成了1bits。在循环内部也通过位运算间接地一个bit一个bit检查。BPB=BSIZE*8的意义就是以bit为单位遍历。

// Free a disk block.
static void
bfree(int dev, uint b)
{
  struct buf *bp;
  int bi, m;

  bp = bread(dev, BBLOCK(b, sb));
  bi = b % BPB;
  //同样进行位运算
  m = 1 << (bi % 8);
  if((bp->data[bi/8] & m) == 0)
    panic("freeing free block");
  bp->data[bi/8] &= ~m;
  log_write(bp);
  brelse(bp);
}

bfree找到传入的block的对应的bitmap，然后清空目标bit。这两个函数必须在提交的过程中被使用。







8.8 Inode层 Inode layer

    inode也有两种。

    一种是在磁盘上的真的inode。它们连续排布在inode block中。Inode大小相同（64Bytes ），所以可以很容易通过偏移量找到它们，比如第n个inode在n*64/1024block（xv6中1个block是2个sector大小，即1024Bytes）上，这里的n就是i-number。磁盘中的inode定义为struct dinode：

// On-disk inode structure
struct dinode {
  short type;           // File type
  short major;          // Major device number (T_DEVICE only)
  short minor;          // Minor device number (T_DEVICE only)
  short nlink;          // Number of links to inode in file system
  uint size;            // Size of file (bytes)
  uint addrs[NDIRECT+1];   // Data block addresses
};

属性：type表达这个inode指向的资源是什么：文件，目录，特殊文件（设备），如果type=0表明这个inode没有被使用；nlink是对于指向这个inode的目录的计数，假如nlink=0，就可以考虑把这个inode以及它的data block释放掉；size记录着这个inode的内容的大小；addr记录着data block的位置，是一个block number（block地址）的数组。

    内核中也在内存维护着一些inode，定义为struct inode，作为磁盘上的对应的dinode的复制品。

// in-memory copy of an inode
struct inode {
  uint dev;           // Device number
  uint inum;          // Inode number
  int ref;            // Reference count
  struct sleeplock lock; // protects everything below here
  int valid;          // inode has been read from disk?

  short type;         // copy of disk inode
  short major;
  short minor;
  short nlink;
  uint size;
  uint addrs[NDIRECT+1];
};

这些inode放在itable中:

struct {
  struct spinlock lock;
  struct inode inode[NINODE];
} itable;

只有在存在C指针指向这个inode，内核才会持续保存它：ref是对于这样的指针的计数，当ref=0，这个inode就会被放弃。注意区分nlink和ref。

    iget和iput分别获取和释放inode的一个指针，并且修改ref。这样的指针可能被用在fd，正在使用的目录，或者类似exec的代码。

    xv6中inode相关的代码中存在4中锁和类似锁的机制。

  1. itable.lock：保护不变量：一个inode只能在itable中出现最多一个，且ref=指针数量。

  2. 每个inode都有自己的lock，它是一个睡眠锁，只能有一个进程在一个时刻访问这个inode和它的内容block。

  3. 不变量：ref不为0，itable就不能舍弃它。

  4. 每个inode结构体也有nlink，只要它不是0，xv6就不会从磁盘中释放它。

    iget返回的inode一定是可以用的，直到对应的iput被调用。在此期间这个inode结构体不会被删除，内容不会被别的inode替换。iget可以多次分配同一个inode结构体给多个指针。很多文件系统的操作依赖于这一特性，比如需要长期持有某个inode作为打开的文件/目录，并且在多进程并发访问一个inode时避免死锁的同时预防了竞争。

    iget返回的inode实际上没有目标磁盘上的inode的有用信息。这点类似于bget。进程通过调用ilock读取磁盘上的inode的信息，并且赋值给inode结构体。iunlock执行相反的操作。iget和ilock对itable中的inode的分配和赋值做出了模组化，这样防止了某些情况下的死锁。比如目录遍历只需要iget，而不需要ilock（只有ilock获取了inode的lock）。所以虽然多个进程可以调用iget，但是只有一个进程可以使用ilock。

    itable的主要作用类似bcahce，避免并发问题的同时起到缓存常用inode的作用。iupdate可以把对一个inode做出的修改同步回磁盘上。







8.9 代码：Inode Code：Inode

    当新建一个文件的时候，需要从磁盘的inode区域划出一个新的inode给到这个文件。xv6通过ialloc分配磁盘中的闲置的inode:

struct inode*
ialloc(uint dev, short type)
{
  int inum;
  struct buf *bp;
  struct dinode *dip;

  for(inum = 1; inum < sb.ninodes; inum++){
    bp = bread(dev, IBLOCK(inum, sb));
    dip = (struct dinode*)bp->data + inum%IPB;
    if(dip->type == 0){  // a free inode
      memset(dip, 0, sizeof(*dip));
      dip->type = type;
      log_write(bp);   // mark it allocated on the disk
      brelse(bp);
      return iget(dev, inum);
    }
    brelse(bp);
  }
  panic("ialloc: no inodes");
}

类似balloc，它会遍历inode block中的inode，直到找到一个type==0的inode，（这些被遍历的inode实际上是从磁盘上复制过来的inode，以dinode结构体的形式临时保存在内存）也就是可以被使用的inode。它会把这个inode内容归零0，修改为目标type，记录inum，返回iget返回的值，也就是一个inode结构体。虽然这个函数中没有使用lock，也不用担心会出现类似“多个进程同时分配了一个inode情况”，因为在使用bread把整个inode block加载到内存的行为已经规避了并发问题。

    iget遍历itable去找目标inode是否已经存在于itable中了

static struct inode*
iget(uint dev, uint inum)
{
  struct inode *ip, *empty;

  acquire(&itable.lock);

  // Is the inode already in the table?
  empty = 0;
  for(ip = &itable.inode[0]; ip < &itable.inode[NINODE]; ip++){
    if(ip->ref > 0 && ip->dev == dev && ip->inum == inum){
      ip->ref++;
      release(&itable.lock);
      return ip;
    }
    if(empty == 0 && ip->ref == 0)    // Remember empty slot.
      empty = ip;
  }

  // Recycle an inode entry.
  if(empty == 0)
    panic("iget: no inodes");

  ip = empty;
  ip->dev = dev;
  ip->inum = inum;
  ip->ref = 1;
  ip->valid = 0;
  release(&itable.lock);

  return ip;
}

假如存在，则直接返回itable中的inode。同时iget也会记录itable中第一个可以被用来分配新inode的位置，假如itable中不存在目标inode，就会使用这个位置分配inode。

    在进程读写inode之前，必须调用ilock，一为了上锁，二为了读取磁盘中对应的文件信息。

void
ilock(struct inode *ip)
{
  struct buf *bp;
  struct dinode *dip;

  if(ip == 0 || ip->ref < 1)
    panic("ilock");

  acquiresleep(&ip->lock);

  if(ip->valid == 0){
    bp = bread(ip->dev, IBLOCK(ip->inum, sb));
    dip = (struct dinode*)bp->data + ip->inum%IPB;
    ip->type = dip->type;
    ip->major = dip->major;
    ip->minor = dip->minor;
    ip->nlink = dip->nlink;
    ip->size = dip->size;
    memmove(ip->addrs, dip->addrs, sizeof(ip->addrs));
    brelse(bp);
    ip->valid = 1;
    if(ip->type == 0)
      panic("ilock: no type");
  }
}

ilock不释放sleeplock，所以在完成对inode操作后进程必须调用iunlock，释放sleeplock，唤醒其他想使用这个inode的进程。

void
iunlock(struct inode *ip)
{
  if(ip == 0 || !holdingsleep(&ip->lock) || ip->ref < 1)
    panic("iunlock");

  releasesleep(&ip->lock);
}

iput会释放传入的c指针，并且给原来指向的inode的ref--。当ref==0，itable中的该inode结构体就可能被重新利用

void
iput(struct inode *ip)
{
  acquire(&itable.lock);

  if(ip->ref == 1 && ip->valid && ip->nlink == 0){
    // inode has no links and no other references: truncate and free.

    // ip->ref == 1 means no other process can have ip locked,
    // so this acquiresleep() won't block (or deadlock).
    acquiresleep(&ip->lock);

    release(&itable.lock);

    itrunc(ip);
    ip->type = 0;
    iupdate(ip);
    ip->valid = 0;

    releasesleep(&ip->lock);

    acquire(&itable.lock);
  }

  ip->ref--;
  release(&itable.lock);
}

如果此时这个inode ip->ref == 1 && ip->valid && ip->nlink == 0，那么这个inode和它的内容blocks同时还会自磁盘上被释放。itrunc用来把文件内容删除，释放其内容blocks，修改inode type。

    可以看到iput同时也有清理inode的职责。思考一个情景：线程A调用了iput并且最终释放了inode，而此时有线程B在ilock中等待使用这个被A释放了的inode。这是不可能的，因为input唯一会清理input的前提就是 ip->ref == 1，而这个1必然是调用iput的线程A。假如B也在使用这个inode，那么这个inode的ref就不可能是1了，所以不成立。也不用担心别的进程调用ialloc在当前进程释放inode的期间把这个inode分配出去，因为只有在type=0的时候inode才可能被分配，而input在把type变成0之后仍然有锁，唯一的小问题是调用ialloc的进程可能会睡眠一会(从调用iupdate到释放锁期间：只有在iupdate之后别的进程才能发现这个inode可用了)。

    由于iput可能会修改磁盘上的inode，所以即使是一些只读的系统调用，比如read()，只要最终有调用iput，仍然会“写入”磁盘。换句话说，只读的系统调用如果使用文件系统，就也会涉及交易。

    input不会只因一个inode的nlink=0就把它释放，因为虽然没有目录指向这个inode，也可能有其他C指针指向这个inode。换句话说，可能有其他的进程正在对这个文件进行读写。虽然iput不会造成问题，但是如果nlink=0的时候崩溃发生，那么这个文件有可能被标记为已被分配，但是没有目录指向它。文件系统有两个解决这个问题的方法：

  1. 在开机后遍历所有inode，找到nlink=0但是type！=0的（也就是被分配但是没有目录指向的）

  2. 让文件系统在superblock中记录每一个nlink=0但是ref!=0的inode，当它们ref也等于0了就从记录中删除；在重启之后检查这个记录，释放所有记录中的inode。

xv6没解决这个问题。







8.10 代码：Inode 内容 Code：Inode content

    inode的真正的内容保存在不同的blocks中。用来代表磁盘上的inode的结构体dinode中保存着一个addr数组长13，前NDIRECT（12）个位置保存着NDIRECT个这个inode所使用的block number。这些block被叫做Direct block，因为在把inode读入内存中后，可以直接使用这些block number得到对应的block（共计12kB：1024*12，xv6的block为1024B）。第13个位置也是一个block number，这个block保存着剩下的NINDIRECT（256）个block的number，它们被叫做Indirect block，因为需要首先载入第13个block才能看到这些blocks的位置。

    bmap()函数封装了对于第n个block的访问的操作，它返回一个inode第n个block的真正的block num。

static uint
bmap(struct inode *ip, uint bn)
{
  uint addr, *a;
  struct buf *bp;
  //如果目标block在前十二个block中
  if(bn < NDIRECT){
    if((addr = ip->addrs[bn]) == 0)
      ip->addrs[bn] = addr = balloc(ip->dev);
    return addr;
  }
  //作为offset
  bn -= NDIRECT;
  //换句话说，bn不能超出12+256
  if(bn < NINDIRECT){
    // Load indirect block, allocating if necessary.
    //==0代表第13个block没有被分配
    if((addr = ip->addrs[NDIRECT]) == 0)
      ip->addrs[NDIRECT] = addr = balloc(ip->dev);
    //获取第13个block
    bp = bread(ip->dev, addr);
    a = (uint*)bp->data;
    //遍历间接block
    //不存在就创建
    if((addr = a[bn]) == 0){
      a[bn] = addr = balloc(ip->dev);
      log_write(bp);
    }
    brelse(bp);
    return addr;
  }

  panic("bmap: out of range");
}

它首先检查目标block是否为前12个，若不是，则需要把第13个block载入，然后再从中找。如果目标block不存在，bmap还会分配一个block。如果addr[bn]/a[bn]中的元素值为0，证明这个位置上还没有分配一个block。bmap遇到这种情况会分配一个block，并且修改0为刚分配的block的number。

    itrunc首先会释放前12个block，然后再释放间接blocks

    readi和writei获益于bmap。

int
readi(struct inode *ip, int user_dst, uint64 dst, uint off, uint n)
{
  uint tot, m;
  struct buf *bp;

  if(off > ip->size || off + n < off)
    return 0;
  if(off + n > ip->size)
    n = ip->size - off;

  for(tot=0; tot<n; tot+=m, off+=m, dst+=m){
    //使用bmap获取目标block
    bp = bread(ip->dev, bmap(ip, off/BSIZE));
    //读取最大值为1个block
    m = min(n - tot, BSIZE - off%BSIZE);
    //复制回用户空间
    if(either_copyout(user_dst, dst, bp->data + (off % BSIZE), m) == -1) {
      brelse(bp);
      tot = -1;
      break;
    }
    brelse(bp);
  }
  //tot为总读取量
  return tot;
}

readi首先会检查off，偏移量，没有超过文件边界，然后检查目标读取量是否超过从off到末尾的长度，若超出则更新读取目标长度。在下面的循环中开始读取。使用bmap获取目标block，每次读取的最大单位为1024Bytes，即1个block（OS的block）。最终返回读取的数据量

int
writei(struct inode *ip, int user_src, uint64 src, uint off, uint n)
{
  uint tot, m;
  struct buf *bp;

  if(off > ip->size || off + n < off)
    return -1;
  if(off + n > MAXFILE*BSIZE)
    return -1;

  for(tot=0; tot<n; tot+=m, off+=m, src+=m){
    bp = bread(ip->dev, bmap(ip, off/BSIZE));
    m = min(n - tot, BSIZE - off%BSIZE);
    if(either_copyin(bp->data + (off % BSIZE), user_src, src, m) == -1) {
      brelse(bp);
      break;
    }
    log_write(bp);
    brelse(bp);
  }

  if(off > ip->size)
    ip->size = off;
  iupdate(ip);

  return tot;
}

writei和readi类似，不同于三个方面：

  1. writei会增长file的大小
  2. 循环中使用copyin而非copyout
  3. 若使得文件大小改变需要更新inode.size。

    另一个函数stati读取inode的元数据放入stat结构体。用户函数stat就使用了它。







8.11 代码：目录层 Code：Directory layer

    目录的实现和文件差不多，两者都是inode。目录的inode的类型是T_DIR，并且其内容的block中保存的是目录条目（Directory entries）。这些条目都是dirent结构体

struct dirent {
  ushort inum;
  char name[DIRSIZ];
};

包括一个文件名（最长DIRSIZ(14)）和i-number。当一个dirent的i-num=0，说明这个条目没有被用。

struct inode*
dirlookup(struct inode *dp, char *name, uint *poff)
{
  uint off, inum;
  struct dirent de;

  if(dp->type != T_DIR)
    panic("dirlookup not DIR");

  for(off = 0; off < dp->size; off += sizeof(de)){
    //在内核中调用，不用关心用户空间地址
    if(readi(dp, 0, (uint64)&de, off, sizeof(de)) != sizeof(de))
      panic("dirlookup read");
    if(de.inum == 0)
      continue;
    if(namecmp(name, de.name) == 0){
      // entry matches path element
      if(poff)
        //填入off
        *poff = off;
      inum = de.inum;
      return iget(dp->dev, inum);
    }
  }

  return 0;
}

    函数dirlookup通过一个name在一个目录中找目标条目。如果找到了就会把传入的*poff赋值为这个条目在当前目录中的偏移量，然后通过iget返回这个条目中的inode的指针.

int
dirlink(struct inode *dp, char *name, uint inum)
{
  int off;
  struct dirent de;
  struct inode *ip;

  // 检查重名;ip的唯一用处
  if((ip = dirlookup(dp, name, 0)) != 0){
    iput(ip);
    return -1;
  }

  // 找空着的条目，dirent结构体
  for(off = 0; off < dp->size; off += sizeof(de)){
    if(readi(dp, 0, (uint64)&de, off, sizeof(de)) != sizeof(de))
      panic("dirlink read");
    if(de.inum == 0)
      break;
  }

  strncpy(de.name, name, DIRSIZ);
  de.inum = inum;
  //更新磁盘上的目录的内容block
  if(writei(dp, 0, (uint64)&de, off, sizeof(de)) != sizeof(de))
    panic("dirlink");

  return 0;
}

    dirlink()接受一个目录inode，name，和一个inum作为参数，把目标inum的inode放入这个目录中名为name的条目中。如果重名不会添加成功。使用循环去找没有分配的dirent。







8.12 代码：路径名称 Code：Path names

    对**路径（Path）**的访问涉及这条路径上每一个文件/文件夹（components）的dirlookup的使用。xv6中提供了两个函数:namei与nameiparent:

struct inode*
namei(char *path)
{
  char name[DIRSIZ];
  return namex(path, 0, name);
}

struct inode*
nameiparent(char *path, char *name)
{
  return namex(path, 1, name);
}

前者返回这条路径中最终的inode地址，后者返回这条路径中最终的inode的前一个inode，两者都会把最终的路径元素放入name。它们都是对namex的封装:

static struct inode*
namex(char *path, int nameiparent, char *name)
{
  struct inode *ip, *next;

  //检查路径开头
  if(*path == '/')
    ip = iget(ROOTDEV, ROOTINO);
  else
    ip = idup(myproc()->cwd);

  //skipelem把下一个元素的名称放入name
  //并且把不包含下一个元素之后的路径返回path
  while((path = skipelem(path, name)) != 0){
    ilock(ip);
    //检查inode类型，必须是目录
    if(ip->type != T_DIR){
      iunlockput(ip);
      return 0;
    }
    //如果设置了nameiparent bits，并且path值为null
    if(nameiparent && *path == '\0'){
      // 此时ip中的是倒数第二个路径元素，是一个目录
      iunlock(ip);
      return ip;
    }
    //获取下一个路径元素的inode
    if((next = dirlookup(ip, name, 0)) == 0){
      iunlockput(ip);
      return 0;
    }
    iunlockput(ip);
    //更新inode
    ip = next;
  }
  if(nameiparent){
    iput(ip);
    return 0;
  }
  return ip;
}

最开始这个函数会检查路径的开头：如果是/，就说明从根目录开始查询，如果不是，就从当前目录开始查询。 while循环依次遍历路径中的每一个元素，期间的元素必须是目录。当设置了nameiparent bits，并且path值为null（path现在保存的字符串的第一个路径元素是下下一个路径元素，当path==null说明下一个路径元素就是最后一个元素），返回当前ip，也就是倒数第二个路径元素，或者说目标inode的父inode。而在其他情况下，则使用dirlookup去获取下一个路径元素的inode，继续遍历直到skipelem的返回值为0（该函数对路径末尾的处理为返回0）。

    可以看到对路径的检索涉及很多block，这就导致它可能会与磁盘进行多次交互（如果这些block没有在缓存中）。在xv6的设计中，如果一个线程在路径检索的过程中卡在了一个和磁盘的I/O上，别的线程仍然可以对部分相同的路径进行检索：xv6中，每个路径元素（目录）都是被独立地上锁的，所以对于不同的目录的检索可以平行进行。

    不过这种平行看起来会带来一些危险：假如一个进程在检索某个路径，而另一个进程把；路径上的某一个目录释放掉了，那么会导致前者在检索一个不再存在的inode。但实际上这不会造成很大的问题的，因为前者使用当前目录的时候是通过iget获取的，这个函数增加了当前inode的ref，即使另一个线程清空了nlink，也仍不会导致当前inode立马被清理。另一个危险是死锁。比如在检索.目录的时候，next和ip会在某一时刻都指向.这个inode，如果同时获取两者的inode的锁会导致死锁。xv6避免了这一情景，因为在获取next的锁之前，ip的锁一定被释放了。这里也体现出了ilock和iget（dirlookup中调用）分隔的重要性。







8.13 文件描述符层 File descriptor layer

    Unix中，控制台，管道，和文件都被抽象成了文件，而文件描述符就是造成这样抽象的工具。每一个这样的资源都被封装在了一个file结构体中

struct file {
  enum { FD_NONE, FD_PIPE, FD_INODE, FD_DEVICE } type;
  int ref; // reference count
  char readable;
  char writable;
  struct pipe *pipe; // FD_PIPE
  struct inode *ip;  // FD_INODE and FD_DEVICE
  uint off;          // FD_INODE
  short major;       // FD_DEVICE
};

type保存着这个“文件”的类型。xv6为每个进程提供了自己的table of open files，或者说fd。每次对于open的调用实际上都创建了一个file结构体。如果多个进程独立地打开了同一个文件，它们会有不同的off，但是所使用的inode指针都是一样的（所以inode的ref不会增加）。一个文件可以在一个进程中以多个fd出现，比如使用dup；一个文件也可以在多个进程中出现，比如使用fork。两种情况中的fd指向的都是同一个结构体，结构体中有ref记录自己被指向的个数。

    所有打开的文件都被保存在一个全局的**文件表（File table）**中，ftable。长NFILE(100):

struct devsw devsw[NDEV];
struct {
  struct spinlock lock;
  struct file file[NFILE];
} ftable;

它暴露了几个函数：filealloc遍历这个ftable，找到可以使用的file结构体并分配; filedup增加file的ref并返回原file结构体指针（比如fork中就使用了它）; fileclose减少ref，并在ref=0的情况下关闭资源。这三个函数遵循文件系统函数中常见的模板。filtered/filewrite分别封装了不同的file类型对应的I/O函数：设备，inode，管道; filestat只允许inode和设备类型使用，封装了stati。

int
fileread(struct file *f, uint64 addr, int n)
{
  int r = 0;

  if(f->readable == 0)
    return -1;
  //类型判断
  if(f->type == FD_PIPE){
    r = piperead(f->pipe, addr, n);
  } else if(f->type == FD_DEVICE){
    if(f->major < 0 || f->major >= NDEV || !devsw[f->major].read)
      return -1;
    r = devsw[f->major].read(1, addr, n);
  } else if(f->type == FD_INODE){
    ilock(f->ip);
    if((r = readi(f->ip, 1, addr, f->off, n)) > 0)
      f->off += r;
    iunlock(f->ip);
  } else {
    panic("fileread");
  }

  return r;
}

回忆对于inode的操作需要获取锁，这一行为也保证了对于inode读写不会出现并发问题。







8.14 代码：系统调用 Code：System calls

// Create the path new as a link to the same inode as old.
uint64
sys_link(void)
{
  char name[DIRSIZ], new[MAXPATH], old[MAXPATH];
  struct inode *dp, *ip;
  //获取参数
  if(argstr(0, old, MAXPATH) < 0 || argstr(1, new, MAXPATH) < 0)
    return -1;
  //开始交易
  begin_op();
  //获取目标inode
  if((ip = namei(old)) == 0){
    end_op();
    return -1;
  }

  ilock(ip);
  if(ip->type == T_DIR){
    iunlockput(ip);
    end_op();
    return -1;
  }
  //更新
  ip->nlink++;
  iupdate(ip);
  iunlock(ip);
  //获取目标目录
  if((dp = nameiparent(new, name)) == 0)
    goto bad;
  ilock(dp);
  //填入目录
  if(dp->dev != ip->dev || dirlink(dp, name, ip->inum) < 0){
    iunlockput(dp);
    goto bad;
  }
  iunlockput(dp);
  iput(ip);
  //结束交易
  end_op();

  return 0;

bad:
  ilock(ip);
  ip->nlink--;
  iupdate(ip);
  iunlockput(ip);
  end_op();
  return -1;
}

sys_link接受两个参数，new，old（两个都是路径，而非只是目录名）：把old所指向的节点连接到new所指向的目录：即给一个节点创建新的链接（不会断开之前的链接）。这个系统调用首先会读取传入的参数并且通过begin_op()开始交易。在交易内部，会首先把old所指向的inode读取到进程中，如果这个inode是一个目录，则不处理，反之则继续：为new增加这个inode的nlink。然后找到new所在的目录：获取目录之后就可以真正地创建链接了dirlink(dp, name, ip->inum)。这里有一个问题：可以看到目标inode的nlink实际上在真正被链接之前就增加了，若崩溃发生或许会导致nlink和实际上的链接数量对不上。但这实际上是不会发生的：由于存在交易/提交的机制，这些对于不同的block的修改要么都发生，要么都不发生，即原子化。

    sys_link给已经存在的inode创建了一个新的名字。而create给一个新的inode创建一个新的名字。open()用它创建一个文件，mkdir()用它创建一个目录，mkdev()用它创建一个设备：

static struct inode*
create(char *path, short type, short major, short minor)
{
  struct inode *ip, *dp;
  char name[DIRSIZ];

  if((dp = nameiparent(path, name)) == 0)
    return 0;

  ilock(dp);

  if((ip = dirlookup(dp, name, 0)) != 0){
    iunlockput(dp);
    ilock(ip);
    if(type == T_FILE && (ip->type == T_FILE || ip->type == T_DEVICE))
      return ip;
    iunlockput(ip);
    return 0;
  }

  if((ip = ialloc(dp->dev, type)) == 0)
    panic("create: ialloc");

  ilock(ip);
  ip->major = major;
  ip->minor = minor;
  ip->nlink = 1;
  iupdate(ip);

  if(type == T_DIR){  // Create . and .. entries.
    dp->nlink++;  // for ".."
    iupdate(dp);
    // No ip->nlink++ for ".": avoid cyclic ref count.
    if(dirlink(ip, ".", ip->inum) < 0 || dirlink(ip, "..", dp->inum) < 0)
      panic("create dots");
  }

  if(dirlink(dp, name, ip->inum) < 0)
    panic("create: dirlink");

  iunlockput(dp);

  return ip;
}

由于也要涉及链接，显然需要文件inode和目录inode：和sys_link一样调用nameiparent获取目录，然后通过dirlookup检查是否有重名：如果有重名且inode类行为文件就返回上锁后的该inode（这实际上就是在open中以O_CREATE的方式打开后的一种结果），如果类型不是文件，就返回0。如果没有重名就为这个文件分配一个inode并赋值。若新建的inode是一个目录，会为目录初始化. ..两个dirent（通过dirlink，ip作为目录参数传入），分别指向当前目录和dp，即父目录。如果这个inode是不是目录，也通过dirlink，但是目录参数是dp。最终返回上锁后的新inode。create同时使用了ip,dp的锁，但是不可能死锁，因为ip由于dp未同步而不可能被别的进程看到。
