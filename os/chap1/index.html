导言：

    操作系统的工作是去让多个程序共享一台电脑且提供硬件单独无法提供的服务。操作系统本身可以看作一个程序与硬件，程序与程序之间中转站：例如一个文本编辑器不需要去关心它应当把数据放到磁盘的哪个部位，也不需要关心怎么让另一个程序读到它生成的文本——这些都是操作系统去关心的问题。而更本质地说，内核/操作系统实际上是很多停在内存中等待着被调用的代码：中断（Chap5）驱动着内核的用作。[5]

     MIT 6.S081/Fall 2020 课程基于xv6操作系统教学。

xv6系统简介[1]：

    xv6是由ANSI C编写，使用x86与RISC-V（架构的CPU）指令集的操作系统（本教材使用RISC-V）。它的前身是第六代Unix，即V6。课程使用的是现代重新实现后xv6，其整体相比Unix/Linux十分简单，自2006年被广泛使用于教学目的。

xv6中的 内核（Kernel），一种特殊的程序，提供给人们 接口（Interface） 和它交互。

当一个进程（Process）使用kernel提供的服务/接口的时候，该进程实际上出发了一个系统调用（System call）。

    本章着眼于介绍进程，内存，文件标识符，管道命令，以及文件系统的 UI（User Interface） 的使用。如果说kernel是硬件与程序之间的桥梁，那么shell，也就是用户接口/UI， 则是用户与计算机交互的桥梁：它把从控制台/终端的输入翻译成内核可以理解的语言。Shell本质上只是一个很“普通”的程序：对于同一个操作系统，人们开发出了很多shell，例如Mac OS用户主流使用的zsh以及bash。相比于其他系统，xv6所使用的shell十分简单。







1.1 进程与内存

     进程（Process），即“正在运行的程序”，依存于内存（Memory）。一个进程使用两部分内存：第一个部分是用户空间（user-space memory），包括指令，数据，堆栈等在计算机组成中熟知的“进程内存布局”中的元素；第二个部分是内核空间/内核态（Kernel space），这部分内存只有kernel可见。

    xv6，如同大多操作系统，采用了分时系统（Time-share）。在这种系统中，OS会分配可用的CPU资源给正在等待执行的进程。每一个进程都有属于自己的进程标识符PID（Process Identifier）[2]。 通过PID,可以实现调整进程的优先级或者杀死进程（kill）。下面主要介绍一些系统调用（System call）。这些调用是通过特定的指令/函数封装的，可以让程序在用户空间向kernel申请使用更高权限的服务。C语言又封装了这些调用的库，其中的方法同名于这些调用。

    fork()：在一个进程中使用fork()，可以让该进程“分支”出一个进程：前者称为父进程（Parent process/Calling process），后者称为子进程（Child process）。子进程与父进程有一模一样的内存布局，但是两者的内存是相隔开的。当调用fork()之后，父进程与子进程同时运行：也就是说父进程并不会默认等待子进程运行结束。除了创建子进程，在fork()被调用后，它会在父进程中返回子进程的PID，而在子进程中则会返回0.

int pid = fork(); //父进程中，变量pid的值将会是一个唯一数字，代表子进程的PID
                  //而在子进程的内存中，这个变量的值是0
if(pid > 0){      //这里检测当前进程是否为父进程
    printf("parent: child=%d\n", pid);
...

    exit(status)：当在一个进程中调用exit()，这个进程会“正常终止”。正常终止会自动关闭该进程的一切资源，比如分配的内存空间，曾打开的文件等（PS：反之也存在“非正常终止”）。exit()可以接受一个数字作为参数，来代表终止时的状态：通常0代表成功退出，1代表失败推出（与(非)正常终止无关）。

    wait(*status)：如上述，父子进程会同时运行，但是有时我们希望子进程运行完毕之后父进程再继续运行，这时就需要wait()系统调用。假设一个父进程A有一个子进程a，在A使用wait()后，它会停止运行开始等待a运行完毕。当a exit或被kill，wait()在A返回a的PID，A继续运行。wait()接受一个指针类型的参数，当子进程运行完毕，子进程的退出状态会被放到指针中，如果不关心退出状态，可以填入参数0。值得注意的是，wait()只能“wait”一个子进程：这不是说当计划使用wait()的时候，只能创建一个子进程，而是说当一个父进程的众多子进程中的一个终止后，wait()便返回该子进程的PID，父进程与剩余的子进程继续运行。当当前进程没有子进程时，wait()返回0.我们可以运用这一特性来“wait”所有子进程：while ((wpid = wait(&status)) > 0);[3]

...//接上一个例子
    pid = wait((int *) 0);             //进入等待状态
    printf("child %d is done\n", pid); //子进程结束
} else if(pid == 0){                   //这里判断当前进程是否为子进程
    printf("child: exiting\n"); 
    exit(0);                           //正常退出
} else {
    printf("fork error\n"); 
}

    exec(filePath, argv[]):这个调用实现在进程A中开启进程B。exec()接受两个参数：进程B的文件路径与一个包含B所需的参数的数组。在xv6中，使用 ELF（Executable and Linkable Format） 文件格式作为目标程序的格式（第三章内容）。当A使用了这个调用，B会被加载，将其与A替换（并不是直接替换内存空间，只是会为B新建内存，并且抛弃A的内存）。exec()运行完毕后并不会继续运行A的下一条指令：因为，在被B替代后，A已经完全不存在了。注：C语言的库中不包含exec()，但是有execvp()等使用exec系统调用的函数。这里使用的exec()是xv6封装的一个函数。

//进程A
char *argv[3]; 

argv[0] = "B";                  //传统上，作为参数传入的数组第一个元素为程序名
argv[1] = "hello"; 
argv[2] = 0;                    //当遍历到0时，遍历自动停止
exec("/bin/B", argv);           //意为：打开"/bin"下的B程序，并给予其argv中的参数
printf("exec error\n");         //假如exec成功运行，这条代码不会运行

基于这一特性，可以实现真正的“在进程C中开启进程D”，而非被后者代替。实际上shell使用这一特性，使得用户可以在shell，这一“普通”进程，中开启其他进程: 在此过程中，上述的fork(), wait()也都有被使用：main (user/sh.c:145)：

//shell的main源码的一部分

...

// Read and run input commands. 
while(getcmd(buf, sizeof(buf)) >= 0){  
    
    ......
    //省略对cd指令的判断部分
    ......

    if(fork1() == 0) 
        runcmd(parsecmd(buf)); 
    wait(0); 
    } 
    exit(0); 
    }

最外层的while获取到输入的指令，fork()创建了一个子进程：这个进程为目标的进程开启了一个“壳子”。换句话说，这个子进程只是开启一个新进程的消耗品，因为它会在未来运行exec后完全变成目标新进程。子进程通过runcmd()获取到了输入的指令，通过一系列判断，会识别出这一指令是否是为了运行一个新进程。之后在runcmd()中，exec()会被调用，最终该子进程会被转变为目标进程。

思考： 既然fork()之后总是使用exec()，为什么不把它们两个合并成一个函数？





1.2 I/O 与 文件描述符

     文件描述符（File descriptor） 是很小的数字索引值，该值代表一个被kernel管理/维护的Object，这个“Object”可以是文件，目录，设备等，可以抽象地理解为“文件”。一个进程可以通过fd来与目标“文件”读写。每个进程都有属于自己的文件描述符表（File descriptor table），它记录着这个进程的所有fd。通常来讲，fd=0所指向的“文件”是输入文件，1所指向的是输出文件，2指向的是输出报错信息的文件——三者组成I/O（Input/Output）。默认情况下，三者都指向控制台（console），即键盘和显示器（在现代计算机中，终端和控制台的界限比较模糊，可以粗糙地类比理解[4]）。本节介绍I/O 与 文件描述符相关的系统调用。

    read(fd, buf, n): read()函数需要三个参数，分别为一个fd，一个buffer（指针），和单次读取的最大byte量n。该函数的运行过程解释为：从fd所指向的“文件”中单次最多读取n个byte并复制这些数据到buf指向的内存空间中，最后返回本次读取的bytes量。通常返回的bytes个数=n，除非剩余可以读的bytes个数 < n。若已经读到了“文件”末尾，返回0。

     每一个fd所指向的文件本身都有一个偏移量（offset）。这个偏移量的作用是记录当前位置，标记其为下一次读写的起始位置。比如一个文件中有一个字符串"Apple"，那么在最开始，偏移量 = 0。加入调用read()读取这个字符串，n = 2，那么Ap会被复制到一个buffer中，偏移量 + 2，下次的读写将从第2个p开始。

     write(fd, buf, n): 它从buf中读取最多n个bytes并写入fd中，返回读取的byte个数。他也同样遵守偏移量。

char buf[512]; 
int n;

for(;;){    //从fd = 0指向的“文件”读取数据，每次最多读512bytes
    n = read(0, buf, sizeof buf);
    if(n == 0)       //一直读取，直到read返回0，即到文件末尾
        break; 
    if(n < 0){
        fprintf(2, "read error\n");
        exit(1); 
    }
    if(write(1, buf, n) != n){
        fprintf(2, "write error\n");
        exit(1); 
    }
}

     close(fd)：释放一个fd，也就是这个数字不再指向原来所指向的“文件”。这使得该fd被空闲出来，在未来可以被重新使用。

     open("filePath", mode)：该函数创建一个fd指向第一个参数（一个文件路径）所指向的目标文件。第二个函数是打开的模式，可选值有O_RDONLY, O_WRONLY, O_RDWR, O_CREATE, O_TRUNC，分别对应“只读，只写，读写，‘若不存在就新建’，删除所有内容”。这个函数所分配的fd就是最小可用的fd。Linux中规定，一个进程最多可以有1024个fd。

     fork(), exec()在 I/O 中的特性：fork()创建子进程后，父进程的fd表会完整地复制给子进程，在子进程中使用exec()，虽然子进程会被目标进程代替，但是这个子进程继承来的fd表仍然存在（因为fd由kernel管理），换句话说，新的目标进程仍然可以访问之前子进程的完整fd表。

    close(), open(), fork(), exec()的搭配使用可以实现I/O重新定向（Redirection）。

//这实际上就是xv6实现I/O 重新定向的原理

char *argv[2]; 
argv[0] = "cat"; 
argv[1] = 0; 

if(fork() == 0) {
    close(0);          //释放fd = 0（默认指向console）
                       //打开目标文件，自动分配最小可用fd，即0
    open("input.txt", O_RDONLY);
                       //完成从将fd = 0 指向console重新定向为input.txt
    exec("cat", argv); 
}

这里解释了不能把fork(), exec()合并成一个函数的原因：fork()之后可能会重新定向fd。假如两者合并成了forkexec()，并非完全无法实现重新定向，只不过各有缺点。这里有三种方法：

  • 在 forkexec()前就在父进程修改fd表，那么目标的新进程会继承父进程的表，从而达到效果。缺点：修改了父进程fd，十分不乐观。
  • 给forkexec()设置特定参数（如目标文件路径和想要替换的fd），并且内置重新定向的代码。缺点：违背模块化。
  • 让每个进程自己去重新定向。缺点：违背了用户空间与kernel相隔的理念。

    注意： 虽然 fork()复制了父进程的内存布局和fd表生成了子进程，但是由于“偏移量”是“文件”的属性，父子进程将会共享同一个偏移量。

if(fork() == 0) {           //判断当前是否为子进程
    write(1, "hello ", 6);  //子进程写入数据
    exit(0); 
} else {
    wait(0);
    write(1, "world\n", 6); //父进程在子进程结束后写入数据
}
//fd = 1 所指向的文件将会为 "hello world"

共享偏移量这一特性可以让多个进程顺序添加数据。在命令行中可以实现如下操作：(echo hello; echo world) > output.txt， 解释为两次分别调用echo提供不同的参数，依次输出到 output.txt。

    dup(fd)：这个函数返回一个新的fd，指向作为参数传入的fd所指向的“文件”。换句话说，dup(fd)使得多个fd可以指向同一个“文件”。同样地，分配fd的规则遵守最小可用数字，且偏移量仍然共享于不同的fd。

nfd = dup(1);    //让“nfd”指向fd = 1 所指向的“文件”
write(1, "hello ", 6); 
write(fd, "world\n", 6);

使用dup()可以实现如下命令行：ls existing-file non-existing-file > tmp1 2>&1,解释为将2指向1所指向的“文件”，输出existing-file 与 non-existing-file的文件名到默认的输出fd，即fd = 1（这里1指向了tmp1）。如果文件不存在，输出报错到默认的输出报错fd，即 fd = 2. 由于这里把2重新定向到了1，正常输出与报错输出都会输出到fd = 1所指向的文件。







1.3 管道命令

     管道（Pipe） 是以一对fd的形式暴露给进程的kernel buffer，占据内存中的空间。它可以被抽象地理解为一个“文件”：这个“文件”有两个fd指向它。管道的作用是实现了两个进程之间的通信：通常，两个进程各使用管道的一对fd中的一个，其中一个只读，一个只写。换句话说，管道是单向的。

//这是一个父子进程之间通信的例子

int p[2];           //这是一个int类型的数组，其内容未知且不重要
char *argv[2];

argv[0] = "wc"; 
argv[1] = 0;

pipe(p);            //生成一个管道：把数组p第一个元素变成了输入端，第二个元素变成了输出端
if(fork() == 0) {
    close(0);
    dup(p[0]);      //fd重新定向，使得0指向管道的输入端
    close(p[0]);    //0已经指向了管道这个“文件”，原来的输入端fd可以关闭了
    close(p[1]);    //保证程序安全运行，最好在输入内容的进程中关闭输出端
    exec("/bin/wc", argv);  //wc继承fd表
} else {
    close(p[0]);    
    write(p[1], "hello world\n", 12);   //输入内容
    close(p[1]);                        //关闭资源
}

注： 上面的例子在子进程中关闭了输出端fd，这不是必须的，但是可以增强运行的安全性。这是因为管道的特性所致：当管道中没有数据（调用输出端的进程并没有向管道输出内容），另一端的程序使用输入端读取数据的时候，会一直等待数据出现。结束这一等待有两个途径：数据出现，或所有输出端的fd都被关闭。子进程继承了父进程的一对fd，也就是这个语境下的管道，假如它不关闭输出端fd，而又碰巧调用了输入，它就有可能一直等待下去。不过这不意味着一个进程无法同时使用同一个管道的两端，只是没有任何必要并且危险。

     命令行 grep dir sh.c | wc -l有和上面这一大段代码类似的效果。这个命令行解释为grep去搜索文件sh.c中dir这一正则表达式，它会输出结果给wc -l这一命令，意为“展示dir在sh.c出现的行数”。中间的管道连接了grep的输出和wc的输入。对于操作系统，它会为整个这条命令fork出一个子进程，这个子进程创建管道连接左右两个命令。这个管道，分别作为默认输出（1）和默认输入（0）。然后为两个命令又一次分别fork出一个进程：这个情景便是兄弟进程之间的通信，而非父子进程。当两个进程都结束了，子进程才会消失。 在一些场景中，可能出现类似a | b | c的情况：shell会为a和b | c fork出两个进程，而进程又会分别为b和c分别fork出两个进程：可以看出，多个管道命令的使用可以构建出一个进程树，每个节点（进程）都在等它的左右节点（因|而被fork出的两个进程）结束运行。

    形如echo hello world | wc 的命令其实可以分开写成echo hello world > ./out.txt; wc <./out.txt两段命令：不同于管道创建了一个虚拟的文件，后者真的创建了一个叫out.txt的文件。

     管道命令的优点：

  • 管道命令十分“干净”：不像上面需要创建一个真实文件的命令，管道会自动清理自己的资源。
  • 管道可以传输任意长的数据，而创建真实文件则需要关注磁盘容量。
  • 管道允许平行的管道命令执行：两个通信的进程可以同时运行。而创建真实文件需要输出信息的程序运行完毕然后使用这个信息的程序才能开始。
  • 管道有读写的阻断机制：当没有输出的时候，输入端不会认为输出结束，这更有效率。






1.4 文件系统

     xv6的文件系统提供数据文件（Data file） 和目录（Directory），而目录又包含数据文件和其他目录。一个形如/a/b/c的路径（Path） 意为： 根目录（Root） 下的a中的b里的c，c可能是一个数据文件或者目录。在xv6中，不以\开头的，自动识别为相对于“当前目录”下的文件/文件夹。假设当前身处根目录（由\体现），有两种访问c的方式：

//第一种：通过改变“当前文件夹”
//chdir，即cd

chdir("/a");        //进入目录a，当前目录变成a
chdir("b");         //使用相对路径进入a中的b
open("c", O_RDONLY) //当问目标文件

//第二种，即直接打开目标文件

open("/a/b/c", O_RDONLY);

     xv6中提供了创建目录（文件夹）和文件的系统调用：

mkdir("/dir");          //意为“make directory 'dir'”
fd = open("/dir/file"   O_CREATE|O_WRONLY);     //新建文件
close(fd);

文件名其实是一种抽象的存在：文件名和文件本身是分离的，或者说文件并没有真正的“名字”。文件被叫做inode(index node)，它可以有多个“文件名”，叫做links。每个link由一个目录里的 入口（Entry） 组成，这个入口包括文件名和指向inode的指针。inode中包含这个文件的一切原数据（Metadata），比如文件类型，长度，和文件内容在磁盘上的位置，以及这个文件的link/文件名的数量。我们可以通过系统调用为一个文件创建多个link：

open("a", O_CREATE|O_WRONLY);   //假如a不存在，那么open在创建一个新文件的同时，会给这个新文件一个link-“a”
link("a", "b");                 //把“b”也作为文件“a”的一个名字
unlink("a");                    //让“a”不再作为这个文件的名字
unlink("b");                    //在让“b”不再作为该文件的文件名后，这个文件变成了无名的文件，且该文件占用的资源“可能”会被释放。

在一个文件变成无名文件之后，不意味着这个文件所占用的资源一定会释放：

fd = open("d", O_CREATE|O_RDWR);    //创建一个文件，一个link-“d”，得到一个fd。
unlink("d");                        //此时文件虽然无名，但资源仍未被释放，因为fd仍然没有关闭。

也就是说，如果想要让一个文件使用的资源被释放，它的link和fd都需要被关闭。

     补充： Unix向用户以“用户级程序”的方式提供了一系列的命令比如mkdir, rm等来操作文件。换句话说，这些命令可以被看作通过exec()运行的程序。但是cd，或chdir(), 则是内嵌在shell中的。这是因为前者会通过fork()出的子进程执行exec()，但是同样的逻辑运用于cd的话，那就是改变了子shell的运行目录，而无法涉及父shell。
