导言：

     操作系统通常会 交错（Interleave） 运行不同的活动，这种交错主要由于3种行为

  1. 多核CPU共用ram。由于一台计算机中可以有多个CPU/核心，但是只有一个内存空间，这就导致多个CPU可能在同一时刻访问一个地址并修改。

  2. 即使在单核中，CPU也会进行分时，在不同的线程之间切换，而不同的线程可能会访问同一个数据。

  3. 设备中断。设备中断可能会在不恰当的时间发生，导致修改了它中断掉的代码正在使用的数据。

    内核中有很多这样的例子。其中一个被共用的内存资源就是代码本身。比如多个CPU可能同时使用kalloc.c，后者会去修改那个记录着可用分页的自由表。当两个CPU同时调用其中函数的时候，内存的自由表的表头可能同时会被两个CPU获取。为了禁止这种并发导致的不合法行为，操作系统设计者通过并发控制（Concurrent control）解决。它的实现有多种，锁（Locks） 就是其中一种。一个锁保证了在一个瞬间只有一个CPU（进程）可以拥有它，使得被“锁住”的数据只能被这个CPU访问，从而实现对数据的保护。锁存在一个劣势：由于它会序列化并发的进程，也就是说多个进程访问同一个数据，后面的会等待前面的进程结束再运行。这让并发不再并发，降低性能。随意也随之出现了Lock-free Programming，即不需要锁的实现，但是更加复杂。







6.1 竞争冒险 Race condition

    举个例子：假设有函数push，用来给一个链表添加节点到头节点的位置上。

struct element {
    int data;
    struct element *next; 
};

struct element *list = 0; 

void 
push(int data) 
{
    struct element *l;

    l = malloc(sizeof *l); //13
    l->data = data;        //14
    l->next = list;        //15
    list = l;              //16
}

    假如现在有两个CPU共享一个链表并且几乎同时调用了这个方法。当它们都运行到第15行的时候，某个CPU会率先执行赋值操作，把新建的节点指向之前的头节点list。而在它执行第16行之前，另一个CPU也执行了第15行，让自己的新建的节点也一样指向了之前的头节点。这里就出现了不合法的行为：有两个新建的元素都指向了之前的头节点。当地一个CPU运行了16行之后，后一个CPU将会覆盖前者的修改，导致 更新丢失（Update lost）。

    16行的更新丢失被叫做Race condition，存在“竞争关系”。这种竞争情况导致的结果难以预测，具体的结果取决于它准确地发生在什么时候。这种情况很难重新实现，因为要求的巧合程度较高。比如在debug的时候加入了一个printf，就可能导致bug不存在了。

    使用锁就可以解决上述问题。锁确保了CPU之间的 互斥关系(Mutul-exclusive)，即只可能有1个CPU能执行某些关键代码.

//修改上面的代码...
struct element *list = 0; 
//创建一个锁
struct lock listlock;

void push(int data) {
    struct element *l;
    l = malloc(sizeof *l);
    l->data = data;
    //关键部分
    acquire(&listlock); 
    l->next = list;
    list = l; 
    release(&listlock);
}

被锁住的部分（即acquire(), release()之间的代码）被叫做关键部分（Critical section）。这个例子中可以说“锁在保护list”。当我们说锁在保护某一数据的时候，实际上指的是锁在保护这一数据的 不变量（Invariant） 。这里说的不变量只是数据/数据结构的一种属性，而不是真正的某个值。在上面的例子中的不变量就是“list永远指向第一个元素，而所有元素都指向下一个元素”这一属性。被保护的意思不是说锁让不变量永远为真，而是说在不变量被破坏（不为真）到恢复（为真）的期间能正常运行。比如上面的例子中的l->next = list;就导致list并没有指向第一个元素，代表着不变量不为真，但是下一行又一次让list指向了第一个元素，即不变量得到了恢复。

    锁的代价就是在序列化对关键部分的并发之后的性能降低。多个进程想要获取同一锁的情况叫做冲突（Conflict）或者锁竞争（Lock Contention），内核的设计要避免出现过多的锁竞争.同时，获取锁和释放锁的位置也会影响性能。比如即使在push最开始就获取锁也是正确的，但是会造成更多指令被序列化。







6.2 Code：锁   Code: Lock

    xv6中提供了两种锁:自旋锁（Spinlocks） 和 睡眠锁（Sleep-locks）。本节主要介绍自旋锁。自旋锁本质上只是一个结构体，其中最重要的属性是locked，用来表达锁是否获取了。如果为1则说明锁被获取了，其他进程不能继续运行（后续的进程会一直执行acquire中的一个循环，直到获得了锁，这就是为什么它叫自旋锁）。

// Mutual exclusion lock.
struct spinlock {
  uint locked;     

  // For debugging:
  char *name;        // Name of lock.
  struct cpu *cpu;   // The cpu holding the lock.
};

    观察这个错误 版本的acquire()：

//错误演示：
1  void acquire(struct spinlock *lk) 
2  {   
3    //不停自旋
4     for(;;) {
5         //如果结构体内的locked属性还没为1
6         //说明没有进程正在持有该锁
7         if(lk->locked == 0) {
8             //成功获取
9             lk->locked = 1;
10            break;  
11       }
12  }
13 }

看起来似乎没什么问题，但实际上这个函数本身也可能会由于并发而错误运行。这里存在一个问题就是第7行的代码可能造成像是之前的push中的错误一样，两个CPU可能同时执行这一行的指令。也就是说两个进程都认为此时没有其他进程获取这个锁，并且进入第9行。这导致两个CPU中的进程都成功获得了锁，使得锁没什么用。所以这一判断(7)和赋值(9)就变成了实现锁的障碍。大部分CPU都会支持一些指令，从硬件层面完成这两个步骤，让这两个步骤变成真正意义上的一个步骤，即原子化：即要么完全发生，要么完全不发生。RISC-V使用的指令叫做amoswap r1,r2,a，即 Atomic memory swap。前两个r是寄存器，a是一个物理地址. 这个指令的大致步骤为

lock addr; 
tmp <- *addr; 
*addr <- r1;
r2 <- tmp;
unlock addr;

所以把两个寄存器都传同一个，就可以交换这个寄存器和物理地址里的值。同时有独立的硬件保护（硬件层面上锁而非软件层面）被操作的地址，让别的CPU无法同时访问。CPU具体的实现方法依赖于内存系统的实现，所说的“硬件”可能是内存管理器，也可能是缓存相关的硬件等等。

    xv6中使用了**__sync_lock_test_and_set()**这一C的标准库中的函数，它封装了上述的指令。当这个函数被调用的时候，会调用这一指令把locked设置为1（无论之前的值是什么），并返回locked之前的值，假如返回值是0，则说明这个锁之前没有被其他进程正在使用，即可继续运行；如果返回值是1，则会持续循环，直到返回值为0.

//acquire的正确实现：
// Loops (spins) until the lock is acquired.
void
acquire(struct spinlock *lk)
{
  push_off(); // disable interrupts to avoid deadlock.
  if(holding(lk))
    panic("acquire");

  // On RISC-V, sync_lock_test_and_set turns into an atomic swap:
  //   a5 = 1
  //   s1 = &lk->locked
  //   amoswap.w.aq a5, a5, (s1)
  // 这里就是“自旋的开始”
  while(__sync_lock_test_and_set(&lk->locked, 1) != 0)
    ;

  __sync_synchronize();

  // Record info about lock acquisition for holding() and debugging.
  lk->cpu = mycpu();
}

在设置locked之后，acquire函数会去设置lk->cpu来为debug服务。release()有类似的实现，只不过方向和acquire完全相反。

void
release(struct spinlock *lk)
{
  //微获取就释放
  if(!holding(lk))
    panic("release");
  //清空CPU
  lk->cpu = 0;
  
  __sync_synchronize();
  __sync_lock_release(&lk->locked);
  pop_off();
}

它首先会清空lk->cpu，同样地它会调用一个函数__sync_lock_release来原子化地把locked换成0.后面这个函数同样封装了amoswap。不同于acquire, release不需要把n步合并成一步。但是这里仍然使用了这个amoswap是因为在某些情况下，store可能会分步进行，这取决于更多底层的实现。







6.3 锁的使用  Code: Using Locks

    OS很多地方都使用了锁，而锁的使用需要考虑 “给谁用，用多少”。通常有两个场景需要锁：一个会被一个CPU 写，被另一个CPU读/写的变量，锁可以帮助这两个操作不重合；给不变量使用，就像push中所指出的。除了这一考虑，还需要思考什么时候不用锁防止滥用。其中一个情况就是不存在并发/平行运行的时候：此时只需要关心一个线程。有一些简单的操作系统曾运行于单核，当它们被移植到多核处理器上之后，常常会实现Big kernel lock。在这种设计下，一旦进入内核，便会获取一个锁，直到返回到用户空间才会释放：一个大锁锁住了整个内核，使得并发不可能。xv6中存在锁很大程度牺牲性能的例子，比如kalloc。当多个进程同时申请堆内存的时候，这些进程会被序列化，在得到锁之前只能等待。一个解决方法是使用多个自由表来保存可用的page，同时使用多个锁去分别保护每一个自由表。正向的例子是xv6对文件的管理：每一个文件都有自己的锁，所以处理不同文件的进程可以并行。甚至在此基础上可以更加优化：比如使用不同的锁给一个文件不同的部位，这样多个进程就能同时操作同一个文件了。

    现在有三种理解锁的视角：

  1. 锁避免更新失败，比如push中所演示的。

  2. 锁使得多个操作/指令原子化

  3. 锁保护不变量







6.4 死锁和锁的顺序 Deadlock and lock ordering

    在理想状态下，当任何代码使用多个锁的时候，它们获取锁的顺序一定要相对位置上一样。举个例子，有两段代码，代码1先获取 A锁 再获取 获取 B锁 再获取 A锁，此时可以说这两个代码的获取锁的顺序是不同的。而这种情况可能导致死锁（Deadlock） ：假如代码1刚好获取了A锁，代码2获取了B锁，在运行一段时间之后，在两者都没有释放自己的锁的情况下，前者又去获取B锁，后者又去获取A锁，那么这个时候，它们就会互相等待对方先释放自己的锁，这就会造成两个进程永远自旋，导致死锁。 这种对锁使用的相对位置的要求叫Lock-order chain。这种链对OS来说是全局的，即每个进程都需要遵守。xv6中有很多针对进程锁（每个进程都有自己的锁，保存在结构体proc中），长度为2的链。比如说在上一章的consoleintr中，当接收到一行输入之后，这个函数会先获取cons.lock，然后再获取进程的锁，从而叫醒睡眠的进程：cons.lock必须先于获取进程的锁。xv6中最长的链存在于文件系统中。

    设计一个很好的chain是很复杂的，有以下3个难点：

  1. 有时候程序运行的逻辑和链的顺序相反，比方说一个获取了一个锁的模组使用了另一个模组且再次获取了一个锁，而根据链，后面这个模组中的锁应当先获取

  2. 有时候并无法得知下一步获取的锁是什么。比如模组A使用了模组B，前者需要知道后者使用了什么锁，这样才能根据链看是否符合顺序，但由于模组化的抽象封装，这种一个模组获取另一个模组内部内容的行为不可取

  3. 锁越多对性能的副作用越小，但是会造成设计更加复杂。比如为了优化性能可能要分裂一个数据结构并使用更多锁从而实现更好的并行。但是最优的分裂很难达成，而分裂后也需要重构之前使用的锁和这个数据结构的代码，工作量大。

通常设计锁的使用的过程：1.设计一个粗糙（Coarse-grained） 的锁的使用。2.测量性能，性能差说明有锁竞争。3.重新设计







6.5 锁和中断处理器 Locks and interrupt handlers

    有时候锁会被用于保护会被线程和设备中断处理代码使用的一些数据。比如clockintr()会去更新tick，而sys_sleep可能在同一时刻去读取ticks。假设sleep先去读取了这个ticks，并且在读取前获取了一个锁阻止其他进程访问这个数据。在它释放这个锁之前，一个时钟中断发生了，clockintr想要去更新ticks，那么后者会把sleep打断并开始运行。可由于ticks上了锁，该中断也需要等待锁被释放次才能更新ticks。这时死锁就出现了：中断等着sleep释放自己的锁，sleep等待中断运行结束并返回。（死锁并不是只有两个锁才能出现）。换句话说，中断可能去获取它所中断的代码已经获取的锁。

    为了避免这种情况的发生，xv6在一个进程获取了锁之后会禁止中断在该CPU上发生。换句话说，当一个CPU在处理关键部分的内容时，此时发生的中断应当交给其他CPU去处理，这样虽然会中断其他CPU，但是并不会导致当前CPU死锁（如果发生了上面的情况，此时中断可以在另一个CPU上自旋）。

    实现方式：在acquire和release中修改同一个计数器，这个计数器是当前CPU（核心）的一个属性（保存在这个CPU核心的结构体内），当这个计数器不为0的时候，证明当前CPU正在处理关键内容且此时中断禁止。比如在acquire中，在修改locked的值之前，第一个调用的函数就是push_off。

void
push_off(void)
{
  int old = intr_get();
  //禁止中断
  intr_off();
  //查看计数器
  if(mycpu()->noff == 0)
    mycpu()->intena = old;
  mycpu()->noff += 1;
}

这个函数会首先禁止中断发生，增加mycpu->noff的值。禁止中断通过调用函数intr_off实现，底层是一个RISC-V的指令。noff就是上述的“计数器”。在修改locked之前修改noff很重要：假如顺序相反，那么可能在修改了locked还没有修改noff（禁止中断）之前有中断发生。正确的顺序防止窗口期存在。

    release也是相似的，只不过完全相反，它会在locked修改之后调用pop_off并在检查noff==0后开启中断。

void
pop_off(void)
{
  struct cpu *c = mycpu();
  if(intr_get())
    panic("pop_off - interruptible");
  if(c->noff < 1)
    panic("pop_off");
  //减少noff
  c->noff -= 1;
  if(c->noff == 0 && c->intena)
  //开启中断
    intr_on();
}

由于一段代码可能会使用多个锁，noff这个计数器会在锁增加的过程中同样增加，保证即使有的锁被复原之后，计数器仍能保持一个安全的值（如果在中间的一个pop_off中发现noff!=0，那么中断仍然不会开启）。







6.6 指令和内存顺序 Instruction and memory ordering

    在一段连续的指令中，编译器可能为了优化把代码顺序打乱，CPU也可能不会按顺序依次执行代码，而优先执行某些指令。当然这些指令是相互独立的，也就是说这种不按顺序执行的结果一定是正确的。可是如果考虑锁存在的场景，这种重新排序可能让锁失效。比如：

1 l = malloc(sizeof *l); 
2 l->data = data; 
3 acquire(&listlock); 
4 l->next = list; 
5 list = l; 
6 release(&listlock);
7 //假如第4行在这里

假如第4行在第6行之后执行，那么就是说当执行第4行之前，实际上已经没有锁来保护list了。此时可能有别的进程获取到了这个锁，然后它会发现这个list指向null（因为此时第5行也运行完了）。在xv6中，使用函数__sync_synchronize()去让在锁释放之前，形成一个内存屏障（memory barrier），告诉CPU和编译器，所有在这调用这个代码行之前的代码，都只能在这个屏障之前排序，也就是说屏障之前的任何指令都不可能和之后的指令有任何重新排序。由于acquire和release都有调用它，这保证了这两个函数之间的所有代码不会超出两者之间的屏障。







6.7 睡眠锁 Sleep-locks

    当一个进程得到了自旋锁，这个CPU将无法yield：

  1. 假如yield，那么可能会导致另一个线程也尝试需获取同一个锁，进入自旋，自旋发生后可能会阻碍第一个线程继续运行，造成死锁。

  2. 同时，在lock的过程中出现yield本身也破坏了CPU持有锁的期间不能出现中断的规则。

sleep-locks的出现解决了睡眠的缺点。其获取和释放函数分别是acquiresleep, releasesleep。前者会调用sleep，sleep会让线程进入等待并且自动yield该CPU，给其他线程运行机会。由于Sleep-locks允许中断，它不能用在处理中断的代码中（这些代码禁止中断），也不能运行在被自旋锁保护的代码中。Sleep-lock适合保护长代码，自旋锁适合短代码。







6.7 UART中的锁 Case study

    uart这个设备有一个自己的锁uart_tx_lock。这个锁保护着uart的buffer指针的不变量，比如“uart_tx_r在uart_tx_w后面”，“前者与后者之间是准备输出的数据”，“后者与前者（注：这个buffer使用了循环数组）之间是可用的缓存空间”等等。

    以uartputc为例。这个锁从最开始就被获取，相当于锁住了整个UART设备的输出。

void
uartputc(int c)
{
    //获取这个锁
  acquire(&uart_tx_lock);

  if(panicked){
    for(;;)
      ;
  }

  while(1){
    if(uart_tx_w == uart_tx_r + UART_TX_BUF_SIZE){
        //输入一个chan和lock 第七章内容
      sleep(&uart_tx_r, &uart_tx_lock);
    } else {
      uart_tx_buf[uart_tx_w % UART_TX_BUF_SIZE] = c;
      uart_tx_w += 1;
      uartstart();
      release(&uart_tx_lock);
      return;
    }
  }
}

由于一直到返回之前锁都存在，这个锁将会贯穿对于uartstart的调用。在uartstart的执行过程中，会对UART的寄存器进行读写且修改uart_tx_r。由于锁的贯穿，这些寄存器和指针也被保护：同时一时间，一个寄存器只能有一个写入线程/读出进程。







6.8 练习 Exercises

  1. Comment out the calls to acquire and release in kalloc (kernel/kalloc.c:69). This seems like it should cause problems for kernel code that calls kalloc; what symptoms do you expect to see? When you run xv6, do you see these symptoms? How about when running usertests? If you don’t see a problem, why not? See if you can provoke a problem by inserting dummy loops into the critical section of kalloc.

  2. Suppose that you instead commented out the locking in kfree (after restoring locking in kalloc). What might now go wrong? Is lack of locks in kfree less harmful than in kalloc?

  3. If two CPUs call kalloc at the same time, one will have to wait for the other, which is bad for performance. Modify kalloc.c to have more parallelism, so that simultaneous calls to kalloc from different CPUs can proceed without waiting for each other.

  4. Write a parallel program using POSIX threads, which is supported on most operating sys- tems. For example, implement a parallel hash table and measure if the number of puts/gets scales with increasing number of cores.

  5. Implement a subset of Pthreads in xv6. That is, implement a user-level thread library so that a user process can have more than 1 thread and arrange that these threads can run in parallel on different CPUs. Come up with a design that correctly handles a thread making a blocking system call and changing its shared address space.
