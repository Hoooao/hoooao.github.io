数据库管理系统概述 Database Management System

导言：

    **数据库管理系统（DBMS：Database Management System）**由大量相互关联的数据以及操作这些数据的程序组成。数据的集合被叫做数据库。DBMS的目的是实现一个高效的 存储（Storage） 与 取出（Retrieve）。数据的管理主要有三个方面：1.对数据存储的结构的定义； 2.对数据的操作； 3.安全性。







1.1 DBMS应用 DBMS Applications

     被DBMS管理的数据有三个特点：高价值，数据量大，并发使用。其本质上是一个复杂的数据管理系统。早期，这个系统是现实世界的后勤部门，处理纸面数据。而现代的DBMS是一个软件系统，有强大的交互性和操作便捷。其主要应用体现在两个方面：1.线上交易处理(OLTP：Online Transaction Processing)：即对数据库最熟知的使用，体现为用户大量访问数据，但每次访问只涉及数据库中一小部分数据； 2.数据分析，通常适用于商业决策。







1.2 DBMS目的 DBMS Purposes

    OS的 文件管理系统(File-processing system) 一定程度上可以看作一个DBMS。假设大学存在一个课程注册的需求，那么会有三个子需求：添加删除学生，教师，和课程；注册学生进入课程；分配成绩，计算GPA，生成成绩单。看起来需求不多，可以使用几个程序实现。但这样的系统的性能和拓展性会很差。其缺点主要体现在以下几个方面：

  1. 数据冗余&不连贯（Data redundance & inconsistency）。在长期使用后，不同的文件（例如不同的部门的文件）可能会使用不同的文件格式，而不同的需求可能会使用不同的语言实现。同时，同一份数据可能会出现在多个文件中。比如一个双专业的学生的信息可能会出现在两个部门的文件中（冗余）。而且当这个学生修改了自己的手机号后，这个修改可能无法同步到所有部门的文件（不连贯）。
  2. 访问困难。主要体现在对有特定 属性（Attribute） 的数据的访问。比如有人想统计所有走读生的信息，那么就需要一个新的程序来检索文件：程序开发导致效率低下且使得整个系统臃肿。而使用原有的程序可能无法完美实现该需求。
  3. 数据隔离。同一类数据可能分布在不同的文件中，再次层架访问困难。比如走读生的信息没有集合与一个文件，而是分散在各个部门的文件中。
  4. 原子性（Atomicity）。一个操作要么（成功）发生，要么完全不发生：不能发生一半。比如给一个课程分发预算，大学预算总额的减少与课程预算的增加一起发生，不能只发生一个。
  5. 并发（Concurrency）。对单一数据的同时可的不同操作
  6. 安全性。应当保证不同的用户有不同的访问权限。

DBMS，提供通用的方法，用来解决这些问题。







1.3 数据模型 Data Models

    数据模型是数据库之下的结构，它规范了对数据的描述，数据关系，数据语义，以及对数据的一些限制。存在四种数据模型分类

  1. 关系模型（Relational Model）。关系模型使用一系列 表单（Tables） 去描述数据的种类和关系。每一个表单规范一种 种类（Type）：种类中的数据存在关系。表单的每一列都是某一个特定的属性，每一行代表一个“个体”。如 Fig 1 所体现的一个表单。RM是一种 基于记录的数据模型（Record-based model），也就是说，它有对一些种类的固定的格式的记录。
  2. 实体关系模型（ERD：Entity Relationship Diagram/model）。ERD由很多基础的对象（也就是“实体”）和对象之间的关系组成。每个实体是现实世界中的物体。
  3. 半结构化数据（Semi-structured Data Model）。由JSON和XML这类语言支持的模型。每个相同的种类的属性可以不一样。
  4. 面向对象模型（Object-Based Data Model）。通常可以看作是对于关系模型的升级：支持了OOP语言的特性：封装，方法，和对象。可以储存和运行程序。






1.4 数据抽象 Data Abstraction

     数据抽象分为三个层级，不同的人员接触不同的层级。

     物理层面（Physical level）。这是最底层的抽象，它描述数据是如何真实地被储存的。比如它会描述 instructor 这部分数据是如何以Byte连续在磁盘中存储的。这关系到底层的算法/数据查找方式和一定程度的硬件。另外，数据库使用一种特殊的数据结构 索引（Index） 去找特定的数据，这也是物理层面的关注点。**数据库管理员（DB Administrator）**会面对这一层面。

     逻辑层面（Logical level）。逻辑层面关心储存什么数据，和数据之间什么关系。换句话说，这个层级用来为数据库中的数据的种类和属性在一定程度上做出定义和修改。比如下面的代码就是逻辑层面的体现：

type instructor = record
        ID : char (5); 
        instr_name : char (20); 
        dept_name : char (20); 
        salary : numeric (8,2);
end;

这段代码定义了一个种类instructor，它有四个属性，以键值对的方式声明。例如ID : char (5);表达：这个属性是ID，是一个char类型的数据，长度为20。程序员写SQL就是在面对逻辑层面。

     视界层面（View level）。即用户面对的层面。一个数据库可能提供多个视界，一个用户可能面对一个或多个世界。除了作为一层抽象省去不必要的细节，这个层面也增加了安全性。比如不同的view有不同的权限：一个老师只能看见学生的成绩，看不到学生的住址。

     这三个抽象层及对应三种 架构（Schema） ：physical schema，logical schema，subschemas。一个架构是对数据设计的描述。与之对应的概念叫做 实例（Instances）：它是真实存在于数据库中数据的统称。前者可以类比为对一个变量（DB）的声明，而后者则是所赋的值。通常来说，物理架构的改变不会影响逻辑架构，这种行为叫做物理数据独立性（Physical data independenc）。







1.5 数据库语言 DB Languages

     一个数据库系统会提供两种语言：**数据定义语言（DDL：Data-definition Language）**和 数据操作语言（DML：Data-manipulation Language）。通常来说，一个数据库系统所使用的两者实际上会是同一种语言，比如SQL。

     DDL会对数据种类做出定义，同时也会实现一些 数据完整性约束（Data Integrity Constraints）。上面instructor的例子就是使用SQL为DDL。完整性约束通常有这几种：

  1. 值域限制（Domain Constraints）。它是最基础的限制：比如学校的预算必须大于等于0。
  2. 引用完整性（Referential Integrity）。比如希望某个数据（属性）出现在不同的关系（种类）中，它确保这个属性真的有出现，在使用的时候不会找不到。
  3. 授权（Authorization）。实现不同的使用者有不同的 增删改查（CRUD） 权限。

DDL语句的输出被存放在 数据目录（Data Index），元数据（Metadata） 保存在此。

    DML实现CRUD。存在两种DML：1. 过程化DMLs（Procedural DMLs）：让使用者去指明修改什么数据以及如何得到这些数据；2. 声明式 DMLs（Declarative/nonprocedural DMLs）：只需要指明访问什么数据。前者更加接近底层，而后者更加简单：把指定如何得到数据的问题交给了数据库自己去处理。**查询语言（Query Language）**通常指DML之下负责取出数据的这部分，但两者通常可以交替使用。使用SQL为DML：

select instructor.instr_name 
from instructor 
where instructor.dept_name = 'History';

    不同于其他计算机语言，比如Java，C++，数据库语言无法实现I/O，网络等。想要实现这些功能必须和其他高级语言合作。应用程序就是这一合作的体现。用户和应用程序交互，应用程序会发送DML语句，到数据库并执行。通过API实现应用程序和DB的交互：C语言有ODBC，Java有JDBC。







1.6 数据库设计 DB Design

     数据库的涉及主要就是对架构（Logical Schema）们的设计。      最开始，通常要弄清这个数据库未来的需求，比如交互方式等。之后，设计者需要选一个DB模型然后把需求和模型相结合实现一个概念设计（Conceptual Design）。比如以关系模型为例，概念设计设计需要高清什么属性去涵盖，如何组织归纳这些属性到各个表单中。前者通常是商业决策，而后者是技术相关。有两种解决后者的办法：1. 使用E-R模型；2. 使用一系列算法来 规范化（Normalization） 输入的属性和输出的表单。同时，一个合格的概念设计也包括增删改查相关的设计。在概念设计阶段之后首先是 逻辑设计阶段（Logical-design phase），之后是物理设计阶段（Physical-design phase）。前者实现数据模型，后者实现文件组织方式和内部储存结构等。







1.7 数据库引擎 DB Engine

     一个DBMS通常有三个部分组成：1. 储存管理器（Storage Manager） 2. 查询处理器（Query Processor） 3. 交易管理器（Transaction Management）

    储存管理器提供了数据库的数据与应用程序之间的接口，支持向DB发送的query。它会把DML语句翻译成底层的文件系统的指令。通过它实现CRUD。

     查询处理器包含DDL intepreter，DML编译器等。是底层相关的。      交易管理保证原子化，连贯性，持久性（Durability），是数据安全重要的部分。“交易”不能单纯理解为商业交易，而应当理解为数据的“交易”。为了保证数据安全，交易管理由 并发管理器（Concurrency-control manager），恢复管理器（Recovery manager） 组成。后者在交易失败的情况下恢复数据，保证持久性。







1.8 数据库和应用 Database and Application Architecture

     早期的DBMS有两层：客户端-服务端。应用程序安装在客户端，向服务端发送请求实现访问数据库。现代的DBMS有三层：客户端（只作为前端）- 应用服务器（Application Server） - DBMS。这种模式安全性和性能都更好

     关系模型（RM：Relational Model） 是最常用的数据库模型，因为它简单。经过半个世纪的发展，它现在有很多 对象关系（Object-relational） 的特性。这一模型独立于底层的数据结构和算法。

     一个关系模型实际上是一个表单的集合，如 Fig.4。







1.9 概念 Concepts

    在这个模型中三个概念可以交替使用：table/relation/type：即表单，关系，种类。这张表可以称为“Instructor Relation”：内部数据都和教师有关。通常来说，row代表一个表单中一组有关联的数据，描述一个实体。每一行也被称为 元组（Tuple） 。而列则代表 属性（Attributes）。n个attr代表每个元组是 n-tuple，即一个tuple中有n个元素。attr的数量也被叫做arity：n个attr也被叫做n-ary。

    关系实例（Relation instance） 就是一个形式的关系的实例化。比如这张实质上存在的表单就是“Department”这一关系的实例。

     值域（Domain）。每个属性都有自己的值域，它是一个囊括了这一attr所有可能出现的值的集合。传统地说，是 **原子化（Atomic）**的，即不可再分：值域中的元素一定不是另一个集合或者数组。但是现在这一限制通常不再重要了。

     NULL值被很多DBMS支持，表示未知/不存在值。







1.10 DB 架构 DB Schema

     如之前所说，架构（Schema） 与实例是对立的概念。这里的schema通常指logical level的schema。一个relation schema可以类比成类，而instnace可以类比为schema实例化的对象。比如对于上面的 Department 实例， 他的架构形如department (dept name, building, budget)。







1.11 键 Keys

     超级键（Superkey） ：超级键是由一个或多个attr组成的集合，使用这些属性可以找到某架构中唯一的tuple。比如在 Instructor Relation 中，dept_name 无法构成一个超级键。但是 ID 可以：每个ID都是独一无二的，因此可以识别出唯一的Tuple（有的DBMS会自动分配这种唯一的编号给所有Tuple）。虽然ID本身可以当作超级键的唯一元素使之有效，任何包含ID的其他集合也可以成为超级键（这是因为ID可以识别唯一tuple）。比如{ID，dept_name}：后者不起什么作用，但不妨碍这个集合仍然是一个超级键。

     候选键（Candidate keys）： 用最少的元素分别出一个Tuple的超级键。候选键可以有多个：假设每个部门中肯定没有同名的教师，那么{ID}与{name,dept_name}都是候选键，虽然元素数量不同，但都没有多余的元素，保证了“元素最少”。

     通常数据库设计者会选出一个候选键，那么这个被选中的候选键就被叫做关系键（Primary key）。在schema中表现为带下划线的attr：

        classroom (building, room number, capacity)；

这些选中的属性的值必须很少被更改或者永远不会变，且非NULL。超级键能得到唯一Tuple也被叫做 超级键限制（Superkey constraint） 。

     另一种和attr有关的限制是 外键限制（Foreign-key constraint）。这一限制表达的是：设A为r1的某几个attr的集合，A中元素在每个tuple中的值必须和r2中的primary key B中同样的attr在一些tuple中的值一样。A被称为一个来自r1的外键，引用r2. r1被称为引用关系，r2被称为被引用关系。比如，教师关系中的dept_name就是一个外键，引用了Depatment中的值且dept_name是Department的关系键。实际上它是一种引用完整性限制（Referential integrity constrain）。相比后者，前者要求引用的attr同时也必须是被引用方的primary key。它们都可以防止垃圾信息的输入。

chema之间的这种关系可以用图表来表达：双箭头表达attr1引用attr2，单箭头表达A引用了B的所有primary key的元素。







1.12 关系查询语言 Relational Query Language

     与数据库交互的语言，分为三大类：

  1. Imperative：描述一系列操作来得到结果，比如使用一个循环来得到符合某一条件的数据。是早期的主要方式。
  2. functional：查询的结果是函数运行的结果，函数可能操作与数据库活着另一个函数的结果。
  3. declarative：只需要告知想要的信息，不关心如何取得

SQL三种特性都有。

     两个原生的搜索语言，即没任何语法糖

Relational algebra：是一种函数语言，也是SQL的基础（但是它仍高于物理层面）。 Relational calculus：declarative，不讨论。







1.13 关系代数 Relational algebra

     关系代数 （Relational algebra）： 把一或两个关系（实例）作为输入，通过一系列操作（组成一个 表达式（Expression） ）得到一个新的关系/实例。涉及1个关系的叫做单元（Unary），如：选择，投影，重命名；涉及2个关系叫做双元（Binary） ，如：并集，笛卡尔积，集合差。注： 一个关系本身也可以算作一个最简单的表达式

     选择（Select）：写作：$\sigma_{predicate}(E)$；表示在表达式 $E$ 中选择符合 $predicate$，即一个判断条件， 的tuples.

$$\text{Fig.7 }\sigma_{\text{depat_name= ‘Physics’}}(\textit{Instructor})$$

     投影（Project）：标记为：$\Pi_{attrs}(E)$：把$E$中的指定attrs提取出来变成新的关系。operation之间可以嵌套：$\Pi_{\text{name}}(\sigma_{\text{depat_name= ‘Physics’}})$,把 $\sigma$ 的结果当作E。这些都叫做relational-algebra expression。

     笛卡尔积(Cartesian product)：标记为 $A\times B$，$A,B$ 都是关系。它会把两个关系排列组合成一个关系，大小为两者大小的乘积。笛卡尔积有1个限制：不能同名的关系相乘。这导致自己不能乘自己，也不能乘一个表达式（表达式的结果无名）：通过重命名操作解决。结果的R的schema中若有同名的attr，就加前缀，比如 $$\text{Instructor}\times \text{Department}$$ 中就会有Instructor.dept_name与Department.dept_name

     合并（Join）：如果说 $\times$ 是一种没有选择的合并/完全的排列组合，那么Join就是有选择的合并，形式为 $r\Join_\theta s$：$r,s$ 是两个关系，$\theta$ 是判断条件。它本质上是select+X的实现：$ r\Join_\theta s = \sigma_\theta (r \times s)$。比如 $$\sigma_{\text{Instructor.dept_name=Department.dept_name}}(\text{Instructor}\times \text{Department}) $$

$$=$$

$$ \text{Instructor}\Join_\text{Instructor.dept_name=Department.dept_name} \text{Department}$$

     集合操作（Set operation）：

         并集（Union）：写作 $A\cup B$：把两个关系前后相连。

         交集（ Intersection）：写作 $A\cap B$：提取出两个关系中一样的Tuples。

         集合差（Set difference）：写作 $A - B$：删掉A与B共享的tuple后的A。

限制：两个关系的attrs完全一致：名称，个数，顺序（同一个schema）。

     赋值（Assignment）：标记为 varName $\leftarrow E$. 可以说是给E的结果关系一个变量名。并没有什么用，只能让结构看起来更清晰。

     重命名（Rename）：标记为：$\rho_x(E)$ ，$x$为$E$的新名字。可以通过 $\rho_{x(A_1,A_2..A_n)}(E)$ 来同时给属性重命名。
